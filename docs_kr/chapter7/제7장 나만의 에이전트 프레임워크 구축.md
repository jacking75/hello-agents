# 제7장 나만의 에이전트 프레임워크 구축

앞선 장들에서 에이전트의 기초 지식을 설명하고 주류 프레임워크가 제공하는 개발의 편리함을 체험했습니다. 이 장부터는 더 도전적이고 가치 있는 단계에 들어갑니다: **처음부터 에이전트 프레임워크 HelloAgents를 단계적으로 구축합니다**.

학습 과정의 연속성과 재현성을 보장하기 위해, HelloAgents는 버전 반복 방식으로 개발을 진행합니다. 각 장은 이전 장을 기반으로 새로운 기능 모듈을 추가하고, 에이전트 관련 지식 포인트를 연결하여 구현합니다. 최종적으로 이 자체 구축 프레임워크를 활용하여 이 책의 후속 장들의 고급 응용 사례를 효율적으로 구현합니다.

## 7.1 프레임워크 전체 아키텍처 설계

### 7.1.1 자체 Agent 프레임워크가 필요한 이유

에이전트 기술이 빠르게 발전하는 오늘날, 시장에는 이미 많은 성숙한 Agent 프레임워크가 있습니다. 그렇다면 왜 처음부터 새로운 프레임워크를 구축해야 할까요?

（1）시장 프레임워크의 빠른 반복과 한계

에이전트 분야는 빠르게 발전하는 분야로 언제든 새로운 개념이 생성될 수 있습니다. 에이전트 설계에 대해 각 프레임워크는 자체적인 포지셔닝과 이해가 있지만, 에이전트의 핵심 지식 포인트는 일관됩니다.

- **과도한 추상화의 복잡성**: 많은 프레임워크가 범용성을 추구하다 보니 대량의 추상 계층과 설정 옵션을 도입합니다. LangChain을 예로 들면, 체인 호출 메커니즘은 유연하지만 초학자에게는 학습 곡선이 가파르며, 간단한 작업을 완료하려면 많은 개념을 이해해야 합니다.
- **빠른 반복으로 인한 불안정성**: 상업화 프레임워크는 시장 점유를 위해 API 인터페이스가 빈번하게 변경됩니다. 개발자는 버전 업그레이드 후 코드가 실행되지 않는 문제에 자주 직면하며 유지보수 비용이 높습니다.
- **블랙박스화된 구현 로직**: 많은 프레임워크가 핵심 로직을 지나치게 엄밀하게 캡슐화하여 개발자가 Agent의 내부 동작 메커니즘을 이해하기 어렵고, 깊은 커스터마이즈 능력이 부족합니다. 문제 발생 시 문서와 커뮤니티 지원에만 의존해야 하며, 특히 커뮤니티가 충분히 활성화되지 않은 경우 피드백이 오래 걸려 후속 개발 효율성에 영향을 줄 수 있습니다.
- **의존 관계의 복잡성**: 성숙한 프레임워크는 종종 대량의 의존 패키지를 포함하며, 설치 패키지가 크고, 다른 프로젝트 코드와 함께 사용할 때 의존성 충돌 문제가 발생할 수 있습니다.

（2）사용자에서 구축자로의 능력 도약

자체 Agent 프레임워크를 구축하는 것은 실제로 "사용자"에서 "구축자"로 변화하는 과정입니다. 이 변화가 가져오는 가치는 장기적입니다.

- **Agent 작동 원리의 깊은 이해**: 각 컴포넌트를 직접 구현함으로써 개발자는 Agent의 사고 과정, 도구 호출 메커니즘, 다양한 설계 패턴의 장단점을 진정으로 이해할 수 있습니다.
- **완전한 제어권 획득**: 자체 구축 프레임워크는 모든 코드에 대한 완전한 제어를 의미하며, 구체적인 요구사항에 따라 정확한 최적화를 수행할 수 있고 서드파티 프레임워크 설계 이념의 제약을 받지 않습니다.
- **시스템 설계 능력 배양**: 프레임워크 구축 과정은 모듈화 설계, 인터페이스 추상화, 오류 처리 등 소프트웨어 엔지니어링 핵심 기술을 포함하며, 이러한 능력은 개발자의 장기적 성장에 중요한 가치가 있습니다.

（3）커스터마이즈 요구사항과 깊은 숙달의 필요성

실제 응용에서 다양한 시나리오에 대한 에이전트 요구사항의 차이가 크며, 범용 프레임워크를 기반으로 이차 개발이 필요한 경우가 많습니다.

- **특정 도메인 최적화 요구사항**: 금융, 의료, 교육 등 수직 분야는 종종 특정 프롬프트 템플릿, 특수 도구 통합, 커스터마이즈 보안 전략이 필요합니다.
- **성능과 리소스의 정확한 제어**: 프로덕션 환경에서 응답 시간, 메모리 사용량, 동시 처리 능력에 대한 엄격한 요구사항이 있으며, 범용 프레임워크의 "일률적인" 방안은 종종 세분화된 요구사항을 충족하지 못합니다.
- **학습과 교육의 투명성 요구**: 교육 시나리오에서 학습자는 에이전트의 각 구축 단계를 명확하게 볼 수 있기를 기대하며, 다양한 패러다임의 작동 메커니즘을 이해하기를 원합니다. 이는 프레임워크가 높은 관찰 가능성과 설명 가능성을 갖출 것을 요구합니다.

### 7.1.2 HelloAgents 프레임워크의 설계 이념

새로운 Agent 프레임워크를 구축하는 데 있어 핵심은 기능의 많고 적음이 아니라, 설계 이념이 기존 프레임워크의 문제점을 진정으로 해결할 수 있는가입니다. HelloAgents 프레임워크의 설계는 하나의 핵심 문제를 중심으로 전개됩니다: 학습자가 빠르게 시작할 수 있으면서도 Agent의 작동 원리를 깊이 이해할 수 있게 하려면 어떻게 해야 할까요?

성숙한 프레임워크를 처음 접할 때, 풍부한 기능에 매료될 수 있지만 곧 한 가지 문제를 발견합니다: 간단한 작업을 완료하려면 종종 Chain, Agent, Tool, Memory, Retriever 등 십여 개의 다른 개념을 이해해야 합니다. 각 개념마다 자체 추상 계층이 있어 학습 곡선이 비정상적으로 가파르게 됩니다. 이 복잡성은 강력한 기능을 가져오지만 초학자에게 장벽이 되기도 합니다. HelloAgents 프레임워크는 기능 완전성과 학습 친화성 사이의 균형점을 찾으려 하며, 네 가지 핵심 설계 이념을 형성했습니다.

（1）경량 및 교육 친화적 균형

훌륭한 학습 프레임워크는 완전한 가독성을 갖추어야 합니다. HelloAgents는 핵심 코드를 챕터별로 구분하며, 이는 간단한 원칙에 기반합니다: 어느 정도 프로그래밍 기초가 있는 개발자라면 합리적인 시간 내에 프레임워크의 작동 원리를 완전히 이해할 수 있어야 합니다. 의존성 관리에서 프레임워크는 극도로 간소한 전략을 채용합니다. OpenAI의 공식 SDK와 몇 가지 필수 기반 라이브러리 외에 무거운 의존성을 도입하지 않습니다. 문제 발생 시 프레임워크 자체 코드를 직접 찾을 수 있으며, 복잡한 의존 관계에서 답을 찾을 필요가 없습니다.

（2）표준 API 기반의 실용적 선택

OpenAI의 API는 이미 업계 표준이 되었으며, 거의 모든 주류 LLM 제공업체가 이 인터페이스를 호환하기 위해 노력하고 있습니다. HelloAgents는 새로운 추상 인터페이스를 재창조하는 대신 이 표준 위에 구축하기로 선택했습니다. 이 결정은 주로 몇 가지 동기에서 비롯됩니다. 첫째, 호환성 보장으로 HelloAgents 사용법을 익힌 후 다른 프레임워크로 마이그레이션하거나 기존 프로젝트에 통합할 때 기본 API 호출 로직이 완전히 일관됩니다. 둘째, 학습 비용 절감입니다. 이미 익숙한 표준 인터페이스를 기반으로 하므로 새로운 개념 모델을 배울 필요가 없습니다.

（3）점진적 학습 경로의 정교한 설계

HelloAgents는 명확한 학습 경로를 제공합니다. 각 장의 학습 코드를 pip으로 다운로드할 수 있는 히스토리 버전으로 저장하여, 코드 사용 비용을 걱정할 필요 없이 각 핵심 기능이 자신이 직접 작성한 것이 됩니다. 이 설계는 자신의 필요와 속도에 맞춰 진행할 수 있습니다. 각 단계의 업그레이드는 자연스럽고 개념적 도약이나 이해의 단절이 발생하지 않습니다. 특히 이 장의 내용도 앞의 여섯 장의 내용을 기반으로 완성됩니다. 마찬가지로 이 장은 후속 고급 지식 학습 부분을 위한 프레임워크 기반을 마련합니다.

（4）통일된 "도구" 추상화: 모든 것이 도구

경량 및 교육 친화적 이념을 철저히 관철하기 위해 HelloAgents는 아키텍처에서 핵심 에이전트 클래스를 제외한 모든 것을 Tools로 통일하는 핵심 단순화를 했습니다. 다른 프레임워크에서 독립적으로 학습해야 하는 Memory(메모리), RAG(검색 증강 생성), RL(강화 학습), MCP(프로토콜) 등 모듈이 HelloAgents에서는 모두 일종의 "도구"로 통일 추상화됩니다. 이 설계의 초기 목적은 불필요한 추상 계층을 제거하고 학습자가 가장 직관적인 "에이전트가 도구를 호출한다"는 핵심 로직으로 돌아가도록 하여, 빠른 시작과 깊은 이해의 통일을 진정으로 실현하는 것입니다.

"먼저 체험, 후 구현"의 학습 경로를 추천합니다. 이 장에서는 완전한 테스트 파일을 제공하며, 핵심 함수를 다시 작성하고 테스트를 실행하여 구현이 올바른지 검증할 수 있습니다. 이 학습 방식은 실천성을 보장하고 학습 효과를 확보합니다. 프레임워크의 구현 세부 사항을 깊이 알고 싶거나 프레임워크 개발에 참여하고 싶다면 이 [GitHub 저장소](https://github.com/jjyaoao/helloagents)를 방문할 수 있습니다.




## 7.2 HelloAgentsLLM 확장

이 절의 내용은 4.1.3절에서 생성한 `HelloAgentsLLM`을 기반으로 반복 업그레이드합니다. 이 기본 클라이언트를 더 적응력 있는 모델 호출 허브로 개조합니다. 이번 업그레이드는 주로 다음 세 가지 목표를 중심으로 합니다:

1. **다중 제공업체 지원**: OpenAI, ModelScope, 智谱AI 등 다양한 주류 LLM 서비스 제공업체로의 원활한 전환을 구현하여 프레임워크가 특정 공급업체에 바인딩되지 않도록 합니다.
2. **로컬 모델 통합**: VLLM과 Ollama 두 가지 고성능 로컬 배포 방안을 도입하여, 3.2.3절의 Hugging Face Transformers 방안의 프로덕션급 보완으로 데이터 프라이버시와 비용 제어 요구사항을 충족합니다.
3. **자동 감지 메커니즘**: 자동 식별 메커니즘을 구축하여 프레임워크가 환경 정보에 따라 지능적으로 사용 중인 LLM 서비스 유형을 추론하고 사용자의 설정 과정을 단순화합니다.

### 7.2.1 다중 제공업체 지원

이전에 정의한 `HelloAgentsLLM` 클래스는 이미 `api_key`와 `base_url` 두 핵심 파라미터를 통해 OpenAI 인터페이스와 호환되는 모든 서비스에 연결할 수 있습니다. 이론적으로 범용성을 보장하지만 실제 응용에서는 다른 서비스 제공업체가 환경 변수 이름, 기본 API 주소, 권장 모델 등에서 차이가 있습니다. 서비스 제공업체를 전환할 때마다 사용자가 수동으로 조회하고 코드를 수정해야 한다면 개발 효율성에 크게 영향을 줍니다. 이 문제를 해결하기 위해 `provider`를 도입합니다. 개선 아이디어는: `HelloAgentsLLM`이 내부적으로 다른 서비스 제공업체의 설정 세부 사항을 처리하여 사용자에게 통일되고 간결한 호출 경험을 제공하도록 합니다. 구체적인 구현 세부 사항은 7.2.3절 "자동 감지 메커니즘"에서 자세히 설명하며, 여기서는 먼저 이 메커니즘을 활용하여 프레임워크를 확장하는 방법에 집중합니다.

아래에서는 `HelloAgentsLLM`을 상속하여 ModelScope 플랫폼 지원을 추가하는 방법을 시연합니다. 독자가 프레임워크를 "사용"하는 방법뿐만 아니라 어떻게 "확장"하는지를 익힐 수 있기를 바랍니다. 설치된 라이브러리 소스 코드를 직접 수정하는 것은 권장되지 않는 방법입니다. 이후 라이브러리 업그레이드를 어렵게 만들기 때문입니다.

（1）커스텀 LLM 클래스 생성 및 상속

프로젝트 디렉토리에 `my_llm.py` 파일이 있다고 가정합니다. 먼저 `hello_agents` 라이브러리에서 `HelloAgentsLLM` 기반 클래스를 가져온 다음 이를 상속하는 `MyLLM` 클래스를 생성합니다.

```python
# my_llm.py
import os
from typing import Optional
from openai import OpenAI
from hello_agents import HelloAgentsLLM

class MyLLM(HelloAgentsLLM):
    """
    커스텀 LLM 클라이언트로, 상속을 통해 ModelScope 지원을 추가합니다.
    """
    pass # 임시로 비워둠
```

（2）새 공급업체를 지원하도록 `__init__` 메서드 재정의

다음으로 `MyLLM` 클래스에서 `__init__` 메서드를 재정의합니다. 목표는: 사용자가 `provider="modelscope"`를 전달하면 커스텀 로직을 실행하고, 그렇지 않으면 부모 클래스 `HelloAgentsLLM`의 원래 로직을 호출하여 OpenAI 등 다른 내장 공급업체를 계속 지원하는 것입니다.

```python
class MyLLM(HelloAgentsLLM):
    def __init__(
        self,
        model: Optional[str] = None,
        api_key: Optional[str] = None,
        base_url: Optional[str] = None,
        provider: Optional[str] = "auto",
        **kwargs
    ):
        # provider가 처리하려는 'modelscope'인지 확인
        if provider == "modelscope":
            print("커스텀 ModelScope Provider 사용 중")
            self.provider = "modelscope"
            
            # ModelScope 자격 증명 파싱
            self.api_key = api_key or os.getenv("MODELSCOPE_API_KEY")
            self.base_url = base_url or "https://api-inference.modelscope.cn/v1/"
            
            # 자격 증명 존재 여부 검증
            if not self.api_key:
                raise ValueError("ModelScope API key가 없습니다. MODELSCOPE_API_KEY 환경 변수를 설정하세요.")

            # 기본 모델 및 기타 파라미터 설정
            self.model = model or os.getenv("LLM_MODEL_ID") or "Qwen/Qwen2.5-VL-72B-Instruct"
            self.temperature = kwargs.get('temperature', 0.7)
            self.max_tokens = kwargs.get('max_tokens')
            self.timeout = kwargs.get('timeout', 60)
            
            # 획득한 파라미터로 OpenAI 클라이언트 인스턴스 생성
            self._client = OpenAI(api_key=self.api_key, base_url=self.base_url, timeout=self.timeout)

        else:
            # modelscope가 아닌 경우, 부모 클래스의 원래 로직을 사용
            super().__init__(model=model, api_key=api_key, base_url=base_url, provider=provider, **kwargs)

```

이 코드는 "재정의"의 사상을 보여줍니다: `provider="modelscope"` 경우를 인터셉트하여 특별 처리하고, 다른 모든 경우는 `super().__init__(...)`을 통해 부모 클래스에 넘겨 원래 프레임워크의 모든 기능을 보존합니다.

（3）커스텀 `MyLLM` 클래스 사용

이제 프로젝트의 비즈니스 로직에서 네이티브 `HelloAgentsLLM`을 사용하는 것처럼 자체 `MyLLM` 클래스를 사용할 수 있습니다.

먼저 `.env` 파일에서 ModelScope API 키를 설정합니다:

```bash
# .env file
MODELSCOPE_API_KEY="your-modelscope-api-key"
```

그런 다음 메인 프로그램에서 `MyLLM`을 가져와 사용합니다:

```python
# my_main.py
from dotenv import load_dotenv
from my_llm import MyLLM # 참고: 여기서 자체 클래스를 가져옴

# 환경 변수 로드
load_dotenv()

# 재정의된 클라이언트를 인스턴스화하고 provider 지정
llm = MyLLM(provider="modelscope") 

# 메시지 준비
messages = [{"role": "user", "content": "안녕하세요, 자기소개를 해주세요."}]

# 호출 시작, think 등 메서드는 부모 클래스에서 상속되어 재정의 불필요
response_stream = llm.think(messages)

# 응답 출력
print("ModelScope Response:")
for chunk in response_stream:
    # chunk는 my_llm 라이브러리에서 이미 한 번 출력되었으므로 pass만 하면 됩니다
    # print(chunk, end="", flush=True)
    pass
```

이 단계를 통해 `hello-agents` 라이브러리 소스 코드를 수정하지 않고도 새로운 기능을 성공적으로 확장했습니다. 이 방법은 코드의 깔끔함과 유지보수성을 보장하고, 향후 `hello-agents` 라이브러리를 업그레이드할 때 커스터마이즈 기능이 손실되지 않습니다.

### 7.2.2 로컬 모델 호출

3.2.3절에서 Hugging Face Transformers 라이브러리를 사용하여 로컬에서 직접 오픈소스 모델을 실행하는 방법을 배웠습니다. 이 방법은 입문 학습과 기능 검증에 매우 적합하지만, 높은 동시성 요청 처리 시 성능이 제한적이며 일반적으로 프로덕션 환경의 첫 번째 선택이 아닙니다.

로컬에서 고성능, 프로덕션급 모델 추론 서비스를 구현하기 위해 커뮤니티에서 VLLM과 Ollama와 같은 우수한 도구들이 등장했습니다. 연속 배치 처리, PagedAttention 등 기술을 통해 모델의 처리량과 실행 효율을 크게 향상시키고, 모델을 OpenAI 표준과 호환되는 API 서비스로 캡슐화합니다. 이는 `HelloAgentsLLM`에 원활하게 통합할 수 있음을 의미합니다.

**VLLM**

VLLM은 LLM 추론을 위해 설계된 고성능 Python 라이브러리입니다. PagedAttention 등 첨단 기술을 통해 표준 Transformers 구현보다 몇 배 높은 처리량을 달성할 수 있습니다. 로컬에 VLLM 서비스를 배포하는 완전한 단계는 다음과 같습니다:

먼저 하드웨어 환경(특히 CUDA 버전)에 맞게 VLLM을 설치해야 합니다. 버전 불일치 문제를 피하기 위해 [공식 문서](https://docs.vllm.ai/en/latest/getting_started/installation.html)를 따르는 것을 권장합니다.

```python
pip install vllm
```

설치 완료 후 다음 명령으로 OpenAI 호환 API 서비스를 시작할 수 있습니다. VLLM은 지정된 모델 가중치를 Hugging Face Hub에서 자동으로 다운로드합니다(로컬에 없는 경우). Qwen1.5-0.5B-Chat 모델을 예로 들겠습니다:

```
# VLLM 서비스를 시작하고 Qwen1.5-0.5B-Chat 모델 로드
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen1.5-0.5B-Chat \
    --host 0.0.0.0 \
    --port 8000
```

서비스가 시작되면 `http://localhost:8000/v1` 주소에서 OpenAI 호환 API를 제공합니다.

**Ollama**

Ollama는 로컬 모델 관리와 배포를 더욱 단순화하며, 모델 다운로드, 설정, 서비스 시작 등 단계를 하나의 명령으로 캡슐화합니다. 빠른 시작에 매우 적합합니다. Ollama [공식 사이트](https://ollama.com)를 방문하여 운영 체제에 맞는 클라이언트를 다운로드하여 설치합니다.

설치 후 터미널을 열고 다음 명령을 실행하면 모델을 다운로드하고 실행할 수 있습니다(Llama 3을 예로). Ollama는 모델 다운로드, 서비스 캡슐화, 하드웨어 가속 설정을 자동으로 처리합니다.

```
# 처음 실행 시 모델을 자동으로 다운로드하고, 이후에는 직접 서비스를 시작합니다
ollama run llama3
```

터미널에서 모델의 상호작용 프롬프트가 보이면 서비스가 백그라운드에서 성공적으로 시작된 것입니다. Ollama는 기본적으로 `http://localhost:11434/v1` 주소에서 OpenAI 호환 API 인터페이스를 노출합니다.

**`HelloAgentsLLM`에 통합**

VLLM과 Ollama 모두 업계 표준 API를 따르므로 `HelloAgentsLLM`에 통합하는 과정이 매우 간단합니다. 클라이언트를 인스턴스화할 때 새로운 `provider`로 취급하면 됩니다.

예를 들어 로컬에서 실행 중인 **VLLM** 서비스에 연결:

```python
llm_client = HelloAgentsLLM(
    provider="vllm",
    model="Qwen/Qwen1.5-0.5B-Chat", # 서비스 시작 시 지정한 모델과 일치해야 함
    base_url="http://localhost:8000/v1",
    api_key="vllm" # 로컬 서비스는 일반적으로 실제 API 키가 필요하지 않으며, 임의의 비어있지 않은 문자열을 입력할 수 있습니다
)
```

또는 환경 변수를 설정하고 클라이언트가 자동으로 감지하도록 하여 코드 수정 없이 구현:

```bash
# .env 파일에서 설정
LLM_BASE_URL="http://localhost:8000/v1"
LLM_API_KEY="vllm"

# Python 코드에서 바로 인스턴스화
llm_client = HelloAgentsLLM() # vllm으로 자동 감지됩니다
```

마찬가지로 로컬 **Ollama** 서비스에 연결도 간단합니다:

```python
llm_client = HelloAgentsLLM(
    provider="ollama",
    model="llama3", # `ollama run`으로 지정한 모델과 일치해야 함
    base_url="http://localhost:11434/v1",
    api_key="ollama" # 로컬 서비스도 실제 키가 필요하지 않습니다
)
```

이 통일된 설계를 통해 에이전트 핵심 코드를 수정하지 않고도 클라우드 API와 로컬 모델 간에 자유롭게 전환할 수 있습니다. 이는 후속 애플리케이션 개발, 배포, 비용 제어, 데이터 프라이버시 보호에 큰 유연성을 제공합니다.

### 7.2.3 자동 감지 메커니즘

사용자의 설정 부담을 최대한 줄이고 "설정보다 관례" 원칙을 따르기 위해 `HelloAgentsLLM` 내부에는 두 가지 핵심 보조 메서드인 `_auto_detect_provider`와 `_resolve_credentials`가 설계되어 있습니다. 이들은 협력하여 작동하며, `_auto_detect_provider`는 환경 정보에 따라 서비스 제공업체를 추론하고, `_resolve_credentials`는 추론 결과에 따라 구체적인 파라미터 설정을 완료합니다.

`_auto_detect_provider` 메서드는 환경 정보에 따라 다음 우선순위 순서로 서비스 제공업체를 자동으로 추론합니다:

1. **최고 우선순위: 특정 서비스 제공업체의 환경 변수 확인** 가장 직접적이고 신뢰할 수 있는 판단 근거입니다. 프레임워크는 순차적으로 `MODELSCOPE_API_KEY`, `OPENAI_API_KEY`, `ZHIPU_API_KEY` 등 환경 변수가 있는지 확인합니다. 어느 하나를 발견하면 즉시 해당 서비스 제공업체를 결정합니다.

2. **차순위: `base_url`에 따른 판단** 사용자가 특정 서비스 제공업체의 키를 설정하지 않았지만 범용 `LLM_BASE_URL`을 설정한 경우, 프레임워크는 이 URL을 파싱합니다.

   - **도메인 매칭**: URL에 `"api-inference.modelscope.cn"`, `"api.openai.com"` 등 특징 문자열이 포함되어 있는지 확인하여 클라우드 서비스 제공업체를 식별합니다.

   - **포트 매칭**: URL에 `:11434`(Ollama), `:8000`(VLLM) 등 로컬 서비스의 표준 포트가 포함되어 있는지 확인하여 로컬 배포 방안을 식별합니다.

3. **보조 판단: API 키 형식 분석** 일부 경우에 위 두 가지 방법으로 결정하지 못하면, 프레임워크는 범용 환경 변수 `LLM_API_KEY`의 형식을 분석합니다. 예를 들어 일부 서비스 제공업체의 API 키는 고정 접두사나 독특한 인코딩 형식이 있습니다. 그러나 이 방법은 모호성이 있을 수 있어(여러 서비스 제공업체의 키 형식이 유사한 경우) 우선순위가 낮으며 보조 수단으로만 사용됩니다.

핵심 코드의 일부는 다음과 같습니다:

```python
def _auto_detect_provider(self, api_key: Optional[str], base_url: Optional[str]) -> str:
    """
    LLM 제공업체 자동 감지
    """
    # 1. 특정 제공업체의 환경 변수 확인 (최고 우선순위)
    if os.getenv("MODELSCOPE_API_KEY"): return "modelscope"
    if os.getenv("OPENAI_API_KEY"): return "openai"
    if os.getenv("ZHIPU_API_KEY"): return "zhipu"
    # ... 다른 서비스 제공업체의 환경 변수 확인

    # 범용 환경 변수 가져오기
    actual_api_key = api_key or os.getenv("LLM_API_KEY")
    actual_base_url = base_url or os.getenv("LLM_BASE_URL")

    # 2. base_url에 따른 판단
    if actual_base_url:
        base_url_lower = actual_base_url.lower()
        if "api-inference.modelscope.cn" in base_url_lower: return "modelscope"
        if "open.bigmodel.cn" in base_url_lower: return "zhipu"
        if "localhost" in base_url_lower or "127.0.0.1" in base_url_lower:
            if ":11434" in base_url_lower: return "ollama"
            if ":8000" in base_url_lower: return "vllm"
            return "local" # 기타 로컬 포트

    # 3. API 키 형식에 따른 보조 판단
    if actual_api_key:
        if actual_api_key.startswith("ms-"): return "modelscope"
        # ... 다른 키 형식 판단

    # 4. 기본값으로 'auto' 반환, 범용 설정 사용
    return "auto"
```

`provider`가 결정되면(사용자 지정이든 자동 감지든), `_resolve_credentials` 메서드가 서비스 제공업체별 설정을 처리합니다. `provider` 값에 따라 해당 환경 변수를 찾고 기본 `base_url`을 설정합니다. 핵심 구현 일부는 다음과 같습니다:

```python
def _resolve_credentials(self, api_key: Optional[str], base_url: Optional[str]) -> tuple[str, str]:
    """provider에 따라 API 키와 base_url 파싱"""
    if self.provider == "openai":
        resolved_api_key = api_key or os.getenv("OPENAI_API_KEY") or os.getenv("LLM_API_KEY")
        resolved_base_url = base_url or os.getenv("LLM_BASE_URL") or "https://api.openai.com/v1"
        return resolved_api_key, resolved_base_url

    elif self.provider == "modelscope":
        resolved_api_key = api_key or os.getenv("MODELSCOPE_API_KEY") or os.getenv("LLM_API_KEY")
        resolved_base_url = base_url or os.getenv("LLM_BASE_URL") or "https://api-inference.modelscope.cn/v1/"
        return resolved_api_key, resolved_base_url
    
    # ... 다른 서비스 제공업체의 로직
```

자동 감지가 가져다주는 편의성을 간단한 예제로 느껴봅시다. 사용자가 로컬 Ollama 서비스를 사용하고 싶다면 `.env` 파일에서 다음과 같이 설정하기만 하면 됩니다:

```bash
LLM_BASE_URL="http://localhost:11434/v1"
LLM_MODEL_ID="llama3"
```

`LLM_API_KEY`를 설정하거나 코드에서 `provider`를 지정할 필요가 전혀 없습니다. 그런 다음 Python 코드에서 `HelloAgentsLLM`을 인스턴스화하면 됩니다:


## 7.3 프레임워크 인터페이스 구현

앞 절에서 대형 언어 모델과의 통신 핵심 문제를 해결하는 `HelloAgentsLLM` 핵심 컴포넌트를 구축했습니다. 그러나 데이터 흐름 처리, 설정 관리, 예외 대응, 상위 애플리케이션 구축을 위한 명확하고 통일된 구조를 제공하는 일련의 지원 인터페이스와 컴포넌트가 추가로 필요합니다. 이 절에서는 다음 세 가지 핵심 파일을 설명합니다:

- `message.py`: 프레임워크 내 통일된 메시지 형식을 정의하여 에이전트와 모델 간 정보 전달의 표준화를 보장합니다.
- `config.py`: 중앙화된 설정 관리 방안을 제공하여 프레임워크의 동작을 쉽게 조정하고 확장할 수 있습니다.
- `agent.py`: 모든 에이전트의 추상 기반 클래스(`Agent`)를 정의하여 후속 다양한 유형의 에이전트 구현을 위한 통일된 인터페이스와 규범을 제공합니다.

### 7.3.1 Message 클래스

에이전트와 대형 언어 모델의 상호작용에서 대화 히스토리는 매우 중요한 컨텍스트입니다. 이 정보를 규범적으로 관리하기 위해 간단한 `Message` 클래스를 설계했습니다. 후속 컨텍스트 엔지니어링 장에서 확장됩니다.

```python
"""메시지 시스템"""
from typing import Optional, Dict, Any, Literal
from datetime import datetime
from pydantic import BaseModel

# 메시지 역할의 유형 정의, 취값 제한
MessageRole = Literal["user", "assistant", "system", "tool"]

class Message(BaseModel):
    """메시지 클래스"""
    
    content: str
    role: MessageRole
    timestamp: datetime = None
    metadata: Optional[Dict[str, Any]] = None
    
    def __init__(self, content: str, role: MessageRole, **kwargs):
        super().__init__(
            content=content,
            role=role,
            timestamp=kwargs.get('timestamp', datetime.now()),
            metadata=kwargs.get('metadata', {})
        )
    
    def to_dict(self) -> Dict[str, Any]:
        """딕셔너리 형식으로 변환 (OpenAI API 형식)"""
        return {
            "role": self.role,
            "content": self.content
        }
    
    def __str__(self) -> str:
        return f"[{self.role}] {self.content}"
```

이 클래스 설계에는 몇 가지 핵심 포인트가 있습니다. 먼저 `typing.Literal`을 통해 `role` 필드의 취값을 `"user"`, `"assistant"`, `"system"`, `"tool"` 네 가지로 엄격하게 제한합니다. 이는 OpenAI API의 규범에 직접 대응하여 타입 안전성을 보장합니다. `content`와 `role` 두 핵심 필드 외에 `timestamp`와 `metadata`도 추가하여 로그 기록과 미래 기능 확장을 위한 공간을 마련합니다. 마지막으로 `to_dict()` 메서드는 내부 사용 `Message` 객체를 OpenAI API와 호환되는 딕셔너리 형식으로 변환하는 역할을 하며, "내부적으로 풍부하고, 외부적으로 호환"하는 설계 원칙을 구현합니다.

### 7.3.2 Config 클래스

`Config` 클래스의 역할은 코드의 하드코딩 설정 파라미터를 집중화하고 환경 변수에서 읽기를 지원하는 것입니다.

```python
"""설정 관리"""
import os
from typing import Optional, Dict, Any
from pydantic import BaseModel

class Config(BaseModel):
    """HelloAgents 설정 클래스"""
    
    # LLM 설정
    default_model: str = "gpt-3.5-turbo"
    default_provider: str = "openai"
    temperature: float = 0.7
    max_tokens: Optional[int] = None
    
    # 시스템 설정
    debug: bool = False
    log_level: str = "INFO"
    
    # 기타 설정
    max_history_length: int = 100
    
    @classmethod
    def from_env(cls) -> "Config":
        """환경 변수에서 설정 생성"""
        return cls(
            debug=os.getenv("DEBUG", "false").lower() == "true",
            log_level=os.getenv("LOG_LEVEL", "INFO"),
            temperature=float(os.getenv("TEMPERATURE", "0.7")),
            max_tokens=int(os.getenv("MAX_TOKENS")) if os.getenv("MAX_TOKENS") else None,
        )
    
    def to_dict(self) -> Dict[str, Any]:
        """딕셔너리로 변환"""
        return self.dict()
```

먼저 설정 항목을 `LLM 설정`, `시스템 설정` 등으로 논리적으로 분류하여 구조를 한눈에 볼 수 있습니다. 다음으로 각 설정 항목에는 합리적인 기본값이 설정되어 있어 프레임워크가 제로 설정으로도 작동할 수 있습니다. 가장 핵심적인 것은 `from_env()` 클래스 메서드로, 사용자가 환경 변수를 설정하여 기본 설정을 재정의할 수 있게 합니다. 코드를 수정할 필요가 없으며 다른 환경에 배포할 때 특히 유용합니다.

### 7.3.3 Agent 추상 기반 클래스

`Agent` 클래스는 전체 프레임워크의 최상위 추상입니다. 에이전트가 갖추어야 할 공통 동작과 속성을 정의하지만 구체적인 구현 방식은 신경 쓰지 않습니다. Python의 `abc`(Abstract Base Classes) 모듈을 통해 구현하며, 이는 모든 구체적인 에이전트 구현(후속 장의 `SimpleAgent`, `ReActAgent` 등)이 동일한 "인터페이스"를 따르도록 강제합니다.

```python
"""Agent 기반 클래스"""
from abc import ABC, abstractmethod
from typing import Optional, Any
from .message import Message
from .llm import HelloAgentsLLM
from .config import Config

class Agent(ABC):
    """Agent 기반 클래스"""
    
    def __init__(
        self,
        name: str,
        llm: HelloAgentsLLM,
        system_prompt: Optional[str] = None,
        config: Optional[Config] = None
    ):
        self.name = name
        self.llm = llm
        self.system_prompt = system_prompt
        self.config = config or Config()
        self._history: list[Message] = []
    
    @abstractmethod
    def run(self, input_text: str, **kwargs) -> str:
        """Agent 실행"""
        pass
    
    def add_message(self, message: Message):
        """히스토리에 메시지 추가"""
        self._history.append(message)
    
    def clear_history(self):
        """히스토리 지우기"""
        self._history.clear()
    
    def get_history(self) -> list[Message]:
        """히스토리 가져오기"""
        return self._history.copy()
    
    def __str__(self) -> str:
        return f"Agent(name={self.name}, provider={self.llm.provider})"
```

이 클래스의 설계는 객체 지향의 추상화 원칙을 구현합니다. 먼저 `ABC`를 상속하여 직접 인스턴스화할 수 없는 추상 클래스로 정의합니다. 생성자 `__init__`은 Agent의 핵심 의존성을 명확하게 정의합니다: 이름, LLM 인스턴스, 시스템 프롬프트, 설정. 가장 중요한 부분은 `@abstractmethod`로 데코레이팅된 `run` 메서드로, 모든 하위 클래스가 이 메서드를 구현하도록 강제하여 모든 에이전트가 통일된 실행 진입점을 갖도록 보장합니다. 또한 기반 클래스는 `Message` 클래스와 협력하는 범용 히스토리 관리 메서드를 제공하며, 이는 컴포넌트 간 연계를 보여줍니다.

이로써 `HelloAgents` 프레임워크 핵심 기반 컴포넌트의 설계와 구현을 완료했습니다.

## 7.4 Agent 패러다임의 프레임워크화 구현

이 절의 내용은 4장에서 구축한 세 가지 클래식 Agent 패러다임(ReAct, Plan-and-Solve, Reflection)을 기반으로 프레임워크화 재구성을 진행하고, SimpleAgent를 기본 대화 패러다임으로 새롭게 추가합니다. 이러한 독립적인 Agent 구현을 통일된 아키텍처 기반의 프레임워크 컴포넌트로 개조합니다. 이번 재구성은 주로 다음 세 가지 핵심 목표를 중심으로 합니다:

1. **프롬프트 엔지니어링의 체계적 향상**: 4장의 프롬프트를 심층 최적화하여 특정 작업 지향에서 범용 설계로 전환하고, 형식 제약과 역할 정의를 강화합니다.
2. **인터페이스와 형식의 표준화 통일**: 통일된 Agent 기반 클래스와 표준화된 실행 인터페이스를 구축하며, 모든 Agent가 동일한 초기화 파라미터, 메서드 서명, 히스토리 관리 메커니즘을 따릅니다.
3. **고도로 설정 가능한 커스터마이즈 능력**: 사용자 커스텀 프롬프트 템플릿, 설정 파라미터, 실행 전략을 지원합니다.

### 7.4.1 SimpleAgent

SimpleAgent는 가장 기본적인 Agent 구현으로, 프레임워크 기반 위에 완전한 대화 에이전트를 구축하는 방법을 보여줍니다. 프레임워크 기반 클래스를 상속하여 SimpleAgent를 재작성합니다. 먼저 프로젝트 디렉토리에 `my_simple_agent.py` 파일을 생성합니다:

```python
# my_simple_agent.py
from typing import Optional, Iterator
from hello_agents import SimpleAgent, HelloAgentsLLM, Config, Message

class MySimpleAgent(SimpleAgent):
    """
    재작성된 간단한 대화 Agent
    프레임워크 기반 클래스를 기반으로 커스텀 Agent를 구축하는 방법을 보여줍니다
    """

    def __init__(
        self,
        name: str,
        llm: HelloAgentsLLM,
        system_prompt: Optional[str] = None,
        config: Optional[Config] = None,
        tool_registry: Optional['ToolRegistry'] = None,
        enable_tool_calling: bool = True
    ):
        super().__init__(name, llm, system_prompt, config)
        self.tool_registry = tool_registry
        self.enable_tool_calling = enable_tool_calling and tool_registry is not None
        print(f"✅ {name} 초기화 완료, 도구 호출: {'활성화' if self.enable_tool_calling else '비활성화'}")
```

다음으로 Agent 기반 클래스의 추상 메서드 `run`을 재작성합니다. SimpleAgent는 선택적 도구 호출 기능을 지원하며 후속 장의 확장도 편리합니다:

```python
# my_simple_agent.py에 계속 추가
import re

class MySimpleAgent(SimpleAgent):
    # ... 앞의 __init__ 메서드

    def run(self, input_text: str, max_tool_iterations: int = 3, **kwargs) -> str:
        """
        재작성된 실행 메서드 - 간단한 대화 로직 구현, 선택적 도구 호출 지원
        """
        print(f"🤖 {self.name} 처리 중: {input_text}")

        # 메시지 목록 구성
        messages = []

        # 시스템 메시지 추가 (도구 정보 포함 가능)
        enhanced_system_prompt = self._get_enhanced_system_prompt()
        messages.append({"role": "system", "content": enhanced_system_prompt})

        # 히스토리 메시지 추가
        for msg in self._history:
            messages.append({"role": msg.role, "content": msg.content})

        # 현재 사용자 메시지 추가
        messages.append({"role": "user", "content": input_text})

        # 도구 호출이 활성화되지 않은 경우 간단한 대화 로직 사용
        if not self.enable_tool_calling:
            response = self.llm.invoke(messages, **kwargs)
            self.add_message(Message(input_text, "user"))
            self.add_message(Message(response, "assistant"))
            print(f"✅ {self.name} 응답 완료")
            return response

        # 다중 라운드 도구 호출 로직 지원
        return self._run_with_tools(messages, input_text, max_tool_iterations, **kwargs)

    def _get_enhanced_system_prompt(self) -> str:
        """도구 정보를 포함한 강화된 시스템 프롬프트 구성"""
        base_prompt = self.system_prompt or "당신은 유용한 AI 어시스턴트입니다."

        if not self.enable_tool_calling or not self.tool_registry:
            return base_prompt

        # 도구 설명 가져오기
        tools_description = self.tool_registry.get_tools_description()
        if not tools_description or tools_description == "사용 가능한 도구가 없습니다":
            return base_prompt

        tools_section = "\n\n## 사용 가능한 도구\n"
        tools_section += "다음 도구를 사용하여 질문에 답할 수 있습니다:\n"
        tools_section += tools_description + "\n"

        tools_section += "\n## 도구 호출 형식\n"
        tools_section += "도구를 사용해야 할 때 다음 형식을 사용하세요:\n"
        tools_section += "`[TOOL_CALL:{tool_name}:{parameters}]`\n"
        tools_section += "예: `[TOOL_CALL:search:Python 프로그래밍]` 또는 `[TOOL_CALL:memory:recall=사용자 정보]`\n\n"
        tools_section += "도구 호출 결과는 자동으로 대화에 삽입되고, 결과를 기반으로 계속 답변할 수 있습니다.\n"

        return base_prompt + tools_section
```

이제 도구 호출의 핵심 로직을 구현합니다:

```python
# my_simple_agent.py에 계속 추가
class MySimpleAgent(SimpleAgent):
    # ... 앞의 메서드

    def _run_with_tools(self, messages: list, input_text: str, max_tool_iterations: int, **kwargs) -> str:
        """도구 호출을 지원하는 실행 로직"""
        current_iteration = 0
        final_response = ""

        while current_iteration < max_tool_iterations:
            # LLM 호출
            response = self.llm.invoke(messages, **kwargs)

            # 도구 호출 여부 확인
            tool_calls = self._parse_tool_calls(response)

            if tool_calls:
                print(f"🔧 {len(tool_calls)}개의 도구 호출 감지")
                # 모든 도구 호출 실행 및 결과 수집
                tool_results = []
                clean_response = response

                for call in tool_calls:
                    result = self._execute_tool_call(call['tool_name'], call['parameters'])
                    tool_results.append(result)
                    # 응답에서 도구 호출 마커 제거
                    clean_response = clean_response.replace(call['original'], "")

                # 도구 결과를 포함한 메시지 구성
                messages.append({"role": "assistant", "content": clean_response})

                # 도구 결과 추가
                tool_results_text = "\n\n".join(tool_results)
                messages.append({"role": "user", "content": f"도구 실행 결과:\n{tool_results_text}\n\n이 결과를 기반으로 완전한 답변을 제공하세요."})

                current_iteration += 1
                continue

            # 도구 호출 없음, 이것이 최종 답변
            final_response = response
            break

        # 최대 반복 횟수 초과 시 마지막 답변 가져오기
        if current_iteration >= max_tool_iterations and not final_response:
            final_response = self.llm.invoke(messages, **kwargs)

        # 히스토리에 저장
        self.add_message(Message(input_text, "user"))
        self.add_message(Message(final_response, "assistant"))
        print(f"✅ {self.name} 응답 완료")

        return final_response

    def _parse_tool_calls(self, text: str) -> list:
        """텍스트에서 도구 호출 파싱"""
        pattern = r'\[TOOL_CALL:([^:]+):([^\]]+)\]'
        matches = re.findall(pattern, text)

        tool_calls = []
        for tool_name, parameters in matches:
            tool_calls.append({
                'tool_name': tool_name.strip(),
                'parameters': parameters.strip(),
                'original': f'[TOOL_CALL:{tool_name}:{parameters}]'
            })

        return tool_calls

    def _execute_tool_call(self, tool_name: str, parameters: str) -> str:
        """도구 호출 실행"""
        if not self.tool_registry:
            return f"❌ 오류: 도구 레지스트리가 설정되지 않았습니다"

        try:
            # 지능적 파라미터 파싱
            if tool_name == 'calculator':
                # 계산기 도구에는 직접 표현식 전달
                result = self.tool_registry.execute_tool(tool_name, parameters)
            else:
                # 다른 도구는 지능적 파라미터 파싱 사용
                param_dict = self._parse_tool_parameters(tool_name, parameters)
                tool = self.tool_registry.get_tool(tool_name)
                if not tool:
                    return f"❌ 오류: 도구 '{tool_name}'을 찾을 수 없습니다"
                result = tool.run(param_dict)

            return f"🔧 도구 {tool_name} 실행 결과:\n{result}"

        except Exception as e:
            return f"❌ 도구 호출 실패: {str(e)}"

    def _parse_tool_parameters(self, tool_name: str, parameters: str) -> dict:
        """도구 파라미터 지능적 파싱"""
        param_dict = {}

        if '=' in parameters:
            # 형식: key=value 또는 action=search,query=Python
            if ',' in parameters:
                # 다중 파라미터: action=search,query=Python,limit=3
                pairs = parameters.split(',')
                for pair in pairs:
                    if '=' in pair:
                        key, value = pair.split('=', 1)
                        param_dict[key.strip()] = value.strip()
            else:
                # 단일 파라미터: key=value
                key, value = parameters.split('=', 1)
                param_dict[key.strip()] = value.strip()
        else:
            # 직접 파라미터 전달, 도구 유형에 따라 지능적 추론
            if tool_name == 'search':
                param_dict = {'query': parameters}
            elif tool_name == 'memory':
                param_dict = {'action': 'search', 'query': parameters}
            else:
                param_dict = {'input': parameters}

        return param_dict
```

```python
class MySimpleAgent(SimpleAgent):
    # ... 앞의 메서드

    def stream_run(self, input_text: str, **kwargs) -> Iterator[str]:
        """
        커스텀 스트리밍 실행 메서드
        """
        print(f"🌊 {self.name} 스트리밍 처리 시작: {input_text}")

        messages = []

        if self.system_prompt:
            messages.append({"role": "system", "content": self.system_prompt})

        for msg in self._history:
            messages.append({"role": msg.role, "content": msg.content})

        messages.append({"role": "user", "content": input_text})

        # 스트리밍 LLM 호출
        full_response = ""
        print("📝 실시간 응답: ", end="")
        for chunk in self.llm.stream_invoke(messages, **kwargs):
            full_response += chunk
            print(chunk, end="", flush=True)
            yield chunk

        print()  # 줄 바꿈

        # 완전한 대화를 히스토리에 저장
        self.add_message(Message(input_text, "user"))
        self.add_message(Message(full_response, "assistant"))
        print(f"✅ {self.name} 스트리밍 응답 완료")

    def add_tool(self, tool) -> None:
        """Agent에 도구 추가 (편의 메서드)"""
        if not self.tool_registry:
            from hello_agents import ToolRegistry
            self.tool_registry = ToolRegistry()
            self.enable_tool_calling = True

        self.tool_registry.register_tool(tool)
        print(f"🔧 도구 '{tool.name}' 추가 완료")

    def has_tools(self) -> bool:
        """사용 가능한 도구가 있는지 확인"""
        return self.enable_tool_calling and self.tool_registry is not None
    
    def remove_tool(self, tool_name: str) -> bool:
        """도구 제거 (편의 메서드)"""
        if self.tool_registry:
            self.tool_registry.unregister(tool_name)
            return True
        return False
    
    def list_tools(self) -> list:
        """모든 사용 가능한 도구 목록"""
        if self.tool_registry:
            return self.tool_registry.list_tools()
        return []
```

### 7.4.2 ReActAgent

프레임워크화된 ReActAgent는 핵심 로직을 유지하면서 코드의 조직성과 유지보수성을 향상시킵니다. 주로 프롬프트 최적화와 프레임워크 도구 시스템과의 통합을 통해 이루어집니다.

（1）프롬프트 템플릿 개선

원래의 형식 요구사항을 유지하면서 "한 번에 하나의 단계만 실행"을 강조하여 혼란을 피하고, 두 가지 Action의 사용 시나리오를 명확히 했습니다.

```python

# my_react_agent.py
import re
from typing import Optional, List, Tuple
from hello_agents import ReActAgent, HelloAgentsLLM, Config, Message, ToolRegistry

class MyReActAgent(ReActAgent):

    재작성된 ReAct Agent - 추론과 행동이 결합된 에이전트
    """

    def __init__(
        self,
        name: str,
        llm: HelloAgentsLLM,
        tool_registry: ToolRegistry,
        system_prompt: Optional[str] = None,
        config: Optional[Config] = None,
        max_steps: int = 5,
        custom_prompt: Optional[str] = None
    ):

        self.tool_registry = tool_registry
        self.max_steps = max_steps
```

초기화 파라미터의 의미는 다음과 같습니다:

- `name`: Agent의 이름.
- `llm`: 대형 언어 모델과 통신을 담당하는 `HelloAgentsLLM` 인스턴스.
- `tool_registry`: Agent가 사용할 수 있는 도구를 관리하고 실행하는 `ToolRegistry` 인스턴스.
- `system_prompt`: 시스템 프롬프트, Agent의 역할과 행동 준칙 설정에 사용.
- `config`: 설정 객체, 프레임워크 수준 설정 전달에 사용.
- `max_steps`: ReAct 루프의 최대 실행 단계 수, 무한 루프 방지.
- `custom_prompt`: 커스텀 프롬프트 템플릿, 기본 ReAct 프롬프트 대체에 사용.

프레임워크화된 ReActAgent는 실행 흐름을 명확한 단계로 분해합니다:

```python
def run(self, input_text: str, **kwargs) -> str:
    """ReAct Agent 실행"""
    self.current_history = []
    current_step = 0

    print(f"\n🤖 {self.name} 문제 처리 시작: {input_text}")

    while current_step < self.max_steps:
        current_step += 1
        print(f"\n--- 단계 {current_step} ---")

        # 1. 프롬프트 구성
        tools_desc = self.tool_registry.get_tools_description()
        history_str = "\n".join(self.current_history)
        prompt = self.prompt_template.format(
            tools=tools_desc,
            question=input_text,
            history=history_str
        )

        # 2. LLM 호출
        messages = [{"role": "user", "content": prompt}]
        response_text = self.llm.invoke(messages, **kwargs)

        # 3. 출력 파싱
        thought, action = self._parse_output(response_text)

        # 4. 완료 조건 확인
        if action and action.startswith("Finish"):
            final_answer = self._parse_action_input(action)
            self.add_message(Message(input_text, "user"))
            self.add_message(Message(final_answer, "assistant"))
            return final_answer

        # 5. 도구 호출 실행
        if action:
            tool_name, tool_input = self._parse_action(action)
            observation = self.tool_registry.execute_tool(tool_name, tool_input)
            self.current_history.append(f"Action: {action}")
            self.current_history.append(f"Observation: {observation}")

    # 최대 단계 수 도달
    final_answer = "죄송합니다, 제한된 단계 내에 이 작업을 완료할 수 없습니다."
    self.add_message(Message(input_text, "user"))
    self.add_message(Message(final_answer, "assistant"))
    return final_answer
```

위의 재구성을 통해 ReAct 패러다임을 프레임워크에 성공적으로 통합했습니다. 핵심 개선은 통일된 `ToolRegistry` 인터페이스를 활용하고, 설정 가능하고 형식이 더 엄격한 프롬프트 템플릿을 통해 에이전트의 사고-행동 루프 실행 안정성을 향상시킨 것입니다. ReAct의 테스트 사례는 도구 호출이 필요하므로 파일 끝에 테스트 코드를 통합하여 제공합니다.

```python
    name="나의 코드 생성 어시스턴트",
```

### 7.4.5 FunctionCallAgent

FunctionCallAgent는 hello-agents 0.2.8 이후에 도입된 Agent로, OpenAI 네이티브 함수 호출 메커니즘을 기반으로 한 Agent입니다. OpenAI의 함수 호출 메커니즘을 사용하여 Agent를 구축하는 방법을 보여줍니다.
다음 기능을 지원합니다:

- _build_tool_schemas: 도구의 description으로 OpenAI의 function calling schema 구축
- _extract_message_content: OpenAI 응답에서 텍스트 추출
- _parse_function_call_arguments: 모델이 반환한 JSON 문자열 파라미터 파싱
- _convert_parameter_types: 파라미터 유형 변환

이러한 기능으로 네이티브 OpenAI Function Calling 능력을 갖추며, 프롬프트 제약 방식과 비교하여 더 강한 견고성을 가집니다.

```python
def _invoke_with_tools(self, messages: list[dict[str, Any]], tools: list[dict[str, Any]], tool_choice: Union[str, dict], **kwargs):
        """OpenAI 클라이언트를 직접 호출하여 함수 호출 실행"""
        client = getattr(self.llm, "_client", None)
        if client is None:
            raise RuntimeError("HelloAgentsLLM 클라이언트가 올바르게 초기화되지 않아 함수 호출을 실행할 수 없습니다.")

        client_kwargs = dict(kwargs)
        client_kwargs.setdefault("temperature", self.llm.temperature)
        if self.llm.max_tokens is not None:
            client_kwargs.setdefault("max_tokens", self.llm.max_tokens)

        return client.chat.completions.create(
            model=self.llm.model,
            messages=messages,
            tools=tools,
            tool_choice=tool_choice,
            **client_kwargs,
        )
```

## 7.5 도구 시스템

이 절에서는 앞서 구축한 Agent 기반 아키텍처를 토대로 도구 시스템의 설계와 구현을 심층 탐구합니다. 기반 시설 구축에서 시작하여 커스텀 개발 설계까지 점진적으로 심화합니다. 이 절의 학습 목표는 다음 세 가지 핵심 측면을 중심으로 합니다:

1. **통일된 도구 추상화 및 관리**: 표준화된 Tool 기반 클래스와 ToolRegistry 등록 메커니즘을 구축하여 도구의 개발, 등록, 검색, 실행을 위한 통일된 기반 시설을 제공합니다.

2. **실전 중심 도구 개발**: 수학 계산 도구를 사례로 커스텀 도구를 설계하고 구현하는 방법을 보여주며, 독자가 도구 개발의 완전한 프로세스를 익힐 수 있게 합니다.

3. **고급 통합 및 최적화 전략**: 다중 소스 검색 도구 설계를 통해 여러 외부 서비스를 통합하고, 지능적인 백엔드 선택, 결과 병합, 내결함성 처리를 구현하는 방법을 보여줍니다. 복잡한 시나리오에서 도구 시스템의 설계 사고를 구현합니다.

### 7.5.1 도구 기반 클래스 및 등록 메커니즘 설계

확장 가능한 도구 시스템을 구축하려면 먼저 표준화된 기반 시설을 구축해야 합니다. 이 기반 시설은 Tool 기반 클래스, ToolRegistry 등록 테이블, 도구 관리 메커니즘을 포함합니다.

（1）Tool 기반 클래스의 추상 설계

Tool 기반 클래스는 전체 도구 시스템의 핵심 추상으로, 모든 도구가 따라야 할 인터페이스 규범을 정의합니다:

````python
class Tool(ABC):
    """도구 기반 클래스"""

    def __init__(self, name: str, description: str):
        self.name = name
        self.description = description

    @abstractmethod
    def run(self, parameters: Dict[str, Any]) -> str:
        """도구 실행"""
        pass

    @abstractmethod
    def get_parameters(self) -> List[ToolParameter]:
        """도구 파라미터 정의 가져오기"""
        pass
````

이 설계는 객체 지향 설계의 핵심 사상을 구현합니다: 통일된 `run` 메서드 인터페이스를 통해 모든 도구가 일관된 방식으로 실행될 수 있으며, 딕셔너리 파라미터를 받아 문자열 결과를 반환하여 프레임워크의 일관성을 보장합니다. 동시에 도구는 자기 설명 능력을 갖추며, `get_parameters` 메서드를 통해 호출자에게 자신에게 필요한 파라미터를 명확히 알릴 수 있습니다. 이 내성 메커니즘은 자동 문서 생성과 파라미터 검증을 위한 기반을 제공합니다. name과 description 등 메타데이터 설계는 도구 시스템의 좋은 검색 가능성과 이해 가능성을 갖게 합니다.

（2）ToolParameter 파라미터 정의 시스템

복잡한 파라미터 검증과 문서 생성을 지원하기 위해 ToolParameter 클래스를 설계했습니다:

````python
class ToolParameter(BaseModel):
    """도구 파라미터 정의"""
    name: str
    type: str
    description: str
    required: bool = True
    default: Any = None
````

이 설계를 통해 도구가 자신의 파라미터 요구사항을 정확하게 설명할 수 있으며, 타입 확인, 기본값 설정, 자동 문서 생성을 지원합니다.

（3）ToolRegistry 등록 테이블 구현

ToolRegistry는 도구 시스템의 관리 허브로, 도구의 등록, 검색, 실행 등 핵심 기능을 제공합니다. 이 절에서 주로 사용하는 기능은 다음과 같습니다:

````python
class ToolRegistry:
    """HelloAgents 도구 등록 테이블"""

    def __init__(self):
        self._tools: dict[str, Tool] = {}
        self._functions: dict[str, dict[str, Any]] = {}

    def register_tool(self, tool: Tool):
        """Tool 객체 등록"""
        if tool.name in self._tools:
            print(f"⚠️ 경고: 도구 '{tool.name}'이 이미 존재합니다, 덮어쓰게 됩니다.")
        self._tools[tool.name] = tool
        print(f"✅ 도구 '{tool.name}'이 등록되었습니다.")
        
    def register_function(self, name: str, description: str, func: Callable[[str], str]):
        """
        함수를 직접 도구로 등록 (간편한 방법)

        Args:
            name: 도구 이름
            description: 도구 설명
            func: 도구 함수, 문자열 파라미터를 받아 문자열 결과를 반환
        """
        if name in self._functions:
            print(f"⚠️ 경고: 도구 '{name}'이 이미 존재합니다, 덮어쓰게 됩니다.")

        self._functions[name] = {
            "description": description,
            "func": func
        }
        print(f"✅ 도구 '{name}'이 등록되었습니다.")
````

ToolRegistry는 두 가지 등록 방식을 지원합니다:

1. **Tool 객체 등록**: 복잡한 도구에 적합하며, 완전한 파라미터 정의와 검증을 지원합니다
2. **함수 직접 등록**: 간단한 도구에 적합하며, 기존 함수를 빠르게 통합합니다

（4）도구 검색 및 관리 메커니즘

등록 테이블은 풍부한 도구 관리 기능을 제공합니다:

````python
def get_tools_description(self) -> str:
    """모든 사용 가능한 도구의 형식화된 설명 문자열 가져오기"""
    descriptions = []

    # Tool 객체 설명
    for tool in self._tools.values():
        descriptions.append(f"- {tool.name}: {tool.description}")

    # 함수 도구 설명
    for name, info in self._functions.items():
        descriptions.append(f"- {name}: {info['description']}")

    return "\n".join(descriptions) if descriptions else "사용 가능한 도구가 없습니다"
````

이 메서드가 생성하는 설명 문자열은 Agent의 프롬프트를 구성하는 데 직접 사용할 수 있어 Agent가 사용 가능한 도구를 알 수 있습니다.

````python
def to_openai_schema(self) -> Dict[str, Any]:
        """OpenAI function calling schema 형식으로 변환

        FunctionCallAgent에 사용, 도구가 OpenAI 네이티브 function calling을 사용할 수 있게 합니다

        Returns:
            OpenAI function calling 표준에 부합하는 schema
        """
        parameters = self.get_parameters()

        # properties 구성
        properties = {}
        required = []

        for param in parameters:
            # 기본 속성 정의
            prop = {
                "type": param.type,
                "description": param.description
            }

            # 기본값이 있는 경우 설명에 추가 (OpenAI schema는 default 필드를 지원하지 않음)
            if param.default is not None:
                prop["description"] = f"{param.description} (기본값: {param.default})"

            # 배열 유형인 경우 items 정의 추가
            if param.type == "array":
                prop["items"] = {"type": "string"}  # 기본 문자열 배열

            properties[param.name] = prop

            # 필수 파라미터 수집
            if param.required:
                required.append(param.name)

        return {
            "type": "function",
            "function": {
                "name": self.name,
                "description": self.description,
                "parameters": {
                    "type": "object",
                    "properties": properties,
                    "required": required
                }
            }
        }
````

이 메서드가 생성하는 schema는 네이티브 OpenAI SDK의 도구 호출에 직접 사용할 수 있습니다.

### 7.5.2 커스텀 도구 개발

기반 시설을 갖춘 후, 완전한 커스텀 도구를 개발하는 방법을 살펴봅니다. 수학 계산 도구는 간단하고 직관적이어서 좋은 예제입니다. 가장 직접적인 방법은 ToolRegistry의 함수 등록 기능을 사용하는 것입니다.

커스텀 수학 계산 도구를 만들어 봅니다. 먼저 프로젝트 디렉토리에 `my_calculator_tool.py`를 생성합니다:

```python
# my_calculator_tool.py
import ast

from hello_agents import ToolRegistry

def my_calculate(expression: str) -> str:
    """간단한 수학 계산 함수"""
    if not expression.strip():
        return "계산식이 비어있을 수 없습니다"

    # 지원하는 기본 연산
    operators = {
        ast.Add: operator.add,      # +
        ast.Sub: operator.sub,      # -
        ast.Mult: operator.mul,     # *
        ast.Div: operator.truediv,  # /
    }

    # 지원하는 기본 함수
    functions = {
        'sqrt': math.sqrt,
        'pi': math.pi,
    }

    try:
        node = ast.parse(expression, mode='eval')
        result = _eval_node(node.body, operators, functions)
        return str(result)
    except:
        return "계산 실패, 표현식 형식을 확인하세요"

def _eval_node(node, operators, functions):
    """간소화된 표현식 평가"""
    if isinstance(node, ast.Constant):
        return node.value
    elif isinstance(node, ast.BinOp):
        left = _eval_node(node.left, operators, functions)
        right = _eval_node(node.right, operators, functions)
        op = operators.get(type(node.op))
        return op(left, right)
    elif isinstance(node, ast.Call):
        func_name = node.func.id
        if func_name in functions:
            args = [_eval_node(arg, operators, functions) for arg in node.args]
            return functions[func_name](*args)
    elif isinstance(node, ast.Name):
        if node.id in functions:
            return functions[node.id]

def create_calculator_registry():
    """계산기를 포함한 도구 등록 테이블 생성"""
    registry = ToolRegistry()

    # 계산기 함수 등록
    registry.register_function(
        name="my_calculator",
        description="기본 연산(+,-,*,/)과 sqrt 함수를 지원하는 간단한 수학 계산 도구",
        func=my_calculate
    )

    return registry
```

### 7.5.3 다중 소스 검색 도구

실제 응용에서 여러 외부 서비스를 통합하여 더 강력한 기능을 제공해야 하는 경우가 많습니다. 검색 도구가 전형적인 예입니다. 여러 검색 엔진을 통합하여 더 완비된 실제 정보를 제공할 수 있습니다. 1장에서 Tavily의 검색 API를 사용했고, 4장에서 SerpApi의 검색 API를 사용했습니다. 따라서 이번에는 이 두 API를 사용하여 다중 소스 검색 기능을 구현합니다. 해당 Python 의존성이 설치되지 않은 경우 다음 스크립트를 실행할 수 있습니다:

```bash
pip install "hello-agents[search]==0.1.1"
```

（1）검색 도구의 통일된 인터페이스 설계

HelloAgents 프레임워크 내장 SearchTool은 고급 다중 소스 검색 도구를 설계하는 방법을 보여줍니다:

````python
class SearchTool(Tool):
    """
    지능형 혼합 검색 도구

    여러 검색 엔진 백엔드를 지원하며, 최적의 검색 소스를 지능적으로 선택합니다:
    1. 혼합 모드 (hybrid) - TAVILY 또는 SERPAPI를 지능적으로 선택
    2. Tavily API (tavily) - 전문 AI 검색
    3. SerpApi (serpapi) - 전통적인 Google 검색
    """

    def __init__(self, backend: str = "hybrid", tavily_key: Optional[str] = None, serpapi_key: Optional[str] = None):
        super().__init__(
            name="search",
            description="지능형 웹 검색 엔진. 혼합 검색 모드를 지원하며, 최적의 검색 소스를 자동으로 선택합니다."
        )
        self.backend = backend
        self.tavily_key = tavily_key or os.getenv("TAVILY_API_KEY")
        self.serpapi_key = serpapi_key or os.getenv("SERPAPI_API_KEY")
        self.available_backends = []
        self._setup_backends()
````

이 설계의 핵심 사상은 사용 가능한 API 키와 의존 라이브러리에 따라 최적의 검색 백엔드를 자동으로 선택하는 것입니다.

（2）TAVILY와 SERPAPI 검색 소스 통합 전략

프레임워크는 지능적인 백엔드 선택 로직을 구현합니다:

````python
def _search_hybrid(self, query: str) -> str:
    """혼합 검색 - 최적의 검색 소스 지능적 선택"""
    # Tavily 우선 사용 (AI 최적화 검색)
    if "tavily" in self.available_backends:
        try:
            return self._search_tavily(query)
        except Exception as e:
            print(f"⚠️ Tavily 검색 실패: {e}")
            # Tavily가 실패하면 SerpApi 시도
            if "serpapi" in self.available_backends:
                print("🔄 SerpApi 검색으로 전환")
                return self._search_serpapi(query)

    # Tavily를 사용할 수 없는 경우 SerpApi 사용
    elif "serpapi" in self.available_backends:
        try:
            return self._search_serpapi(query)
        except Exception as e:
            print(f"⚠️ SerpApi 검색 실패: {e}")

    # 둘 다 사용할 수 없는 경우 사용자에게 API 설정 안내
    return "❌ 사용 가능한 검색 소스가 없습니다, TAVILY_API_KEY 또는 SERPAPI_API_KEY 환경 변수를 설정하세요"
````

이 설계는 고가용성 시스템의 핵심 이념을 구현합니다: 다운그레이드 메커니즘을 통해 시스템이 최적의 검색 소스에서 사용 가능한 대체 방안으로 점차 다운그레이드할 수 있습니다. 모든 검색 소스를 사용할 수 없을 때 사용자에게 올바른 API 키 설정을 명확히 안내합니다.

（3）검색 결과의 통일된 형식화

서로 다른 검색 엔진이 반환하는 결과 형식이 다르므로, 프레임워크는 통일된 형식화 방법으로 처리합니다:

````python
def _search_tavily(self, query: str) -> str:
    """Tavily를 사용하여 검색"""
    response = self.tavily_client.search(
        query=query,
        search_depth="basic",
        include_answer=True,
        max_results=3
    )

    result = f"🎯 Tavily AI 검색 결과: {response.get('answer', '직접 답변을 찾을 수 없습니다')}\n\n"

    for i, item in enumerate(response.get('results', [])[:3], 1):
        result += f"[{i}] {item.get('title', '')}\n"
        result += f"    {item.get('content', '')[:200]}...\n"
        result += f"    출처: {item.get('url', '')}\n\n"

    return result
````

프레임워크의 설계 사상을 기반으로 자체 고급 검색 도구를 만들 수 있습니다. 이번에는 클래스 방식을 사용하여 다른 구현 방법을 보여줍니다. `my_advanced_search.py`를 생성합니다:

```python
# my_advanced_search.py
import os
from typing import Optional, List, Dict, Any
from hello_agents import ToolRegistry

class MyAdvancedSearchTool:
    """
    커스텀 고급 검색 도구 클래스
    다중 소스 통합 및 지능형 선택 설계 패턴을 보여줍니다
    """

    def __init__(self):
        self.name = "my_advanced_search"
        self.description = "지능형 검색 도구, 여러 검색 소스를 지원하며 최적의 결과를 자동으로 선택"
        self.search_sources = []
        self._setup_search_sources()

    def _setup_search_sources(self):
        """사용 가능한 검색 소스 설정"""
        # Tavily 가용성 확인
        if os.getenv("TAVILY_API_KEY"):
            try:
                from tavily import TavilyClient
                self.tavily_client = TavilyClient(api_key=os.getenv("TAVILY_API_KEY"))
                self.search_sources.append("tavily")
                print("✅ Tavily 검색 소스 활성화됨")
            except ImportError:
                print("⚠️ Tavily 라이브러리가 설치되지 않음")

        # SerpApi 가용성 확인
        if os.getenv("SERPAPI_API_KEY"):
            try:
                import serpapi
                self.search_sources.append("serpapi")
                print("✅ SerpApi 검색 소스 활성화됨")
            except ImportError:
                print("⚠️ SerpApi 라이브러리가 설치되지 않음")

        if self.search_sources:
            print(f"🔧 사용 가능한 검색 소스: {', '.join(self.search_sources)}")
        else:
            print("⚠️ 사용 가능한 검색 소스 없음, API 키를 설정해 주세요")

    def search(self, query: str) -> str:
        """지능형 검색 실행"""
        if not query.strip():
            return "❌ 오류: 검색 쿼리는 비워둘 수 없습니다"

        # 사용 가능한 검색 소스 확인
        if not self.search_sources:
            return """❌ 사용 가능한 검색 소스가 없습니다. 다음 API 키 중 하나를 설정해 주세요:

1. Tavily API: 환경 변수 TAVILY_API_KEY 설정
   획득 주소: https://tavily.com/

2. SerpAPI: 환경 변수 SERPAPI_API_KEY 설정
   획득 주소: https://serpapi.com/

설정 후 프로그램을 다시 실행하세요."""

        print(f"🔍 지능형 검색 시작: {query}")

        # 여러 검색 소스를 시도하여 최적의 결과 반환
        for source in self.search_sources:
            try:
                if source == "tavily":
                    result = self._search_with_tavily(query)
                    if result and "찾을 수 없음" not in result:
                        return f"📊 Tavily AI 검색 결과:\n\n{result}"

                elif source == "serpapi":
                    result = self._search_with_serpapi(query)
                    if result and "찾을 수 없음" not in result:
                        return f"🌐 SerpApi Google 검색 결과:\n\n{result}"

            except Exception as e:
                print(f"⚠️ {source} 검색 실패: {e}")
                continue

        return "❌ 모든 검색 소스가 실패했습니다. 네트워크 연결과 API 키 설정을 확인해 주세요"

    def _search_with_tavily(self, query: str) -> str:
        """Tavily로 검색"""
        response = self.tavily_client.search(query=query, max_results=3)

        if response.get('answer'):
            result = f"💡 AI 직접 답변: {response['answer']}\n\n"
        else:
            result = ""

        result += "🔗 관련 결과:\n"
        for i, item in enumerate(response.get('results', [])[:3], 1):
            result += f"[{i}] {item.get('title', '')}\n"
            result += f"    {item.get('content', '')[:150]}...\n\n"

        return result

    def _search_with_serpapi(self, query: str) -> str:
        """SerpApi로 검색"""
        import serpapi

        search = serpapi.GoogleSearch({
            "q": query,
            "api_key": os.getenv("SERPAPI_API_KEY"),
            "num": 3
        })

        results = search.get_dict()

        result = "🔗 Google 검색 결과:\n"
        if "organic_results" in results:
            for i, res in enumerate(results["organic_results"][:3], 1):
                result += f"[{i}] {res.get('title', '')}\n"
                result += f"    {res.get('snippet', '')}\n\n"

        return result

def create_advanced_search_registry():
    """고급 검색 도구가 포함된 레지스트리 생성"""
    registry = ToolRegistry()

    # 검색 도구 인스턴스 생성
    search_tool = MyAdvancedSearchTool()

    # 검색 도구의 메서드를 함수로 등록
    registry.register_function(
        name="advanced_search",
        description="고급 검색 도구, Tavily와 SerpAPI 여러 검색 소스를 통합하여 더 포괄적인 검색 결과 제공",
        func=search_tool.search
    )

    return registry
```

### 7.5.4 도구 시스템의 고급 기능

기본 도구 개발과 다중 소스 통합을 마스터한 후, 도구 시스템의 고급 기능을 살펴보겠습니다. 이러한 기능들은 복잡한 프로덕션 환경에서 도구 시스템을 안정적으로 실행하고 Agent에게 더 강력한 능력을 제공할 수 있도록 합니다.

(1) 도구 체인 호출 메커니즘

실제 응용에서 Agent는 종종 여러 도구를 조합하여 복잡한 작업을 완료해야 합니다. 제6장에서 언급한 그래프 개념을 참고하여 이러한 시나리오를 지원하는 도구 체인 관리자를 설계할 수 있습니다:

```python
# tool_chain_manager.py
from typing import List, Dict, Any, Optional
from hello_agents import ToolRegistry

class ToolChain:
    """도구 체인 - 여러 도구의 순차 실행을 지원"""

    def __init__(self, name: str, description: str):
        self.name = name
        self.description = description
        self.steps: List[Dict[str, Any]] = []

    def add_step(self, tool_name: str, input_template: str, output_key: str = None):
        """
        도구 실행 단계 추가

        Args:
            tool_name: 도구 이름
            input_template: 입력 템플릿, 변수 치환 지원
            output_key: 출력 결과의 키 이름, 후속 단계에서 참조하는 데 사용
        """
        self.steps.append({
            "tool_name": tool_name,
            "input_template": input_template,
            "output_key": output_key or f"step_{len(self.steps)}_result"
        })

    def execute(self, registry: ToolRegistry, initial_input: str, context: Dict[str, Any] = None) -> str:
        """도구 체인 실행"""
        context = context or {}
        context["input"] = initial_input

        print(f"🔗 도구 체인 실행 시작: {self.name}")

        for i, step in enumerate(self.steps, 1):
            tool_name = step["tool_name"]
            input_template = step["input_template"]
            output_key = step["output_key"]

            # 템플릿의 변수 치환
            try:
                tool_input = input_template.format(**context)
            except KeyError as e:
                return f"❌ 도구 체인 실행 실패: 템플릿 변수 {e}를 찾을 수 없음"

            print(f"  단계 {i}: {tool_name}으로 '{tool_input[:50]}...' 처리")

            # 도구 실행
            result = registry.execute_tool(tool_name, tool_input)
            context[output_key] = result

            print(f"  ✅ 단계 {i} 완료, 결과 길이: {len(result)} 자")

        # 마지막 단계의 결과 반환
        final_result = context[self.steps[-1]["output_key"]]
        print(f"🎉 도구 체인 '{self.name}' 실행 완료")
        return final_result

class ToolChainManager:
    """도구 체인 관리자"""

    def __init__(self, registry: ToolRegistry):
        self.registry = registry
        self.chains: Dict[str, ToolChain] = {}

    def register_chain(self, chain: ToolChain):
        """도구 체인 등록"""
        self.chains[chain.name] = chain
        print(f"✅ 도구 체인 '{chain.name}' 등록됨")

    def execute_chain(self, chain_name: str, input_data: str, context: Dict[str, Any] = None) -> str:
        """지정된 도구 체인 실행"""
        if chain_name not in self.chains:
            return f"❌ 도구 체인 '{chain_name}'이 존재하지 않음"

        chain = self.chains[chain_name]
        return chain.execute(self.registry, input_data, context)

    def list_chains(self) -> List[str]:
        """모든 도구 체인 목록"""
        return list(self.chains.keys())

# 사용 예시
def create_research_chain() -> ToolChain:
    """연구 도구 체인 생성: 검색 -> 계산 -> 요약"""
    chain = ToolChain(
        name="research_and_calculate",
        description="정보를 검색하고 관련 계산을 수행"
    )

    # 단계 1: 정보 검색
    chain.add_step(
        tool_name="search",
        input_template="{input}",
        output_key="search_result"
    )

    # 단계 2: 검색 결과를 기반으로 계산 (필요한 경우)
    chain.add_step(
        tool_name="my_calculator",
        input_template="다음 정보를 기반으로 관련 수치를 계산하세요: {search_result}",
        output_key="calculation_result"
    )

    return chain
```

(2) 비동기 도구 실행 지원

시간이 많이 걸리는 도구 작업의 경우 비동기 실행 지원을 제공할 수 있습니다:

```python
# async_tool_executor.py
import asyncio
import concurrent.futures
from typing import Dict, Any, List, Callable
from hello_agents import ToolRegistry

class AsyncToolExecutor:
    """비동기 도구 실행기"""

    def __init__(self, registry: ToolRegistry, max_workers: int = 4):
        self.registry = registry
        self.executor = concurrent.futures.ThreadPoolExecutor(max_workers=max_workers)

    async def execute_tool_async(self, tool_name: str, input_data: str) -> str:
        """단일 도구를 비동기로 실행"""
        loop = asyncio.get_event_loop()

        def _execute():
            return self.registry.execute_tool(tool_name, input_data)

        result = await loop.run_in_executor(self.executor, _execute)
        return result

    async def execute_tools_parallel(self, tasks: List[Dict[str, str]]) -> List[str]:
        """여러 도구를 병렬로 실행"""
        print(f"🚀 {len(tasks)}개의 도구 작업 병렬 실행 시작")

        # 비동기 작업 생성
        async_tasks = []
        for task in tasks:
            tool_name = task["tool_name"]
            input_data = task["input_data"]
            async_task = self.execute_tool_async(tool_name, input_data)
            async_tasks.append(async_task)

        # 모든 작업 완료 대기
        results = await asyncio.gather(*async_tasks)

        print(f"✅ 모든 도구 작업 실행 완료")
        return results

    def __del__(self):
        """리소스 정리"""
        if hasattr(self, 'executor'):
            self.executor.shutdown(wait=True)

# 사용 예시
async def test_parallel_execution():
    """병렬 도구 실행 테스트"""
    from hello_agents import ToolRegistry

    registry = ToolRegistry()
    # 검색 및 계산 도구가 이미 등록되어 있다고 가정

    executor = AsyncToolExecutor(registry)

    # 병렬 작업 정의
    tasks = [
        {"tool_name": "search", "input_data": "Python 프로그래밍"},
        {"tool_name": "search", "input_data": "머신러닝"},
        {"tool_name": "my_calculator", "input_data": "2 + 2"},
        {"tool_name": "my_calculator", "input_data": "sqrt(16)"},
    ]

    # 병렬 실행
    results = await executor.execute_tools_parallel(tasks)

    for i, result in enumerate(results):
        print(f"작업 {i+1} 결과: {result[:100]}...")
```

위의 설계와 구현 경험을 바탕으로 도구 시스템 개발의 핵심 이념을 정리할 수 있습니다: 설계 측면에서 각 도구는 단일 책임 원칙을 따르고, 특정 기능에 집중하면서 인터페이스의 일관성을 유지해야 하며, 완전한 예외 처리와 보안 우선의 입력 검증을 기본 요구사항으로 삼아야 합니다. 성능 최적화 측면에서는 비동기 실행을 활용하여 동시 처리 능력을 높이고, 외부 연결과 시스템 리소스를 합리적으로 관리해야 합니다.

프레임워크의 구체적인 구현에서 우리는 네 가지 고전적인 Agent 패러다임을 다시 구현했습니다. SimpleAgent의 기본 대화 패턴부터 ReActAgent의 추론과 행동 결합까지, ReflectionAgent의 자기 성찰과 반복 최적화부터 PlanAndSolveAgent의 분해 계획과 단계별 실행까지. 그리고 도구 시스템은 Agent 능력 확장의 핵심으로, 그 구축 과정은 완전한 엔지니어링 실습이었습니다.
