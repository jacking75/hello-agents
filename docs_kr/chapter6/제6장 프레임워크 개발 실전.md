# 제6장 프레임워크 개발 실전

4장에서는 네이티브 코드를 작성하여 ReAct, Plan-and-Solve, Reflection 등 여러 에이전트의 핵심 워크플로를 구현했습니다. 이 과정을 통해 에이전트의 내부 실행 로직을 이해할 수 있었습니다. 이후 5장에서는 "사용자" 관점으로 전환하여 로우코드 플랫폼이 가져다주는 편의성과 효율성을 체험했습니다.

이번 장의 목표는 업계 주류의 **에이전트 프레임워크**를 활용하여 신뢰성 있는 에이전트 애플리케이션을 효율적이고 체계적으로 구축하는 방법을 탐구하는 것입니다. 먼저 현재 시장의 주류 에이전트 프레임워크를 개괄하고, 대표적인 몇 가지 프레임워크에 대해 완전한 실전 사례를 통해 프레임워크 기반 개발 방식을 체험해 봅니다.

## 6.1 수동 구현에서 프레임워크 개발로

일회성 스크립트 작성에서 성숙한 프레임워크 사용으로 전환하는 것은 소프트웨어 엔지니어링 분야에서 중요한 사고의 도약입니다. 4장에서 작성한 코드는 주로 교육과 이해를 목적으로 합니다. 특정 작업을 잘 수행할 수 있지만, 여러 개의 다양한 유형의 복잡한 로직을 가진 에이전트 애플리케이션을 구축하는 데 사용한다면 금방 한계에 부딪히게 됩니다.

프레임워크의 본질은 검증된 "표준"을 제공하는 것입니다. 모든 에이전트에 공통적이고 반복적인 작업(메인 루프, 상태 관리, 도구 호출, 로그 기록 등)을 추상화하고 캡슐화하여, 새로운 에이전트를 구축할 때 범용적인 하위 구현이 아닌 고유한 비즈니스 로직에 집중할 수 있게 합니다.

### 6.1.1 에이전트 프레임워크가 필요한 이유

실전에 들어가기 전에 먼저 왜 프레임워크를 사용해야 하는지 명확히 해야 합니다. 독립적인 에이전트 스크립트를 직접 작성하는 것과 비교하여, 프레임워크 사용의 가치는 주로 다음과 같은 측면에서 나타납니다:

1. <strong>코드 재사용성 및 개발 효율 향상</strong>: 가장 직접적인 가치입니다. 좋은 프레임워크는 에이전트 실행의 핵심 루프(Agent Loop)를 캡슐화한 범용 `Agent` 기반 클래스 또는 실행기를 제공합니다. ReAct이든 Plan-and-Solve이든 프레임워크가 제공하는 표준 컴포넌트를 기반으로 빠르게 구축할 수 있어 중복 작업을 피할 수 있습니다.
2. <strong>핵심 컴포넌트의 분리 및 확장성 구현</strong>: 견고한 에이전트 시스템은 여러 느슨하게 결합된 모듈로 구성되어야 합니다. 프레임워크 설계는 서로 다른 관심사를 분리하도록 강제합니다:
   - <strong>모델 계층 (Model Layer)</strong>: 대형 언어 모델과의 상호작용을 담당하며, 다양한 모델(OpenAI, Anthropic, 로컬 모델)로 쉽게 교체 가능합니다.
   - <strong>도구 계층 (Tool Layer)</strong>: 표준화된 도구 정의, 등록 및 실행 인터페이스를 제공하여 새 도구를 추가해도 다른 코드에 영향을 주지 않습니다.
   - <strong>메모리 계층 (Memory Layer)</strong>: 단기 및 장기 메모리를 처리하며, 필요에 따라 다양한 메모리 전략(슬라이딩 윈도우, 요약 메모리 등)으로 전환 가능합니다. 이러한 모듈식 설계는 전체 시스템의 확장성을 크게 높이며, 어떤 컴포넌트를 교체하거나 업그레이드해도 간단합니다.
3. <strong>복잡한 상태 관리의 표준화</strong>: `ReflectionAgent`에서 구현한 `Memory` 클래스는 단순한 시작에 불과합니다. 실제로 장시간 실행되는 에이전트 애플리케이션에서 상태 관리는 거대한 도전입니다. 컨텍스트 창 제한, 히스토리 정보 지속성, 다중 턴 대화 상태 추적 등의 문제를 처리해야 합니다. 프레임워크는 강력하고 범용적인 상태 관리 메커니즘을 제공할 수 있어, 개발자가 매번 이러한 복잡한 문제를 다시 처리할 필요가 없습니다.
4. <strong>관찰성 및 디버깅 과정 단순화</strong>: 에이전트의 동작이 복잡해지면 의사결정 과정을 이해하는 것이 매우 중요합니다. 잘 설계된 프레임워크는 강력한 관찰성 기능을 내장할 수 있습니다. 예를 들어, 이벤트 콜백 메커니즘(Callbacks)을 도입하여 에이전트 생명주기의 핵심 지점(`on_llm_start`, `on_tool_end`, `on_agent_finish` 등)에서 자동으로 로그 기록이나 데이터 보고를 트리거할 수 있어, 에이전트의 완전한 실행 추적을 쉽게 추적하고 디버그할 수 있습니다. 이는 코드에 수동으로 `print` 문을 추가하는 것보다 훨씬 효율적이고 체계적입니다.

따라서 수동 구현에서 프레임워크 개발로 전환하는 것은 코드 구성 방식의 변화뿐만 아니라, 복잡하고 신뢰성 있으며 유지보수 가능한 에이전트 애플리케이션을 구축하기 위한 필수 경로입니다.

### 6.1.2 주류 프레임워크 선택 및 비교

에이전트 프레임워크 생태계는 전례 없는 속도로 발전하고 있습니다. LangChain과 LlamaIndex가 1세대 범용 LLM 애플리케이션 프레임워크의 패러다임을 정의했다면, 새로운 세대의 프레임워크는 특정 영역의 심층적인 도전, 특히 **다중 에이전트 협업 (Multi-Agent Collaboration)**과 **복잡한 워크플로 제어 (Complex Workflow Control)** 해결에 더욱 집중합니다.

이 장의 후속 실전에서는 이러한 최첨단 분야에서 가장 대표적인 네 가지 프레임워크인 AutoGen, AgentScope, CAMEL, LangGraph에 집중합니다. 이들의 설계 철학은 각각 다르며, 복잡한 에이전트 시스템을 구현하는 서로 다른 기술 경로를 대표합니다. 표 6.1에서 이를 확인할 수 있습니다.

- <strong>AutoGen</strong>: AutoGen의 핵심 사상은 대화를 통한 협업입니다<sup>[1]</sup>. 다중 에이전트 시스템을 여러 "대화 가능한" 에이전트로 구성된 그룹 채팅으로 추상화합니다. 개발자는 다양한 역할(`Coder`, `ProductManager`, `Tester` 등)을 정의하고 이들 간의 상호작용 규칙(예: `Coder`가 코드를 작성한 후 `Tester`가 자동으로 인계받는 방식)을 설정할 수 있습니다. 작업 해결 과정은 이러한 에이전트들이 그룹 채팅에서 자동화된 메시지 전달을 통해 지속적으로 대화, 협업, 반복하여 최종 목표를 달성하는 과정입니다.
- <strong>AgentScope</strong>: AgentScope는 다중 에이전트 애플리케이션을 위해 특별히 설계된 기능 완비형 개발 플랫폼입니다<sup>[2]</sup>. 핵심 특징은 **사용 편의성**과 **엔지니어링**입니다. 개발자가 에이전트를 쉽게 정의하고, 통신 네트워크를 구축하며, 전체 애플리케이션 생명주기를 관리할 수 있는 매우 친화적인 프로그래밍 인터페이스를 제공합니다. 내장된 **메시지 전달 메커니즘**과 분산 배포 지원으로 복잡하고 대규모 다중 에이전트 시스템 구축 및 운영에 매우 적합합니다.
- <strong>CAMEL</strong>: CAMEL은 **역할극 (Role-Playing)**이라는 새로운 협업 방법을 제공합니다<sup>[3]</sup>. 핵심 이념은 두 에이전트(예: `AI 연구원`과 `Python 프로그래머`)에게 각각의 역할과 공동 작업 목표를 설정하면, "**초기 프롬프트 (Inception Prompting)**"의 안내 아래 자율적으로 다중 턴 대화를 수행하여 서로 영감을 주고 협력하여 함께 작업을 완료한다는 것입니다. 다중 에이전트 대화 흐름 설계의 복잡도를 크게 낮춥니다.
- <strong>LangGraph</strong>: LangChain 생태계의 확장으로, LangGraph는 에이전트의 실행 흐름을 **그래프 (Graph)**로 모델링하는 독특한 접근 방식을 취합니다<sup>[4]</sup>. 전통적인 체인 구조에서 정보는 단방향으로만 흐를 수 있습니다. LangGraph는 각 단계 작업(LLM 호출, 도구 실행 등)을 그래프의 **노드 (Node)**로 정의하고, **엣지 (Edge)**를 사용하여 노드 간 전환 로직을 정의합니다. 이 설계는 자연스럽게 **순환 (Cycles)**을 지원하여 Reflection과 같은 반복, 수정, 자기 성찰의 복잡한 워크플로를 매우 간단하고 직관적으로 구현할 수 있습니다.

이어지는 소절에서는 이 네 가지 프레임워크 각각에 대해 완전한 실전 사례를 통해 프레임워크 기반 개발 방식을 심층 체험합니다. **참고로**, 모든 데모 프로젝트 소스 파일은 `code` 폴더 아래에 있으며, 본문에서는 원리 부분만 설명합니다.

## 6.2 프레임워크 1: AutoGen

앞서 언급한 바와 같이, AutoGen의 설계 철학은 "대화로 협업을 주도한다"는 것에 뿌리를 두고 있습니다. 복잡한 작업 해결 과정을 다양한 역할의 에이전트 간의 일련의 자동화 대화로 우아하게 매핑합니다. 이 핵심 이념을 바탕으로 AutoGen 프레임워크는 지속적으로 발전했습니다. `0.7.4` 버전을 예시로 들겠습니다. 이 버전은 현재까지 가장 최신 버전으로, 클래스 상속 설계에서 더욱 유연한 조합식 아키텍처로의 중요한 아키텍처 재구성을 나타냅니다. 이 프레임워크를 깊이 이해하고 적용하기 위해 먼저 가장 핵심적인 구성 요소와 기본 대화 상호작용 메커니즘을 설명합니다.

### 6.2.1 AutoGen의 핵심 메커니즘

`0.7.4` 버전의 출시는 AutoGen 발전의 중요한 이정표로, 프레임워크 기반 설계의 근본적인 혁신을 의미합니다. 이번 업데이트는 단순한 기능 추가가 아니라 전체 아키텍처를 재고하여 프레임워크의 모듈성, 동시성 성능 및 개발자 경험을 향상시키는 것을 목표로 합니다.

（1）프레임워크 구조의 진화

그림 6.1에서 볼 수 있듯이, 새 아키텍처의 가장 눈에 띄는 변화는 명확한 계층화와 비동기 우선 설계 이념의 도입입니다.

- <strong>계층화 설계:</strong> 프레임워크는 두 핵심 모듈로 분리됩니다:
  - `autogen-core`: 프레임워크의 하위 기반으로, 언어 모델과의 상호작용, 메시지 전달 등 핵심 기능을 캡슐화합니다. 프레임워크의 안정성과 미래 확장성을 보장합니다.
  - `autogen-agentchat`: `core` 위에 구축되어 대화형 에이전트 애플리케이션 개발을 위한 고급 인터페이스를 제공하며, 다중 에이전트 애플리케이션 개발 과정을 단순화합니다. 이러한 계층화 전략은 각 컴포넌트의 책임을 명확히 하고 시스템의 결합도를 낮춥니다.
- <strong>비동기 우선:</strong> 새 아키텍처는 전면적으로 비동기 프로그래밍(`async/await`)으로 전환합니다. 다중 에이전트 협업 시나리오에서 네트워크 요청(LLM API 호출 등)이 주요 시간 소모 작업입니다. 비동기 모드는 한 에이전트의 응답을 기다리는 동안 다른 작업을 처리할 수 있어 스레드 블로킹을 피하고 동시 처리 능력과 시스템 리소스 활용 효율을 크게 향상시킵니다.

（2）핵심 에이전트 컴포넌트

에이전트는 작업을 수행하는 기본 단위입니다. `0.7.4` 버전에서 에이전트 설계는 더욱 집중적이고 모듈화되었습니다.

- <strong>AssistantAgent (어시스턴트 에이전트):</strong> 작업의 주요 해결자로, 핵심은 대형 언어 모델(LLM)을 캡슐화한 것입니다. 대화 히스토리를 기반으로 논리적이고 지식이 풍부한 응답을 생성하는 역할을 합니다. 예를 들어 계획 제안, 글쓰기, 코드 작성 등을 수행합니다. 다양한 시스템 메시지(System Message)를 통해 다양한 "전문가" 역할을 부여할 수 있습니다.
- <strong>UserProxyAgent (사용자 프록시 에이전트):</strong> AutoGen에서 독특한 기능을 가진 컴포넌트입니다. 이중 역할을 수행합니다: 인간 사용자의 "대변인"으로서 작업을 시작하고 의도를 전달하며, 동시에 신뢰할 수 있는 "실행기"로서 코드 실행이나 도구 호출을 수행하고 결과를 다른 에이전트에 피드백합니다. 이 설계는 "생각"(AssistantAgent가 수행)과 "행동"을 명확히 구분합니다.

（3）GroupChatManager에서 Team으로

여러 에이전트가 협업해야 하는 작업의 경우 대화 흐름을 조율하는 메커니즘이 필요합니다. 이전 버전에서는 `GroupChatManager`가 이 역할을 담당했습니다. 새 아키텍처에서는 더 유연한 `Team` 또는 그룹 채팅 개념인 `RoundRobinGroupChat`이 도입되었습니다.

- <strong>라운드 로빈 그룹 채팅 (RoundRobinGroupChat):</strong> 명확하고 순서화된 대화 조율 메커니즘입니다. 참여하는 에이전트들이 미리 정의된 순서대로 차례로 발언하게 합니다. 이 모드는 고정된 프로세스의 작업에 매우 적합합니다. 예를 들어 전형적인 소프트웨어 개발 프로세스: 제품 관리자가 먼저 요구사항을 제시하고, 엔지니어가 코드를 작성한 후, 코드 리뷰어가 검토합니다.
- <strong>워크플로:</strong>
  1. 먼저 `RoundRobinGroupChat` 인스턴스를 생성하고 협업에 참여하는 모든 에이전트(제품 관리자, 엔지니어 등)를 추가합니다.
  2. 작업이 시작되면 그룹 채팅은 미리 설정된 순서에 따라 해당 에이전트를 순서대로 활성화합니다.
  3. 선택된 에이전트는 현재 대화 컨텍스트를 기반으로 응답합니다.
  4. 그룹 채팅은 새 응답을 대화 히스토리에 추가하고 다음 에이전트를 활성화합니다.
  5. 이 과정은 최대 대화 횟수에 도달하거나 미리 설정된 종료 조건을 충족할 때까지 계속됩니다.

이 방식으로 AutoGen은 복잡한 협업 관계를 프로세스가 명확하고 관리하기 쉬운 자동화된 "원탁회의"로 단순화합니다. 개발자는 각 팀원의 역할과 발언 순서만 정의하면 나머지 협업 프로세스는 그룹 채팅 메커니즘이 자율적으로 구동합니다.

다음 절에서는 소프트웨어 개발 팀을 시뮬레이션하는 사례를 구축하여, 새 아키텍처에서 다양한 역할의 에이전트를 정의하고 `RoundRobinGroupChat`으로 조율되는 그룹 채팅에 구성하여 실제 프로그래밍 작업을 협업으로 완료하는 방법을 직접 체험해 봅니다.

### 6.2.2 소프트웨어 개발 팀

AutoGen의 핵심 컴포넌트와 대화 메커니즘을 이해한 후, 이 절에서는 완전한 실전 사례를 통해 이러한 새로운 기능을 구체적으로 적용하는 방법을 보여줍니다. 다양한 전문 기술을 가진 여러 에이전트로 구성된 소프트웨어 개발 팀 시뮬레이션을 구축하여, 실제 소프트웨어 개발 작업을 협업으로 완료합니다.

（1）비즈니스 목표

목표는 기능이 명확한 웹 애플리케이션을 개발하는 것입니다: **비트코인 현재 가격 실시간 표시**. 이 작업은 작지만 소프트웨어 개발의 전형적인 과정인 요구사항 분석, 기술 선택, 코딩 구현에서 코드 리뷰와 최종 테스트까지 완전히 커버합니다. 이는 AutoGen의 자동화 협업 프로세스를 검증하는 이상적인 시나리오입니다.

（2）에이전트 팀 역할

실제 소프트웨어 개발 프로세스를 시뮬레이션하기 위해 네 가지 명확한 책임을 가진 에이전트 역할을 설계했습니다:

- <strong>ProductManager (제품 관리자):</strong> 사용자의 모호한 요구사항을 명확하고 실행 가능한 개발 계획으로 전환하는 역할을 담당합니다.
- <strong>Engineer (엔지니어):</strong> 개발 계획에 따라 구체적인 애플리케이션 코드를 작성하는 역할을 담당합니다.
- <strong>CodeReviewer (코드 리뷰어):</strong> 엔지니어가 제출한 코드를 검토하여 품질, 가독성 및 견고성을 확보하는 역할을 담당합니다.
- <strong>UserProxy (사용자 프록시):</strong> 최종 사용자를 대표하여 초기 작업을 시작하고, 최종 제공된 코드를 실행 및 검증하는 역할을 담당합니다.

이러한 역할 분류는 다중 에이전트 시스템 설계의 핵심 단계로, 복잡한 작업을 도메인 "전문가"가 처리하는 여러 하위 작업으로 분해합니다.

### 6.2.3 핵심 코드 구현

아래에서 이 자동화 팀의 핵심 코드를 단계별로 분석합니다.

（1）모델 클라이언트 설정

LLM 기반의 모든 에이전트는 언어 모델과 상호작용하기 위한 모델 클라이언트가 필요합니다. AutoGen `0.7.4`는 표준화된 `OpenAIChatCompletionClient`를 제공하며, OpenAI API 규범과 호환되는 모든 모델 서비스(OpenAI 공식 서비스, Azure OpenAI 및 Ollama와 같은 로컬 모델 서비스 포함)와 편리하게 연동할 수 있습니다.

독립적인 함수를 통해 모델 클라이언트를 생성하고 설정하며, 환경 변수를 통해 API Key와 서비스 주소를 관리합니다. 이는 코드의 유연성과 보안성을 향상시키는 좋은 엔지니어링 관행입니다.

```python
from autogen_ext.models.openai import OpenAIChatCompletionClient

def create_openai_model_client():
    """OpenAI 모델 클라이언트 생성 및 설정"""
    return OpenAIChatCompletionClient(
        model=os.getenv("LLM_MODEL_ID", "gpt-4o"),
        api_key=os.getenv("LLM_API_KEY"),
        base_url=os.getenv("LLM_BASE_URL", "https://api.openai.com/v1")
    )
```

（2）에이전트 역할 정의

에이전트를 정의하는 핵심은 고품질의 시스템 메시지(System Message)를 작성하는 것입니다. 시스템 메시지는 에이전트에게 설정된 "행동 준칙"과 "전문 지식 데이터베이스"와 같으며, 에이전트의 역할, 책임, 워크플로, 심지어 다른 에이전트와 상호작용하는 방식을 정확하게 규정합니다. 정교하게 설계된 시스템 메시지는 다중 에이전트 시스템이 효율적이고 정확하게 협업할 수 있는 핵심입니다. 소프트웨어 개발 팀에서는 각 역할마다 독립적인 함수를 생성하여 정의를 캡슐화합니다.

<strong>제품 관리자 (ProductManager)</strong>

제품 관리자는 전체 프로세스를 시작하는 역할입니다. 시스템 메시지는 책임을 정의할 뿐만 아니라 출력 구조를 규범화하고, 대화가 다음 단계(엔지니어)로 전환되도록 안내하는 명확한 지시를 포함합니다.

```python
def create_product_manager(model_client):
    """제품 관리자 에이전트 생성"""
    system_message = """당신은 소프트웨어 제품의 요구사항 분석과 프로젝트 계획을 전담하는 경험 풍부한 제품 관리자입니다.

핵심 책임은 다음과 같습니다:
1. **요구사항 분석**: 사용자 요구사항을 깊이 이해하고 핵심 기능과 경계 조건을 파악
2. **기술 계획**: 요구사항을 기반으로 명확한 기술 구현 경로 수립
3. **리스크 평가**: 잠재적인 기술 리스크와 사용자 경험 문제 파악
4. **조율 소통**: 엔지니어 및 다른 팀원과 효과적으로 소통

개발 작업을 받으면 다음 구조로 분석하세요:
1. 요구사항 이해 및 분석
2. 기능 모듈 분류
3. 기술 선택 제안
4. 구현 우선순위 정렬
5. 수락 기준 정의

간결하고 명확하게 응답하고, 분석이 완료되면 "엔지니어, 구현 시작해 주세요"라고 말하세요."""

    return AssistantAgent(
        name="ProductManager",
        model_client=model_client,
        system_message=system_message,
    )
```

<strong>엔지니어 (Engineer)</strong>

엔지니어의 시스템 메시지는 기술 구현에 집중합니다. 엔지니어의 기술 전문성을 열거하고 작업을 받은 후의 구체적인 행동 단계를 규정하며, 역시 프로세스를 코드 리뷰어로 안내하는 지시를 포함합니다.

```python
def create_engineer(model_client):
    """소프트웨어 엔지니어 에이전트 생성"""
    system_message = """당신은 Python 개발과 웹 애플리케이션 구축에 능숙한 시니어 소프트웨어 엔지니어입니다.

기술 전문성:
1. **Python 프로그래밍**: Python 문법과 모범 사례에 능숙
2. **웹 개발**: Streamlit, Flask, Django 등 프레임워크 정통
3. **API 통합**: 풍부한 서드파티 API 통합 경험
4. **오류 처리**: 코드의 견고성과 예외 처리를 중시

개발 작업을 받으면:
1. 기술 요구사항을 꼼꼼히 분석
2. 적합한 기술 방안 선택
3. 완전한 코드 구현 작성
4. 필요한 주석과 설명 추가
5. 경계 케이스와 예외 처리 고려

완전히 실행 가능한 코드를 제공하고, 완료되면 "코드 리뷰어, 검토해 주세요"라고 말하세요."""

    return AssistantAgent(
        name="Engineer",
        model_client=model_client,
        system_message=system_message,
    )
```

<strong>코드 리뷰어 (CodeReviewer)</strong>

코드 리뷰어의 정의는 코드의 품질, 보안성 및 규범성에 중점을 둡니다. 시스템 메시지는 리뷰의 중점과 프로세스를 상세히 나열하여, 코드 납품 전 품질 관문을 확보합니다.

```python
def create_code_reviewer(model_client):
    """코드 리뷰어 에이전트 생성"""
    system_message = """당신은 코드 품질과 모범 사례에 집중하는 경험 풍부한 코드 리뷰 전문가입니다.

리뷰 중점 사항:
1. **코드 품질**: 코드의 가독성, 유지보수성 및 성능 검사
2. **보안성**: 잠재적인 보안 취약점과 위험 지점 파악
3. **모범 사례**: 코드가 업계 표준과 모범 사례를 따르는지 확인
4. **오류 처리**: 예외 처리의 완전성과 합리성 검증

리뷰 프로세스:
1. 코드 로직을 꼼꼼히 읽고 이해
2. 코드 규범과 모범 사례 검사
3. 잠재적 문제와 개선 포인트 파악
4. 구체적인 수정 제안 제공
5. 코드의 전체적인 품질 평가

구체적인 리뷰 의견을 제공하고, 완료되면 "코드 리뷰 완료, 사용자 프록시 테스트해 주세요"라고 말하세요."""

    return AssistantAgent(
        name="CodeReviewer",
        model_client=model_client,
        system_message=system_message,
    )
```

<strong>사용자 프록시 (UserProxy)</strong>

`UserProxyAgent`는 특별한 에이전트로, LLM에 의존하여 응답하지 않고 시스템에서 사용자의 프록시 역할을 합니다. `description` 필드는 역할을 명확히 기술하며, 특히 중요한 것은 작업이 최종 완료된 후 `TERMINATE` 명령을 발행하여 전체 협업 프로세스를 정상 종료하는 역할을 담당합니다.

```python
def create_user_proxy():
    """사용자 프록시 에이전트 생성"""
    return UserProxyAgent(
        name="UserProxy",
        description="""사용자 프록시로 다음 역할을 담당합니다:
1. 사용자를 대표하여 개발 요구사항 제시
2. 최종 코드 구현 실행
3. 기능이 예상과 일치하는지 검증
4. 사용자 피드백 및 제안 제공

테스트 완료 후 TERMINATE로 회신하세요.""",
    )
```

이 네 가지 독립적인 정의 함수를 통해 기능이 완비된 "가상 팀"을 구축할 뿐만 아니라, 시스템 메시지를 통한 "프롬프트 엔지니어링"이 효율적인 다중 에이전트 애플리케이션 설계의 핵심 단계임을 보여줍니다.

（3）팀 협업 프로세스 정의

이 사례에서 소프트웨어 개발 프로세스는 상대적으로 고정적(요구사항->코딩->리뷰->테스트)이므로 `RoundRobinGroupChat`(라운드 로빈 그룹 채팅)이 이상적인 선택입니다. 비즈니스 로직 순서에 따라 네 에이전트를 참여자 목록에 추가합니다.

```python
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_agentchat.conditions import TextMentionTermination

# 팀 채팅 및 협업 규칙 정의
team_chat = RoundRobinGroupChat(
    participants=[
        product_manager,
        engineer,
        code_reviewer,
        user_proxy
    ],
    termination_condition=TextMentionTermination("TERMINATE"),
    max_turns=20,
)
```

- <strong>참여자 순서:</strong> `participants` 목록의 순서가 에이전트의 발언 순서를 결정합니다.
- <strong>종료 조건:</strong> `termination_condition`은 협업 프로세스가 언제 종료될지를 제어하는 핵심입니다. 여기서는 어떤 메시지에든 키워드 "TERMINATE"가 포함되면 대화가 종료되도록 설정합니다. 설계에서 이 명령은 최종 테스트 완료 후 `UserProxy`가 발행합니다.
- <strong>최대 턴 수:</strong> `max_turns`는 대화가 무한 루프에 빠지는 것을 방지하고 불필요한 리소스 소비를 피하기 위한 안전 밸브입니다.

（4）시작 및 실행

AutoGen `0.7.4`는 비동기 아키텍처를 채용하므로, 전체 협업 프로세스의 시작과 실행은 비동기 함수 내에서 완료되고 최종적으로 `asyncio.run()`으로 실행됩니다.

```python
async def run_software_development_team():
    # ... 클라이언트 및 에이전트 초기화 ...
    
    # 작업 설명 정의
    task = """비트코인 가격 표시 애플리케이션을 개발해야 합니다. 구체적인 요구사항:
            핵심 기능:
            - 비트코인 현재 가격(USD) 실시간 표시
            - 24시간 가격 변화 추세 표시(등락폭 및 등락액)
            - 가격 새로고침 기능 제공

            기술 요구사항:
            - Streamlit 프레임워크를 사용하여 웹 애플리케이션 생성
            - 인터페이스는 간결하고 아름다우며 사용자 친화적
            - 적절한 오류 처리 및 로딩 상태 추가

            팀이 협업하여 이 작업을 완료해 주세요, 요구사항 분석부터 최종 구현까지."""
    
    # 비동기로 팀 협업 실행, 대화 과정을 스트리밍 출력
    result = await Console(team_chat.run_stream(task=task))
    return result

# 메인 프로그램 진입점
if __name__ == "__main__":
    result = asyncio.run(run_software_development_team())
```

프로그램이 실행되면 `task`가 초기 메시지로 `team_chat`에 전달되고, 제품 관리자가 첫 번째 참여자로 해당 메시지를 받으면 전체 자동화 협업 프로세스가 시작됩니다.

### 6.2.4 AutoGen의 장점과 한계 분석

어떤 기술 프레임워크든 특정 적용 시나리오와 설계 트레이드오프가 있습니다. 이 절에서는 AutoGen의 핵심 장점과 실제 적용에서 직면할 수 있는 한계와 도전을 객관적으로 분석합니다.

（1）장점

- 사례에서 볼 수 있듯이, 에이전트 팀을 위해 복잡한 상태 기계나 제어 흐름 로직을 설계할 필요 없이, 완전한 소프트웨어 개발 프로세스를 제품 관리자, 엔지니어, 리뷰어 간의 대화로 자연스럽게 매핑합니다. 이 방식은 인간 팀의 협업 모델에 더 가깝고, 복잡한 작업 모델링의 문턱을 크게 낮춥니다. 개발자는 "누가(역할)"와 "무엇을 하는가(책임)"에 더 많은 에너지를 집중하고 "어떻게 하는가(프로세스 제어)"에는 덜 집중할 수 있습니다.
- 프레임워크는 시스템 메시지(System Message)를 통해 각 에이전트에게 고도로 전문화된 역할을 부여할 수 있습니다. 사례에서 `ProductManager`는 요구사항에, `CodeReviewer`는 품질에 집중합니다. 정교하게 설계된 에이전트는 다양한 프로젝트에서 재사용될 수 있어 유지보수와 확장이 용이합니다.
- 프로세스화된 작업의 경우 `RoundRobinGroupChat`과 같은 메커니즘이 명확하고 예측 가능한 협업 프로세스를 제공합니다. 동시에 `UserProxyAgent` 설계는 "인간 참여 루프"(Human-in-the-loop)를 위한 자연스러운 인터페이스를 제공합니다. 작업 발기자가 될 수도 있고 프로세스의 감독자 및 최종 검수자가 될 수도 있습니다. 이 설계는 자동화 시스템이 항상 인간의 감독 아래 있음을 보장합니다.

（2）한계

- `RoundRobinGroupChat`이 순서화된 프로세스를 제공하지만, LLM 기반 대화는 본질적으로 불확실성을 갖습니다. 에이전트가 예상에서 벗어난 응답을 생성하여 대화가 예상치 못한 방향으로 흘러가거나 루프에 빠질 수 있습니다.
- 에이전트 팀의 작업 결과가 기대에 미치지 못할 때 디버깅 과정이 매우 까다로울 수 있습니다. 전통적인 프로그램과 달리 명확한 오류 스택 대신 긴 대화 히스토리를 얻습니다. 이는 "대화형 디버깅"의 어려움이라 불립니다.

（3）OpenAI 외 모델 설정 보완

DeepSeek, 通義千問 등 OpenAI 외 시리즈 모델을 사용하려면 0.7.4 버전에서 `OpenAIChatCompletionClient` 파라미터에 모델 정보 딕셔너리를 전달해야 합니다. DeepSeek을 예로 들면:

```python
from autogen_ext.models.openai import OpenAIChatCompletionClient

model_client = OpenAIChatCompletionClient(
    model="deepseek-chat",
    api_key=os.getenv("DEEPSEEK_API_KEY"),
    base_url="https://api.deepseek.com/v1",
    model_info={
        "function_calling": True,
        "max_tokens": 4096,
        "context_length": 32768,
        "vision": False,
        "json_output": True,
        "family": "deepseek",
        "structured_output": True,
    }
)
```

이 `model_info` 딕셔너리는 AutoGen이 모델의 능력 경계를 이해하여 다양한 모델 서비스에 더 잘 적응하도록 돕습니다.



## 6.3 프레임워크 2: AgentScope

AutoGen의 설계 철학이 "대화로 협업을 주도한다"는 것이라면, AgentScope는 또 다른 기술 경로를 대표합니다: <strong>엔지니어링 우선의 다중 에이전트 플랫폼</strong>. AgentScope는 알리바바 다모원이 개발했으며, 대규모의 고신뢰성 다중 에이전트 애플리케이션 구축을 위해 특별히 설계되었습니다. 직관적이고 사용하기 쉬운 프로그래밍 인터페이스를 제공할 뿐만 아니라, 더 중요한 것은 분산 배포, 장애 복구, 관찰성 등 엔터프라이즈급 기능을 내장하여 장기적으로 안정적으로 실행되는 프로덕션 환경 애플리케이션 구축에 특히 적합합니다.

### 6.3.1 AgentScope 설계

AutoGen과 비교하여 AgentScope의 핵심 차이점은 <strong>메시지 기반 아키텍처 설계</strong>와 <strong>산업급 엔지니어링 실천</strong>에 있습니다. AutoGen이 유연한 "대화 스튜디오"에 가깝다면, AgentScope는 완전한 "에이전트 운영 체제"로 개발자에게 개발, 테스트, 배포까지의 전체 생명주기 지원을 제공합니다. 많은 프레임워크가 채택한 상속식 설계와 달리 AgentScope는 <strong>조합식 아키텍처</strong>와 <strong>메시지 기반 모드</strong>를 선택했습니다. 이 설계는 시스템의 모듈성을 강화할 뿐만 아니라 뛰어난 동시성 성능과 분산 능력의 기반을 마련합니다.

（1）계층형 아키텍처 체계

그림 6.2와 같이 AgentScope는 명확한 계층형 모듈화 설계를 채택하여, 하위 기반 컴포넌트부터 상위 애플리케이션 오케스트레이션까지 완전한 에이전트 개발 생태계를 형성합니다.

<div align="center">
  <img src="https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/6-figures/03.png" alt="" width="90%"/>
  <p>그림 6.2 AgentScope 아키텍처 다이어그램</p>

이 아키텍처에서 가장 하위 계층은 <strong>기반 컴포넌트 계층 (Foundational Components)</strong>으로, 전체 프레임워크에 핵심 빌딩 블록을 제공합니다. `Message` 컴포넌트는 간단한 텍스트 상호작용에서 복잡한 멀티모달 콘텐츠까지 지원하는 통일된 메시지 형식을 정의합니다. `Memory` 컴포넌트는 단기 및 장기 메모리 관리를 제공합니다. `Model API` 계층은 다양한 대형 언어 모델 호출을 추상화합니다. `Tool` 컴포넌트는 에이전트와 외부 세계의 상호작용 능력을 캡슐화합니다.

기반 컴포넌트 위에 <strong>에이전트 인프라 계층 (Agent-level Infrastructure)</strong>이 더 고급의 추상화를 제공합니다. 이 계층에는 다양한 사전 구축된 에이전트(브라우저 사용 에이전트, 심층 연구 에이전트 등)가 포함될 뿐만 아니라, 클래식 ReAct 패러다임을 구현하고, 에이전트 훅, 병렬 도구 호출, 상태 관리 등 고급 기능을 지원합니다. 특히 주목할 점은 이 계층이 <strong>비동기 실행과 실시간 제어</strong>를 기본적으로 지원한다는 것으로, 이는 AgentScope가 다른 프레임워크에 비해 갖는 중요한 장점입니다.

<strong>다중 에이전트 협업 계층 (Multi-Agent Cooperation)</strong>은 AgentScope의 핵심 혁신입니다. `MsgHub`가 메시지 센터로서 에이전트 간 메시지 라우팅과 상태 관리를 담당합니다. `Pipeline` 시스템은 순차, 병렬 등 다양한 실행 모드를 지원하는 유연한 워크플로 오케스트레이션 능력을 제공합니다. 이 설계를 통해 개발자는 복잡한 다중 에이전트 협업 시나리오를 쉽게 구축할 수 있습니다.

최상위 <strong>개발 및 배포 계층 (Deployment & Development)</strong>은 AgentScope의 엔지니어링 중시를 보여줍니다. `AgentScope Runtime`은 프로덕션급 런타임 환경을 제공하며, `AgentScope Studio`는 개발자에게 완전한 시각화 개발 도구 체인을 제공합니다.

（2）메시지 기반

AgentScope의 핵심 혁신은 <strong>메시지 기반 아키텍처</strong>에 있습니다. 이 아키텍처에서 모든 에이전트 상호작용은 전통적인 함수 호출이 아닌 <strong>메시지</strong>의 전송과 수신으로 추상화됩니다.

```python
from agentscope.message import Msg

# 메시지의 표준 구조
message = Msg(
    name="Alice",           # 전송자 이름
    content="Hello, Bob!",  # 메시지 내용
    role="user",           # 역할 유형
    metadata={             # 메타데이터 정보
        "timestamp": "2024-01-15T10:30:00Z",
        "message_type": "text",
        "priority": "normal"
    }
)
```

메시지를 상호작용의 기본 단위로 사용하면 몇 가지 핵심 장점이 있습니다:

- <strong>비동기 분리</strong>: 메시지의 전송자와 수신자가 시간적으로 분리되어 서로 기다릴 필요 없이 고동시성 시나리오를 기본적으로 지원합니다.
- <strong>위치 투명성</strong>: 에이전트는 다른 에이전트가 로컬 프로세스에 있는지 원격 서버에 있는지 신경 쓸 필요 없이 메시지 시스템이 자동으로 라우팅을 처리합니다.
- <strong>관찰성</strong>: 모든 메시지를 기록, 추적, 분석할 수 있어 복잡한 시스템의 디버깅과 모니터링을 크게 단순화합니다.
- <strong>신뢰성</strong>: 메시지를 영속적으로 저장하고 재시도할 수 있어 시스템에 장애가 발생해도 상호작용의 최종 일관성을 보장하여 시스템의 내결함성을 향상시킵니다.

（3）에이전트 생명주기 관리

AgentScope에서 각 에이전트는 명확한 생명주기(초기화, 실행, 일시정지, 소멸 등)를 갖고 통일된 기반 클래스 `AgentBase`를 기반으로 구현됩니다. 개발자는 일반적으로 핵심 `reply` 메서드에만 집중하면 됩니다.

```python
from agentscope.agents import AgentBase

class CustomAgent(AgentBase):
    def __init__(self, name: str, **kwargs):
        super().__init__(name=name, **kwargs)
        # 에이전트 초기화 로직
    
    def reply(self, x: Msg) -> Msg:
        # 에이전트의 핵심 응답 로직
        response = self.model(x.content)
        return Msg(name=self.name, content=response, role="assistant")
    
    def observe(self, x: Msg) -> None:
        # 에이전트의 관찰 로직 (선택)
        self.memory.add(x)
```

이 설계 패턴은 에이전트의 내부 로직과 외부 통신을 분리하여, 개발자는 `reply` 메서드에서 에이전트가 "생각하고 응답하는" 방식만 정의하면 됩니다.

（4）메시지 전달 메커니즘

AgentScope에는 <strong>메시지 센터 (MsgHub)</strong>가 내장되어 있으며, 이것이 메시지 기반 아키텍처 전체의 중추입니다. MsgHub는 메시지 라우팅과 분배를 담당할 뿐만 아니라 영속화와 분산 통신 등 고급 기능을 통합합니다. 다음과 같은 특징이 있습니다:

- <strong>유연한 메시지 라우팅</strong>: 일대일, 브로드캐스트, 멀티캐스트 등 다양한 통신 모드를 지원하여 유연하고 복잡한 상호작용 네트워크를 구축할 수 있습니다.
- <strong>메시지 영속화</strong>: 모든 메시지를 데이터베이스(SQLite, MongoDB 등)에 자동으로 저장하여 장기 실행 작업의 상태를 복원할 수 있음을 보장합니다.
- <strong>기본 분산 지원</strong>: AgentScope의 특징적인 기능입니다. 에이전트를 다른 프로세스나 서버에 배포할 수 있으며, `MsgHub`가 RPC(원격 프로시저 호출)를 통해 노드 간 통신을 자동으로 처리하여 개발자에게 완전히 투명합니다.

하위 아키텍처가 제공하는 이러한 엔지니어링 능력은 AgentScope가 높은 동시성, 높은 신뢰성이 필요한 복잡한 애플리케이션 시나리오를 처리할 때 전통적인 대화 기반 프레임워크보다 더 큰 장점을 갖게 합니다. 물론 개발자가 메시지 기반 비동기 프로그래밍 패러다임을 이해하고 적응하는 것이 필요합니다.

다음 절에서는 구체적인 실전 사례인 삼국 늑대인간 게임을 통해 AgentScope 프레임워크의 능력, 특히 동시 상호작용 처리에서의 장점을 심층 체험합니다.

### 6.3.2 삼국 늑대인간 게임

AgentScope의 메시지 기반 아키텍처와 다중 에이전트 협업 능력을 깊이 이해하기 위해 중국 고전 문화 요소를 융합한 "삼국 늑대인간" 게임을 구축합니다. 이 사례는 AgentScope가 복잡한 다중 에이전트 상호작용을 처리하는 장점을 보여줄 뿐만 아니라, <strong>실시간 협업</strong>, <strong>역할극</strong>, <strong>전략 게임</strong>이 필요한 시나리오에서 메시지 기반 아키텍처의 위력을 충분히 발휘하는 방법을 시연합니다. 전통적인 늑대인간과 달리, "삼국 늑대인간"은 유비, 관우, 제갈량 등 클래식 캐릭터를 게임에 도입하여 각 에이전트가 늑대인간의 기본 작업(늑대 암살, 예언가 조사, 마을 주민 추리)을 완료할 뿐만 아니라 해당 삼국 인물의 성격 특징과 행동 패턴도 구현합니다. 이 설계를 통해 AgentScope의 <strong>다층 역할 모델링</strong> 처리 성능을 관찰할 수 있습니다.

（1）아키텍처 설계와 핵심 컴포넌트

이 사례의 시스템 설계는 계층적 분리 원칙을 따르며, 게임 로직을 세 개의 독립적인 계층으로 나누고 각 계층은 AgentScope의 하나 이상의 핵심 컴포넌트에 매핑됩니다:

- <strong>게임 제어 계층 (Game Control Layer)</strong>: `ThreeKingdomsWerewolfGame` 클래스가 게임의 주 컨트롤러로서 전역 상태(플레이어 생존 목록, 현재 게임 단계 등) 유지, 게임 프로세스 진행(야간 단계, 주간 단계 호출), 승부 판정을 담당합니다.
- <strong>에이전트 상호작용 계층 (Agent Interaction Layer)</strong>: 완전히 `MsgHub`로 구동됩니다. 늑대인간 간의 비밀 협의부터 낮의 공개 토론까지 모든 에이전트 간 통신이 메시지 센터를 통해 라우팅되고 분배됩니다.
- <strong>역할 모델링 계층 (Role Modeling Layer)</strong>: 각 플레이어는 `DialogAgent` 기반의 인스턴스입니다. 정교하게 설계된 시스템 프롬프트를 통해 각 에이전트에게 "게임 역할"과 "삼국 인격"의 이중 정체성을 부여합니다.

（2）메시지 기반 게임 흐름

이 사례의 가장 핵심적인 설계는 <strong>메시지 기반</strong>으로 <strong>상태 기계</strong>를 대체하여 게임 흐름을 관리하는 것입니다. 전통적인 구현에서 게임 단계 전환은 일반적으로 중앙화된 상태 기계(State Machine)로 제어됩니다. AgentScope 패러다임에서 게임 흐름은 일련의 정의된 메시지 상호작용 패턴으로 자연스럽게 모델링됩니다.

예를 들어 늑대인간 단계 구현은 단순한 함수 호출이 아니라 `MsgHub`를 통해 늑대인간 플레이어만 포함하는 임시 비공개 통신 채널을 동적으로 생성합니다:

```python
async def werewolf_phase(self, round_num: int):
    """늑대인간 단계 - 메시지 기반 협업 모드 시연"""
    if not self.werewolves:
        return None
        
    # 메시지 센터를 통해 늑대인간 전용 통신 채널 구축
    async with MsgHub(
        self.werewolves,
        enable_auto_broadcast=True,
        announcement=await self.moderator.announce(
            f"늑대인간들, 오늘 밤 암살 목표를 논의하세요. 생존 플레이어: {format_player_list(self.alive_players)}"
        ),
    ) as werewolves_hub:
        # 토론 단계: 늑대인간이 메시지를 통해 전략 교환
        for _ in range(MAX_DISCUSSION_ROUND):
            for wolf in self.werewolves:
                await wolf(structured_model=DiscussionModelCN)
        
        # 투표 단계: 늑대인간의 암살 결정 수집 및 집계
        werewolves_hub.set_auto_broadcast(False)
        kill_votes = await fanout_pipeline(
            self.werewolves,
            msg=await self.moderator.announce("암살 목표를 선택하세요"),
            structured_model=WerewolfKillModelCN,
            enable_gather=False,
        )
```

이 설계의 장점은 게임 로직이 "특정 컨텍스트에서 어떤 패턴으로 메시지를 교환하는가"로 명확하게 표현되며, 일련의 경직된 상태 전환이 아니라는 것입니다. 주간 토론(전원 브로드캐스트), 예언가 조사(일대일 요청) 등 단계도 동일한 설계 패러다임을 따릅니다.

（3）구조화된 출력으로 게임 규칙 제약

늑대인간 게임의 핵심 도전 중 하나는 에이전트의 행동이 게임 규칙을 준수하도록 어떻게 보장하는가입니다. AgentScope의 <strong>구조화된 출력 메커니즘</strong>이 이 문제에 대한 해결책을 제공합니다. 다양한 게임 행동에 대해 엄격한 데이터 모델을 정의합니다:

```python
class DiscussionModelCN(BaseModel):
    """토론 단계의 출력 형식"""
    reach_agreement: bool = Field(
        description="합의에 도달했는지 여부",
        default=False
    )
    confidence_level: int = Field(
        description="현재 추론에 대한 신뢰도(1-10)",
        ge=1, le=10,
        default=5
    )
    key_evidence: Optional[str] = Field(
        description="의견을 지지하는 핵심 증거",
        default=None
    )

class WitchActionModelCN(BaseModel):
    """마녀 행동의 출력 형식"""
    use_antidote: bool = Field(description="해독제를 사용할지 여부")
    use_poison: bool = Field(description="독을 사용할지 여부")
    target_name: Optional[str] = Field(description="독 대상 플레이어 이름")
```

이 방식으로 에이전트 출력의 <strong>형식 일관성</strong>을 보장할 뿐만 아니라, 더 중요한 것은 <strong>게임 규칙의 자동화 제약</strong>을 구현합니다. 예를 들어 마녀 에이전트는 같은 목표에 동시에 해독제와 독을 사용할 수 없고, 예언가는 매일 밤 한 명의 플레이어만 조사할 수 있으며, 이러한 제약은 데이터 모델의 필드 정의와 검증 로직을 통해 자동으로 실행됩니다.

（4）역할 모델링의 이중 도전

이 사례에서 가장 흥미로운 기술적 도전은 에이전트가 동시에 두 가지 수준의 역할을 잘 수행하도록 만드는 방법입니다: <strong>게임 기능 역할</strong>(늑대인간, 예언가 등)과 <strong>문화 인격 역할</strong>(유비, 조조 등). 우리는 프롬프트 엔지니어링으로 이 문제를 해결합니다:

```python
def get_role_prompt(role: str, character: str) -> str:
    """역할 프롬프트 가져오기 - 게임 규칙과 인물 성격 융합"""
    base_prompt = f"""당신은 {character}이며, 이 삼국 늑대인간 게임에서 {role} 역할을 맡습니다.

중요 규칙:
1. 대화와 추리를 통해서만 게임에 참여할 수 있습니다
2. 외부 도구나 함수 호출을 시도하지 마세요
3. 요구된 JSON 형식으로 엄격하게 응답하세요

역할 특징:
"""
    
    if role == "늑대인간":
        return base_prompt + f"""
- 당신은 늑대인간 진영으로, 모든 선량한 사람을 제거하는 것이 목표입니다
- 밤에 다른 늑대인간과 암살 목표를 협의할 수 있습니다
- 낮에는 신원을 숨기고 선량한 사람들을 오도해야 합니다
- {character}의 성격으로 말하고 행동하세요
"""
```

이 설계를 통해 흥미로운 현상을 관찰했습니다: 다른 삼국 인물들이 같은 게임 역할을 수행할 때 완전히 다른 전략과 언어 스타일을 보여줍니다. 예를 들어 늑대인간 역할을 맡은 "조조"는 더 교활하고 위장을 잘하며, 늑대인간 역할을 맡은 "장비"는 더 직접적이고 충동적으로 보입니다.

（5）동시 처리와 내결함성 메커니즘

AgentScope의 비동기 아키텍처는 이 다중 에이전트 게임에서 중요한 역할을 합니다. 게임에서는 <strong>여러 에이전트의 결정을 동시에 수집</strong>해야 하는 장면이 자주 나타납니다. 예를 들어 투표 단계:

```python
# 모든 플레이어의 투표 결정을 병렬로 수집
vote_msgs = await fanout_pipeline(
    self.alive_players,
    await self.moderator.announce("제거할 플레이어를 투표로 선택하세요"),
    structured_model=get_vote_model_cn(self.alive_players),
    enable_gather=False,
)
```

`fanout_pipeline`은 모든 에이전트에게 동일한 메시지를 병렬로 전송하고 응답을 비동기로 수집할 수 있습니다. 이는 게임 실행 효율을 높일 뿐만 아니라, 실제 늑대인간 게임의 "동시 투표" 시나리오를 시뮬레이션합니다. 동시에 핵심 단계에 내결함성 처리를 추가했습니다:

```python
try:
    response = await wolf(
        "현재 상황을 분석하고 의견을 표현하세요.",
        structured_model=DiscussionModelCN
    )
except Exception as e:
    print(f"⚠️ {wolf.name} 토론 중 오류 발생: {e}")
    # 기본 응답 생성, 게임이 계속 진행되도록 보장
    default_response = DiscussionModelCN(
        reach_agreement=False,
        confidence_level=5,
        key_evidence="현재 분석 불가"
    )
```

이 설계는 특정 에이전트에 예외가 발생해도 전체 게임 흐름이 계속 진행될 수 있음을 보장합니다.

（6）사례 출력 및 요약

### 6.3.3 AgentScope의 장점과 한계 분석

"삼국 늑대인간" 사례를 통해 AgentScope 프레임워크의 핵심 장점을 깊이 체험했습니다. 메시지 기반 아키텍처를 핵심으로 복잡한 게임 프로세스를 일련의 동시, 비동기 메시지 전달 이벤트로 우아하게 매핑하여 전통적인 상태 기계의 경직성과 복잡성을 피했습니다. 강력한 구조화된 출력 능력과 결합하여 게임 규칙을 코드 수준의 제약으로 직접 전환하여 시스템의 안정성과 예측 가능성을 크게 향상시켰습니다. 이 설계 패러다임은 성능에서 기본 동시성의 장점을 보여줄 뿐만 아니라 내결함성 처리에서도 단일 에이전트에 예외가 발생해도 전체 프로세스가 견고하게 실행될 수 있음을 보장합니다.

그러나 AgentScope의 엔지니어링 장점은 어느 정도의 복잡성 비용을 수반합니다. 메시지 기반 아키텍처는 강력하지만 개발자에게 높은 기술적 요구를 부과하며, 비동기 프로그래밍, 분산 통신 등의 개념을 이해해야 합니다. 간단한 다중 에이전트 대화 시나리오의 경우 이 아키텍처는 과도하게 복잡할 수 있으며, "과도한 엔지니어링"의 위험이 있습니다. 또한 상대적으로 새로운 프레임워크로서 생태계와 커뮤니티 자원이 더 완비될 필요가 있습니다. 따라서 AgentScope는 대규모, 고신뢰성의 프로덕션급 다중 에이전트 시스템 구축이 필요한 경우에 더 적합하며, 빠른 프로토타입 개발이나 간단한 애플리케이션 시나리오의 경우 더 경량화된 프레임워크를 선택하는 것이 더 적합할 수 있습니다.



## 6.4 프레임워크 3: CAMEL

AutoGen과 AgentScope와 같이 기능이 완비된 프레임워크와 달리, CAMEL의 초기 핵심 목표는 최소한의 인간 개입으로 두 에이전트가 "역할극"을 통해 자율적으로 협업하여 복잡한 작업을 해결하는 방법을 탐구하는 것이었습니다.

### 6.4.1 CAMEL의 자율 협업

CAMEL이 자율 협업을 구현하는 초석은 두 가지 핵심 개념입니다: <strong>역할극 (Role-Playing)</strong>과 <strong>유도 프롬프트 (Inception Prompting)</strong>.

（1）역할극

CAMEL의 초기 설계에서 작업은 일반적으로 두 에이전트가 협업하여 완료합니다. 두 에이전트에게는 상호보완적이고 명확히 정의된 "역할"이 부여됩니다. 하나는 <strong>"AI 사용자" (AI User)</strong> 역할로 요구사항 제시, 지시 전달, 작업 단계 구상을 담당하고, 다른 하나는 <strong>"AI 어시스턴트" (AI Assistant)</strong> 역할로 지시에 따라 구체적인 작업을 실행하고 해결책을 제공합니다.

예를 들어 "주식 거래 전략 분석 도구 개발" 작업에서:

- <strong>AI 사용자</strong> 역할은 "시니어 주식 트레이더"가 될 수 있습니다. 시장을 알고 전략을 알지만 프로그래밍은 모릅니다.
- <strong>AI 어시스턴트</strong> 역할은 "뛰어난 Python 프로그래머"입니다. 프로그래밍에 능통하지만 주식 거래에 대해서는 아무것도 모릅니다.

이 설정을 통해 작업 해결 과정은 자연스럽게 두 "분야 간 전문가"의 대화로 변환됩니다. 트레이더가 전문 요구사항을 제시하고, 프로그래머가 이를 코드로 구현하며, 둘이 협업하여 어느 한쪽이 독립적으로 완료할 수 없는 복잡한 작업을 완료합니다.

（2）유도 프롬프트

역할을 설정하는 것만으로는 충분하지 않습니다. 인간의 지속적인 감독 없이 두 AI가 항상 "자신의 역할에 머물러" 효율적으로 공동 목표를 향해 나아갈 수 있도록 어떻게 보장할까요? 이것이 CAMEL의 가장 핵심적인 기술인 유도 프롬프트가 작동하는 부분입니다. "유도 프롬프트"는 대화 시작 전에 두 에이전트에게 각각 주입되는 정교하게 설계된 구조화된 초기 지시(System Prompt)입니다. 이 지시는 에이전트에게 심어진 "행동 강령"과 같으며, 일반적으로 다음 몇 가지 핵심 부분을 포함합니다:

- <strong>자신의 역할 명확화</strong>: 예: "당신은 시니어 주식 트레이더입니다..."
- <strong>협력자 역할 알리기</strong>: 예: "당신은 뛰어난 Python 프로그래머와 협업 중입니다..."
- <strong>공동 목표 정의</strong>: 예: "두 분의 공동 목표는 주식 거래 전략 분석 도구를 개발하는 것입니다."
- <strong>행동 제약 및 통신 프로토콜 설정</strong>: 가장 중요한 부분입니다. 예를 들어 지시는 AI 사용자에게 "한 번에 하나의 명확하고 구체적인 단계만 제시"하고, AI 어시스턴트에게는 "이전 단계를 완료하기 전에 더 많은 세부 사항을 추가로 묻지 말 것"을 요구하며, 동시에 응답 끝에 특정 표시(예: `<SOLUTION>`)를 사용하여 작업 완료를 표시하도록 규정합니다.

이러한 제약 조건들은 대화가 주제에서 벗어나거나 무효 루프에 빠지지 않고 고도로 구조화된 작업 중심 방식으로 전진하도록 보장합니다. 그림 6.3에서 볼 수 있습니다.

### 6.4.2 AI 과학 전자책

CAMEL 프레임워크의 역할극 능력을 이해하기 위해 실제 가치가 있는 협업 사례를 구축합니다: AI 심리학자와 AI 작가가 협업하여 "미루기 심리학"에 관한 단편 전자책을 함께 창작합니다. 이 사례는 CAMEL의 핵심 장점을 구현합니다. 두 에이전트가 각자의 전문 분야에서 강점을 발휘하여 단일 에이전트가 감당하기 어려운 복잡한 창작 작업을 협업으로 완료합니다.

（1）작업 설정

<strong>시나리오 설정</strong>: 일반 독자를 대상으로 한 미루기 심리학 과학 전자책 창작. 과학적 엄밀성과 좋은 가독성을 모두 갖추어야 합니다.

<strong>에이전트 역할</strong>:

- <strong>심리학자 (Psychologist)</strong>: 깊은 심리학 이론 기반을 갖추고, 인지 행동 과학, 신경과학 등 관련 분야에 정통하며, 전문적인 학문적 통찰과 실증 연구 지원을 제공할 수 있습니다.
- <strong>작가 (Writer)</strong>: 뛰어난 글쓰기 기술과 서술 능력을 보유하며, 복잡한 학문적 개념을 생생하고 이해하기 쉬운 문장으로 전환하고, 독자 경험과 콘텐츠 가독성을 중시합니다.

（2）협업 작업 정의

먼저 두 AI 전문가의 공동 목표를 명확히 해야 합니다. 내용이 풍부한 문자열 `task_prompt`를 통해 이 작업을 정의합니다.

```python
from colorama import Fore
from camel.societies import RolePlaying
from camel.utils import print_text_animated
from camel.models import ModelFactory
from camel.types import ModelPlatformType
from dotenv import load_dotenv
import os

load_dotenv()
LLM_API_KEY = os.getenv("LLM_API_KEY")
LLM_BASE_URL = os.getenv("LLM_BASE_URL")
LLM_MODEL = os.getenv("LLM_MODEL")

# 모델 생성, 여기서는 Qwen을 예로 들어 百炼 대형 모델 플랫폼 API 호출
model = ModelFactory.create(
    model_platform=ModelPlatformType.QWEN,
    model_type=LLM_MODEL,
    url=LLM_BASE_URL,
    api_key=LLM_API_KEY
)

# 협업 작업 정의
task_prompt = """
"미루기 심리학"에 관한 단편 전자책을 창작합니다. 심리학에 관심 있는 일반 대중을 대상으로 합니다.
요구사항:
1. 내용이 과학적으로 엄밀하고 실증 연구에 기반
2. 언어가 쉽고 이해하기 쉬우며 전문 용어를 최소화
3. 실용적인 개선 제안과 사례 분석 포함
4. 분량은 8000-10000자 수준
5. 구조가 명확하고 서론, 핵심 장, 결론 포함
"""

print(Fore.YELLOW + f"협업 작업:\n{task_prompt}\n")
```

`task_prompt`는 전체 협업의 "작업 명세서"입니다. 완료해야 할 목표일 뿐만 아니라, CAMEL이 이를 사용하여 "유도 프롬프트"를 생성하고 두 에이전트의 대화가 항상 이 핵심 목표를 중심으로 전개되도록 합니다.

（3）역할극 "사회" 초기화

다음으로 `RolePlaying` 세션 인스턴스를 생성합니다. 이는 CAMEL의 핵심 작업으로, 우리가 제공한 역할과 작업을 기반으로 이중 에이전트 협업 "사회"를 빠르게 구축합니다.

```python
# 역할극 세션 초기화
# AI 작가가 "user"로, 작문 구조와 요구사항 제시 담당
# AI 심리학자가 "assistant"로, 전문 지식과 콘텐츠 제공 담당
role_play_session = RolePlaying(
    assistant_role_name="심리학자",
    user_role_name="작가",
    task_prompt=task_prompt,
    model=model,
    with_task_specify=False, # 이 예에서는 주어진 task_prompt를 직접 사용
)

print(Fore.CYAN + f"구체적 작업 설명:\n{role_play_session.task_prompt}\n")
```

`RolePlaying`은 CAMEL이 제공하는 고급 API로 복잡한 프롬프트 엔지니어링을 캡슐화합니다. 두 역할의 이름과 작업만 전달하면 됩니다. CAMEL의 설계에서 `user` 역할은 대화의 "추진자"와 "요구 측"이고, `assistant` 역할은 "실행자"와 "방안 제공자"입니다. 따라서 구조 계획을 담당하는 "작가"를 `user_role_name`에, 전문 지식 제공을 담당하는 "심리학자"를 `assistant_role_name`에 배정합니다.

（4）자동화 대화 시작 및 실행

마지막으로 전체 대화 과정을 구동하는 루프를 작성하여 두 AI 전문가의 자동화 협업을 시작합니다.

```python
# 협업 대화 시작
chat_turn_limit, n = 30, 0
# init_chat()을 호출하여 AI가 생성한 초기 대화 메시지 가져오기
input_msg = role_play_session.init_chat()

while n < chat_turn_limit:
    n += 1
    # step() 메서드가 완전한 한 라운드의 대화를 구동, AI 사용자와 AI 어시스턴트 각각 한 번씩 발언
    assistant_response, user_response = role_play_session.step(input_msg)
    
    # 메시지가 반환되었는지 확인, 대화의 조기 종료 방지
    if assistant_response.msg is None or user_response.msg is None:
        break
    
    print_text_animated(Fore.BLUE + f"작가 (AI User):\n\n{user_response.msg.content}\n")
    print_text_animated(Fore.GREEN + f"심리학자 (AI Assistant):\n\n{assistant_response.msg.content}\n")
    
    # 작업 완료 표시 확인
    if "<CAMEL_TASK_DONE>" in user_response.msg.content or "<CAMEL_TASK_DONE>" in assistant_response.msg.content:
        print(Fore.MAGENTA + "✅ 전자책 창작 완료!")
        break
    
    # 어시스턴트의 응답을 다음 라운드 대화의 입력으로 사용
    input_msg = assistant_response.msg

print(Fore.YELLOW + f"총 {n} 라운드 협업 대화 진행")
```

이 `while` 루프는 자동화 협업의 핵심입니다. 대화는 `init_chat()` 메서드가 작업과 역할을 기반으로 자동으로 시작되며, 인간이 도입부를 작성할 필요가 없습니다. 루프의 각 단계는 `step()`을 호출하여 완전한 상호작용(작가가 요구사항 제시, 심리학자가 콘텐츠 제공)을 구동하고, 이전 라운드 심리학자의 출력을 다음 라운드의 입력으로 사용하여 연속적인 창작 체인을 형성합니다. 전체 과정은 미리 설정된 대화 라운드 상한에 도달하거나, 어느 에이전트가 작업 완료 표시 `<CAMEL_TASK_DONE>`을 출력할 때까지 계속됩니다.

（5）협업 프로세스 시연

위 코드를 실행하면 단조로운 질문답변이 아니라 인간 전문가 팀과 같은 고도로 구조화된 협업 프로세스가 자동으로 진행되는 것을 관찰할 수 있습니다. 전체 창작 과정은 자연스럽게 몇 가지 단계로 나뉩니다:

<strong>1단계 (약 1-5 라운드): 프레임 구축과 목표 정렬</strong> 대화 초기에 "작가" 에이전트가 먼저 주도자 역할을 맡아 전자책의 전체 구조와 장 배치에 대한 초기 구상을 제시합니다. 이후 "심리학자"가 전문적 관점에서 이 프레임을 검토하고 보완하여 핵심 학문 모듈(이론적 기초, 핵심 개념 등)이 빠지지 않도록 합니다. 협업 시작부터 최종 산출물에 대한 합의를 이룹니다.

<strong>2단계 (약 6-20 라운드): 핵심 콘텐츠 생성과 지식 번역</strong> 가장 효율적인 콘텐츠 창작 단계입니다. 협업 모드는 안정적인 "요청-응답" 루프로 변화합니다:

- <strong>심리학자</strong>: "시간 할인 이론", "실행 기능 결핍" 등 핵심 개념의 과학적 설명과 같은 "하드코어" 전문 지식을 제공하고 관련 실험 연구를 인용하여 관점을 지지합니다.
- <strong>작가</strong>: "번역가" 역할을 발휘하여 이러한 엄밀하지만 어려울 수 있는 학문적 개념을 생생하고 생활에 밀접한 사례로 전환합니다. 예를 들어 "뇌의 '현재 편향'" 개념을 "눈앞의 사탕만 신경 쓰고 장기적 건강을 고려하지 않는 버릇없는 아이"로 비유할 수 있습니다.

<strong>3단계 (약 21-25 라운드): 반복 최적화와 품질 보증</strong> 책의 주요 내용이 완성되면 대화의 중점이 기존 텍스트의 다듬기와 완성으로 이동합니다. 이때 두 에이전트의 역할이 미묘하게 변화합니다:

- <strong>작가</strong>: 글의 전체 유창성, 논리적 연결, 언어 스타일에 더 집중하며 "독자 경험" 관점에서 수정 제안을 합니다.
- <strong>심리학자</strong>: 다시 "팩트 체커" 역할을 맡아 번역과 다듬기 과정에서 핵심 지식의 과학적 정확성이 손실되지 않았는지 확인하고, 일부 관점에 더 강력한 실증 연구 지원을 보완합니다.

<strong>4단계 (마무리): 요약과 승화</strong> 마지막 몇 라운드의 대화에서 양측이 협업하여 실용적 제안 요약과 전체 책의 검토를 완료하며, 전자책이 명확하고 강력한 결말을 가져 독자에게 깊은 인상을 남기고 실질적인 가치를 제공합니다.

```
협업 작업:
"미루기 심리학"에 관한 단편 전자책 창작...

작가:
Instruction: 전자책의 "서론" 장을 위해 400-600자의 초안을 작성해 주세요...

심리학자:
Solution:

작가:
Instruction: 아래 서론 초안을 수정해 주세요...
.....
```

이제 세 가지 핵심 노드를 차례로 생성합니다.

（1） 이해 및 쿼리 노드

이 노드는 워크플로의 첫 번째 단계로, 사용자 의도를 이해하고 최적화된 검색 쿼리를 생성하는 역할입니다.

```python
def understand_query_node(state: SearchState) -> dict:
    """단계1: 사용자 쿼리를 이해하고 검색 키워드 생성"""
    user_message = state["messages"][-1].content
    
    understand_prompt = f"""사용자의 쿼리를 분석하세요: "{user_message}"
두 가지 작업을 완료하세요:
1. 사용자가 알고 싶은 것을 간결하게 요약
2. 검색 엔진에 가장 적합한 키워드 생성 (한국어/영어 모두 가능, 정확해야 함)

형식:
이해: [사용자 요구사항 요약]
검색어: [최적 검색 키워드]"""

    response = llm.invoke([SystemMessage(content=understand_prompt)])
    response_text = response.content
    
    # LLM 출력 파싱, 검색 키워드 추출
    search_query = user_message # 기본적으로 원본 쿼리 사용
    if "검색어:" in response_text:
        search_query = response_text.split("검색어:")[1].strip()
    
    return {
        "user_query": response_text,
        "search_query": search_query,
        "step": "understood",
        "messages": [AIMessage(content=f"다음을 검색하겠습니다: {search_query}")]
    }
```

이 노드는 구조화된 프롬프트를 통해 LLM에게 "의도 이해"와 "키워드 생성" 두 작업을 동시에 완료하도록 요청하고, 파싱된 전용 검색 키워드를 상태의 `search_query` 필드에 업데이트하여 다음 단계의 정확한 검색을 준비합니다.

（2）검색 노드

이 노드는 에이전트의 "도구 사용" 능력을 실행합니다. Tavily API를 호출하여 실제 인터넷 검색을 수행하고 기본적인 오류 처리 기능을 갖춥니다.

```python
def tavily_search_node(state: SearchState) -> dict:
    """단계2: Tavily API를 사용하여 실제 검색 수행"""
    search_query = state["search_query"]
    try:
        print(f"🔍 검색 중: {search_query}")
        response = tavily_client.search(
            query=search_query, search_depth="basic", max_results=5, include_answer=True
        )
        # ... (검색 결과 처리 및 형식화) ...
        search_results = ... # 형식화된 결과 문자열
        
        return {
            "search_results": search_results,
            "step": "searched",
            "messages": [AIMessage(content="✅ 검색 완료! 답변 정리 중...")]
        }
    except Exception as e:
        # ... (오류 처리) ...
        return {
            "search_results": f"검색 실패: {e}",
            "step": "search_failed",
            "messages": [AIMessage(content="❌ 검색에 문제가 발생했습니다...")]
        }
```

이 노드는 `tavily_client.search`를 통해 실제 API 호출을 시작합니다. `try...except` 블록으로 감싸 가능한 예외를 캡처합니다. 검색이 실패하면 `step` 상태를 `"search_failed"`로 업데이트하고, 이 상태가 다음 노드에서 대체 방안을 트리거하는 데 사용됩니다.

（3）답변 노드

마지막 답변 노드는 이전 단계의 검색 성공 여부에 따라 다른 답변 전략을 선택하는 탄력성을 갖추고 있습니다.

```python
def generate_answer_node(state: SearchState) -> dict:
    """단계3: 검색 결과를 기반으로 최종 답변 생성"""
    if state["step"] == "search_failed":
        # 검색 실패 시 대체 전략 실행, LLM 자체 지식으로 답변
        fallback_prompt = f"검색 API를 일시적으로 사용할 수 없습니다, 지식을 기반으로 사용자 질문에 답변하세요:\n사용자 질문: {state['user_query']}"
        response = llm.invoke([SystemMessage(content=fallback_prompt)])
    else:
        # 검색 성공, 검색 결과를 기반으로 답변 생성
        answer_prompt = f"""다음 검색 결과를 기반으로 사용자에게 완전하고 정확한 답변을 제공하세요:
사용자 질문: {state['user_query']}
검색 결과:\n{state['search_results']}
검색 결과를 종합하여 정확하고 유용한 답변을 제공하세요..."""
        response = llm.invoke([SystemMessage(content=answer_prompt)])
    
    return {
        "final_answer": response.content,
        "step": "completed",
        "messages": [AIMessage(content=response.content)]
    }
```

이 노드는 `state["step"]`의 값을 확인하여 조건 로직을 실행합니다. 검색이 실패하면 LLM의 내부 지식을 사용하여 답변하고 사용자에게 상황을 알립니다. 검색이 성공하면 실시간 검색 결과가 포함된 프롬프트를 사용하여 시의성 있고 근거 있는 답변을 생성합니다.

（4）그래프 구축

모든 노드를 연결합니다.

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

def create_search_assistant():
    workflow = StateGraph(SearchState)
    
    # 노드 추가
    workflow.add_node("understand", understand_query_node)
    workflow.add_node("search", tavily_search_node)
    workflow.add_node("answer", generate_answer_node)
    
    # 선형 프로세스 설정
    workflow.add_edge(START, "understand")
    workflow.add_edge("understand", "search")
    workflow.add_edge("search", "answer")
    workflow.add_edge("answer", END)
    
    # 그래프 컴파일
    memory = InMemorySaver()
    app = workflow.compile(checkpointer=memory)
    return app
```

### 6.5.3 LangGraph의 장점과 한계 분석

어떤 기술 프레임워크든 특정 적용 시나리오와 설계 트레이드오프가 있습니다. 이 절에서는 LangGraph의 핵심 장점과 실제 적용에서 직면할 수 있는 한계를 객관적으로 분석합니다.

（1）장점

- 지능형 검색 어시스턴트 사례에서 볼 수 있듯이, LangGraph는 완전한 실시간 질문답변 프로세스를 상태, 노드, 엣지로 구성된 "순서도"로 명시적으로 정의합니다. 이 설계의 가장 큰 장점은 <strong>높은 제어 가능성과 예측 가능성</strong>입니다. 개발자는 에이전트의 각 단계 행동을 정확하게 계획할 수 있으며, 이는 높은 신뢰성과 감사 가능성이 필요한 프로덕션급 애플리케이션 구축에 매우 중요합니다. 가장 강력한 기능은 <strong>순환(Cycles)에 대한 기본 지원</strong>입니다. 조건부 엣지를 통해 "반성-수정" 루프를 쉽게 구축할 수 있습니다. 예를 들어 검색이 실패하면 대체 방안으로 돌아가는 경로를 설계할 수 있습니다. 이것이 자기 최적화와 내결함성 능력을 갖춘 에이전트 구축의 핵심입니다.

- 또한 각 노드가 독립적인 Python 함수이므로 <strong>높은 모듈성</strong>을 제공합니다. 동시에 프로세스에 인간 검토를 기다리는 노드를 삽입하는 것이 매우 직접적이어서 신뢰할 수 있는 "인간-기계 협업"(Human-in-the-loop) 구현을 위한 견고한 기반을 제공합니다.

（2）한계

- 대화 기반 프레임워크에 비해 LangGraph는 개발자가 더 많은 <strong>보일러플레이트 코드(Boilerplate)</strong>를 작성해야 합니다. 상태, 노드, 엣지 정의 등 일련의 작업으로 단순한 작업의 경우 개발 과정이 더 번거롭습니다. 개발자는 "무엇을 하는가(what)"뿐만 아니라 "어떻게 프로세스를 제어하는가(how)"를 더 많이 생각해야 합니다. 워크플로가 미리 정의되어 있어 LangGraph의 동작은 제어 가능하지만, 대화형 에이전트의 동적이고 <strong>"창발적"인 상호작용</strong>이 부족합니다. 강점은 확정적이고 신뢰할 수 있는 프로세스를 실행하는 것이지, 개방적이고 예측 불가능한 사회적 협업을 시뮬레이션하는 것이 아닙니다.

- 디버깅 과정도 도전이 있습니다. 프로세스가 대화 히스토리보다 명확하지만 문제는 여러 단계에서 발생할 수 있습니다: 특정 노드 내부 로직 오류, 노드 간 전달되는 상태 데이터의 변형, 또는 엣지 전환 조건 판단 오류. 이는 개발자가 전체 그래프의 실행 메커니즘에 대한 전역적인 이해를 갖출 것을 요구합니다.

## 6.6 본장 소결

이 장에서는 현재 가장 최첨단의 에이전트 프레임워크를 사례 형태로 직접 체험했습니다.

각 프레임워크마다 에이전트 구축에 대한 자체적인 접근 방식이 있다는 것을 알 수 있습니다:

- <strong>AutoGen</strong>은 복잡한 협업을 여러 역할이 참여하는 자동화 가능한 "그룹 채팅"으로 추상화하며, 핵심은 "대화로 협업을 주도한다"는 것입니다.
- <strong>AgentScope</strong>는 산업급 애플리케이션의 견고성과 확장성에 초점을 맞추며, 높은 동시성, 분산 다중 에이전트 시스템 구축을 위한 견고한 엔지니어링 기반을 제공합니다.
- <strong>CAMEL</strong>은 경량급 "역할극"과 "유도 프롬프트" 패러다임으로 최소한의 코드로 두 전문가 에이전트 간의 깊고 자율적인 협업을 어떻게 이끌어내는지 보여줍니다.
- <strong>LangGraph</strong>는 더 하위 수준의 "상태 기계" 모델로 돌아가, 명시적인 그래프 구조를 통해 개발자에게 워크플로에 대한 정확한 제어권을 부여합니다. 특히 순환 능력은 반성하고 수정할 수 있는 에이전트 구축의 길을 열어줍니다.

이러한 프레임워크에 대한 깊은 분석을 통해 설계 트레이드오프를 추출할 수 있습니다: <strong>"창발적 협업"과 "명시적 제어" 사이의 선택</strong>. AutoGen과 CAMEL은 에이전트의 "역할"과 "목표"를 정의하는 것에 더 많이 의존하여 복잡한 협업 행동이 간단한 대화 규칙에서 "창발"하도록 합니다. 이 방식은 인간의 상호작용 모델에 더 가깝지만 때로는 예측하고 디버그하기 어렵습니다. LangGraph는 개발자가 각 단계와 전환 조건을 명시적으로 정의하도록 요구하여, "창발"의 놀라움을 어느 정도 희생하는 대신 높은 신뢰성, 제어 가능성, 관찰 가능성을 얻습니다. 동시에 AgentScope는 동등하게 중요한 두 번째 차원인 <strong>엔지니어링</strong>을 드러냅니다. 어떤 협업 패러다임을 선택하든 실험적 프로토타입에서 프로덕션 애플리케이션으로 전환하려면 동시성, 내결함성, 분산 배포 등의 엔지니어링 도전에 직면해야 합니다. AgentScope는 바로 이러한 문제를 해결하기 위해 탄생했으며, "실행 가능"에서 "안정적으로 서비스 가능"으로의 핵심 도약을 대표합니다.

요약하자면, 에이전트 구축 방법은 하나뿐이 아닙니다. 이 장에서 탐구한 프레임워크 설계 철학을 깊이 이해하면 더 뛰어난 "도구 사용자"가 될 뿐만 아니라 프레임워크 설계의 다양한 장단점과 트레이드오프를 이해할 수 있습니다.

다음 장에서는 이 튜토리얼의 핵심 내용에 들어가 처음부터 자체 에이전트 프레임워크를 직접 구축하여 모든 이론과 실천을 통합합니다.

   <strong>애플리케이션 B</strong>: 과학 연구 논문 보조 작성 플랫폼으로 "연구원 에이전트"와 "작가 에이전트"가 깊이 협업하여 문헌 검토, 실험 설계, 데이터 분석, 논문 작성을 함께 완료해야 합니다. 에이전트가 여러 라운드의 심층 토론을 수행하고 자율적으로 작업을 진행할 수 있어야 합니다.

   <strong>애플리케이션 C</strong>: 금융 위험 통제 승인 시스템으로 엄격한 프로세스에 따라 대출 신청을 처리해야 합니다: 자료 검토 → 위험 평가 → 한도 계산 → 규정 준수 검사 → 인간 검토 → 최종 결정. 각 단계에는 명확한 판단 기준과 분기 로직이 있으며, 프로세스가 추적 가능하고 감사 가능해야 합니다.

### 6.4.3 CAMEL의 장점과 한계 분석

전자책 창작 사례를 통해 CAMEL 프레임워크의 독특한 역할극 패러다임을 깊이 체험했습니다. 이제 이 설계 이념의 장점과 한계를 객관적으로 분석하여 실제 프로젝트에서 현명한 기술 선택을 할 수 있도록 합니다.

（1）장점

CAMEL의 가장 큰 장점은 "경량 아키텍처, 중점 프롬프트"의 설계 철학에 있습니다. AutoGen의 복잡한 대화 관리나 AgentScope의 분산 아키텍처에 비해, CAMEL은 정교하게 설계된 초기 프롬프트만으로 고품질의 에이전트 협업을 구현할 수 있습니다. 이 자연스럽게 창발되는 협업 행동은 종종 하드코딩된 워크플로보다 더 유연하고 효율적입니다.

주목할 점은 CAMEL 프레임워크가 빠른 발전과 진화를 겪고 있다는 것입니다. [GitHub 저장소](https://github.com/camel-ai/camel)를 보면 CAMEL이 이미 단순한 이중 에이전트 협업 프레임워크를 훨씬 넘어 현재 다음 기능을 갖추고 있습니다:

- <strong>멀티모달 능력</strong>: 텍스트, 이미지, 오디오 등 다양한 모달의 에이전트 협업 지원
- <strong>도구 통합</strong>: 검색, 계산, 코드 실행 등 풍부한 내장 도구 라이브러리
- <strong>모델 적응</strong>: OpenAI, Anthropic, Google, 오픈소스 모델 등 다양한 LLM 백엔드 지원
- <strong>생태계 연동</strong>: LangChain, CrewAI, AutoGen 등 주류 프레임워크와 상호 운용성 구현

（2）주요 한계

1. 프롬프트 엔지니어링에 대한 높은 의존성

CAMEL의 성공은 초기 프롬프트의 품질에 크게 달려 있습니다. 이는 몇 가지 도전을 가져옵니다:

- <strong>프롬프트 설계 문턱</strong>: 목표 도메인과 LLM의 행동 특성을 깊이 이해해야 합니다
- <strong>디버깅 복잡성</strong>: 협업 효과가 좋지 않을 때 역할 정의, 작업 설명, 상호작용 규칙 중 어디에 문제가 있는지 파악하기 어렵습니다
- <strong>일관성 도전</strong>: 다른 LLM은 동일한 프롬프트를 다르게 이해할 수 있습니다

2. 협업 규모의 제한

CAMEL이 이중 에이전트 협업에서 뛰어나지만 대규모 다중 에이전트 시나리오 처리에서 도전에 직면합니다:

- <strong>대화 관리</strong>: AutoGen과 같은 복잡한 대화 라우팅 메커니즘이 없습니다
- <strong>상태 동기화</strong>: AgentScope와 같은 분산 상태 관리 능력이 없습니다
- <strong>충돌 해결</strong>: 여러 에이전트가 의견이 다를 때 효과적인 중재 메커니즘이 부족합니다

3. 작업 적용 범위의 경계

CAMEL은 깊은 협업과 창의적 사고가 필요한 작업에 특히 적합하지만 일부 시나리오에서는 최적 선택이 아닐 수 있습니다:

- <strong>엄격한 프로세스 제어</strong>: 정확한 단계 제어가 필요한 작업의 경우 LangGraph의 그래프 구조가 더 적합합니다
- <strong>대규모 동시성</strong>: AgentScope의 메시지 기반 아키텍처가 고동시성 시나리오에서 더 유리합니다
- <strong>복잡한 의사결정 트리</strong>: AutoGen의 그룹 채팅 모드가 다자 의사결정 시나리오에서 더 유연합니다

전반적으로 CAMEL은 독특하고 우아한 다중 에이전트 협업 패러다임을 대표합니다. "인간 중심"의 역할극 설계를 통해 복잡한 시스템 엔지니어링 문제를 직관적인 인간 협업 모델로 전환합니다. 생태계의 지속적인 완비와 기능 확장으로 CAMEL은 지능형 협업 시스템 구축의 중요한 선택지 중 하나가 되고 있습니다.

## 6.5 프레임워크 4: LangGraph

### 6.5.1 LangGraph 구조 정리

LangChain 생태계의 중요한 확장인 LangGraph는 에이전트 프레임워크 설계의 완전히 새로운 방향을 대표합니다. 앞서 소개한 "대화" 기반 프레임워크(AutoGen, CAMEL 등)와 달리, LangGraph는 에이전트의 실행 흐름을 <strong>상태 기계(State Machine)</strong>로 모델링하고 이를 <strong>유향 그래프(Directed Graph)</strong>로 표현합니다. 이 패러다임에서 그래프의 <strong>노드(Nodes)</strong>는 구체적인 계산 단계(LLM 호출, 도구 실행 등)를 나타내고, <strong>엣지(Edges)</strong>는 한 노드에서 다른 노드로의 전환 로직을 정의합니다. 이 설계의 혁신적인 점은 자연스럽게 순환을 지원하여, 반복, 반성, 자기 수정이 가능한 복잡한 에이전트 워크플로 구축을 전례 없이 직관적이고 간단하게 만든다는 것입니다.

LangGraph를 이해하려면 먼저 세 가지 기본 구성 요소를 파악해야 합니다.

<strong>첫째, 전역 상태(State)</strong>. 전체 그래프 실행 과정은 공유 상태 객체를 중심으로 진행됩니다. 이 상태는 일반적으로 Python의 `TypedDict`로 정의되며 대화 히스토리, 중간 결과, 반복 횟수 등 추적해야 하는 모든 정보를 포함할 수 있습니다. 모든 노드가 이 중앙 상태를 읽고 업데이트할 수 있습니다.

```python
from typing import TypedDict, List

# 전역 상태의 데이터 구조 정의
class AgentState(TypedDict):
    messages: List[str]      # 대화 히스토리
    current_task: str        # 현재 작업
    final_answer: str        # 최종 답변
    # ... 추적이 필요한 다른 상태들
```

<strong>둘째, 노드(Nodes)</strong>. 각 노드는 현재 상태를 입력으로 받아 업데이트된 상태를 출력으로 반환하는 Python 함수입니다. 노드는 구체적인 작업을 수행하는 단위입니다.

```python
# "플래너" 노드 함수 정의
def planner_node(state: AgentState) -> AgentState:
    """현재 작업을 기반으로 계획을 수립하고 상태를 업데이트합니다."""
    current_task = state["current_task"]
    # ... LLM을 호출하여 계획 생성 ...
    plan = f"작업 '{current_task}'을 위해 생성된 계획..."
    
    # 새 메시지를 상태에 추가
    state["messages"].append(plan)
    return state

# "실행자" 노드 함수 정의
def executor_node(state: AgentState) -> AgentState:
    """최신 계획을 실행하고 상태를 업데이트합니다."""
    latest_plan = state["messages"][-1]
    # ... 계획을 실행하고 결과 얻기 ...
    result = f"계획 '{latest_plan}'의 실행 결과..."
    
    state["messages"].append(result)
    return state
```

<strong>셋째, 엣지(Edges)</strong>. 엣지는 노드를 연결하고 워크플로의 방향을 정의합니다. 가장 단순한 엣지는 일반 엣지로, 한 노드의 출력이 항상 다른 고정 노드로 흐르도록 지정합니다. LangGraph의 가장 강력한 기능은 <strong>조건부 엣지(Conditional Edges)</strong>입니다. 함수를 통해 현재 상태를 판단한 후 다음 단계로 어느 노드로 이동할지 동적으로 결정합니다. 이것이 루프와 복잡한 로직 분기를 구현하는 핵심입니다.

```python
def should_continue(state: AgentState) -> str:
    """조건부 함수: 상태를 기반으로 다음 라우팅 결정."""
    # 메시지가 3개 미만이면 계속 계획이 필요하다고 가정
    if len(state["messages"]) < 3:
        # 반환 문자열은 조건부 엣지 추가 시 정의한 키와 일치해야 함
        return "continue_to_planner"
    else:
        state["final_answer"] = state["messages"][-1]
        return "end_workflow"
```

상태, 노드, 엣지를 정의한 후 레고 블록처럼 이들을 조립하여 실행 가능한 워크플로를 만들 수 있습니다.

```python
from langgraph.graph import StateGraph, END

# 상태 그래프를 초기화하고 정의한 상태 구조를 바인딩
workflow = StateGraph(AgentState)

# 노드 함수를 그래프에 추가
workflow.add_node("planner", planner_node)
workflow.add_node("executor", executor_node)

# 그래프의 진입점 설정
workflow.set_entry_point("planner")

# 일반 엣지 추가, planner와 executor 연결
workflow.add_edge("planner", "executor")

# 조건부 엣지 추가, 동적 라우팅 구현
workflow.add_conditional_edges(
    # 시작 노드
    "executor",
    # 판단 함수
    should_continue,
    # 라우팅 매핑: 판단 함수의 반환값을 목표 노드에 매핑
    {
        "continue_to_planner": "planner", # "continue_to_planner"를 반환하면 planner 노드로 돌아감
        "end_workflow": END               # "end_workflow"를 반환하면 프로세스 종료
    }
)

# 그래프를 컴파일하여 실행 가능한 애플리케이션 생성
app = workflow.compile()

# 그래프 실행
inputs = {"current_task": "최근 AI 업계 뉴스 분석", "messages": []}
for event in app.stream(inputs):
    print(event)
```

### 6.5.2 3단계 질문답변 어시스턴트

LangGraph의 핵심 개념을 이해한 후 실전 사례를 통해 학습한 내용을 공고히 합니다. 사용자 질문에 명확하고 고정된 3단계 프로세스로 답변하는 간소화된 질문답변 어시스턴트를 구축합니다:

1. <strong>이해 (Understand)</strong>: 먼저 사용자의 쿼리 의도를 분석합니다.
2. <strong>검색 (Search)</strong>: 그런 다음 의도와 관련된 정보를 검색하는 것을 시뮬레이션합니다.
3. <strong>답변 (Answer)</strong>: 마지막으로 의도와 검색된 정보를 기반으로 최종 답변을 생성합니다.

이 사례는 상태 정의, 노드 생성, 선형 연결하여 완전한 워크플로를 구성하는 방법을 명확하게 보여줍니다. 코드를 상태 정의, 노드 생성, 그래프 구축, 애플리케이션 실행의 네 가지 핵심 단계로 분해합니다.

（1）전역 상태 정의

먼저 전체 워크플로를 관통하는 전역 상태를 정의해야 합니다. <strong>이는 그래프의 각 노드 사이에 전달되는 공유 데이터 구조로 워크플로의 지속 컨텍스트 역할을 합니다.</strong> 각 노드는 이 구조의 데이터를 읽고 업데이트할 수 있습니다.

```python
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class SearchState(TypedDict):
    messages: Annotated[list, add_messages]
    user_query: str      # LLM이 이해한 후의 사용자 요구사항 요약
    search_query: str    # Tavily API용으로 최적화된 검색 쿼리
    search_results: str  # Tavily 검색 반환 결과
    final_answer: str    # 최종 생성된 답변
    step: str            # 현재 단계 표시
```

`SearchState` `TypedDict`를 생성하여 상태 객체의 명확한 데이터 스키마를 정의했습니다. 핵심 설계는 `user_query`와 `search_query` 필드를 동시에 포함한다는 것입니다. 이를 통해 에이전트가 먼저 사용자의 자연어 질문을 검색 엔진에 더 적합한 정제된 키워드로 최적화하여 검색 결과의 품질을 크게 향상시킬 수 있습니다.

（2）워크플로 노드 정의

상태 구조를 정의한 후 다음 단계는 워크플로를 구성하는 각 노드를 생성하는 것입니다. LangGraph에서 각 노드는 구체적인 작업을 수행하는 Python 함수입니다. 이 함수들은 현재 상태 객체를 입력으로 받고 업데이트된 필드를 포함하는 딕셔너리를 반환합니다.

노드 정의를 시작하기 전에 먼저 프로젝트 초기화 설정(환경 변수 로드, 대형 언어 모델 인스턴스화)을 완료합니다.
