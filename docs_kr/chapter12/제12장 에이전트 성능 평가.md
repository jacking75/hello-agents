# 제12장 에이전트 성능 평가

앞선 장들에서 우리는 HelloAgents 프레임워크의 핵심 기능을 구축하며, 다양한 에이전트 패러다임, 도구 시스템, 메모리 메커니즘, 강화학습 훈련 등을 구현했습니다. 에이전트 시스템을 구축할 때, 우리는 또 하나의 핵심 문제를 해결해야 합니다: <strong>에이전트의 성능을 어떻게 객관적으로 평가할 것인가?</strong> 구체적으로, 다음 질문들에 답해야 합니다:

1. 에이전트가 기대하는 능력을 갖추고 있는가?
2. 다양한 작업에서의 성능은 어떤가?
3. 다른 에이전트와 비교했을 때 어느 수준인가?

이 장에서는 HelloAgents에 <strong>성능 평가 시스템(Evaluation System)</strong>을 추가합니다. 에이전트 평가의 이론적 기반을 깊이 이해하고, 평가 도구를 구현해 보겠습니다.

## 12.1 에이전트 평가 기초

### 12.1.1 에이전트 평가가 필요한 이유

현재 우리의 SimpleAgent는 이미 강력한 추론 및 도구 호출 능력을 갖추고 있습니다. 전형적인 사용 시나리오를 살펴보겠습니다:

```python
from hello_agents import SimpleAgent, HelloAgentsLLM
from hello_agents.tools import SearchTool

# LLM과 에이전트 생성
llm = HelloAgentsLLM()

# 도구 사용을 강조하는 시스템 프롬프트 생성
system_prompt = """당신은 검색 도구를 사용하여 최신 정보를 가져올 수 있는 AI 어시스턴트입니다.

정보를 검색해야 할 때는 다음 형식을 사용하세요:
[TOOL_CALL:search:검색키워드]

예시:
- [TOOL_CALL:search:최신AI뉴스]
- [TOOL_CALL:search:Python프로그래밍튜토리얼]

질문에 답하기 전에 검색 도구를 사용하여 최신 정보를 가져오세요."""

agent = SimpleAgent(name="AI어시스턴트", llm=llm, system_prompt=system_prompt)

# 검색 도구 추가
agent.add_tool(SearchTool())

# 예시: 검색 도구를 사용하여 질문에 답하기
response = agent.run("최신 AI 기술 발전 트렌드는 무엇인가요?")
print(f"\n답변：{response}")
```

이 에이전트는 정상적으로 동작하지만, 핵심 문제에 직면합니다: 성능을 어떻게 객관적으로 평가할 것인가? 프롬프트를 최적화하거나 LLM 모델을 교체한 후, 실제로 개선이 있었는지 어떻게 알 수 있을까요? 프로덕션 환경에 배포하기 전에 에이전트의 신뢰성을 어떻게 보장할 수 있을까요? 이러한 문제들은 모두 체계적인 평가를 통해 해결해야 합니다.

에이전트 평가의 핵심 가치는 에이전트의 능력을 측정하는 표준화된 방법을 제공하는 데 있습니다. 평가를 통해 구체적인 수치 지표로 에이전트의 성능을 정량화하고, 다양한 설계 방안의 장단점을 객관적으로 비교하며, 특정 시나리오에서 에이전트의 약점을 적시에 발견하고, 사용자에게 에이전트의 신뢰성을 증명할 수 있습니다.

전통적인 소프트웨어 테스트와 달리, 에이전트 평가는 고유한 도전에 직면합니다. 첫째, 출력의 불확실성: 같은 질문에 여러 개의 정답이 있을 수 있어 단순한 정답/오답으로 판단하기 어렵습니다. 둘째, 평가 기준의 다양성: 다양한 작업에는 다양한 평가 방법이 필요하며, 도구 호출은 함수 시그니처를 확인해야 하고, 질의응답 작업은 의미적 유사도를 평가해야 합니다. 마지막으로, 높은 평가 비용: 매번 평가마다 대량의 API 호출이 필요하여 비용이 수백 위안 이상에 달할 수 있습니다.

이러한 도전에 대응하기 위해 학계와 업계에서는 다수의 표준화된 <strong>평가 기준(Benchmark)</strong>을 제안했습니다. 이러한 기준은 통일된 데이터셋, 평가 지표, 채점 방법을 제공하여 동일한 기준 하에서 다양한 에이전트 시스템을 평가하고 비교할 수 있게 합니다.

### 12.1.2 주요 평가 기준 개요

에이전트 평가 분야에서는 이미 영향력 있는 여러 벤치마크 테스트가 등장했습니다. 주요 평가 기준과 지표들을 소개합니다:

<strong>（1）도구 호출 능력 평가</strong>

도구 호출은 에이전트의 핵심 능력 중 하나입니다. 에이전트는 사용자의 의도를 파악하고, 적절한 도구를 선택하며, 함수 호출을 올바르게 구성해야 합니다. 관련 평가 기준에는 다음이 포함됩니다:

- <strong>BFCL (Berkeley Function Calling Leaderboard)</strong><sup>[1]</sup>: UC Berkeley에서 출시, 1120개 이상의 테스트 샘플 포함, simple, multiple, parallel, irrelevance의 4가지 카테고리 포함, AST 매칭 알고리즘으로 평가, 데이터셋 규모가 적절하며 커뮤니티가 활발합니다.
- <strong>ToolBench</strong><sup>[2]</sup>: 칭화대학교에서 출시, 16000개 이상의 실제 API 호출 시나리오 포함, 현실 세계의 복잡한 도구 사용 시나리오를 포함합니다.
- <strong>API-Bank</strong><sup>[3]</sup>: Microsoft Research에서 출시, 53개의 일반적인 API 도구 포함, 에이전트의 API 문서 이해 및 호출 능력 평가에 특화되어 있습니다.

<strong>（2）범용 능력 평가</strong>

실제 세계 작업에서 에이전트의 종합적인 성능을 평가하며, 다단계 추론, 지식 활용, 멀티모달 이해 등의 능력을 포함합니다:

- <strong>GAIA (General AI Assistants)</strong><sup>[4]</sup>: Meta AI와 Hugging Face가 공동 출시, 466개의 실제 세계 질문 포함, Level 1/2/3의 세 가지 난이도 수준으로 구분, 다단계 추론, 도구 사용, 파일 처리, 웹 브라우징 등의 능력을 평가, 준정확 매칭(Quasi Exact Match) 알고리즘 사용, 작업이 실제적이며 종합성이 강합니다.
- <strong>AgentBench</strong><sup>[5]</sup>: 칭화대학교에서 출시, 8개의 서로 다른 영역의 작업 포함, 에이전트의 범용 능력을 종합적으로 평가합니다.
- <strong>WebArena</strong><sup>[6]</sup>: CMU에서 출시, 실제 웹 환경에서 에이전트의 작업 완료 능력과 웹 상호작용 능력을 평가합니다.

<strong>（3）멀티 에이전트 협업 평가</strong>

여러 에이전트가 협력하는 능력을 평가합니다:

- <strong>ChatEval</strong><sup>[7]</sup>: 멀티 에이전트 대화 시스템의 품질을 평가합니다.
- <strong>SOTOPIA</strong><sup>[8]</sup>: 사회적 시나리오에서 에이전트의 상호작용 능력을 평가합니다.
- <strong>맞춤형 협업 시나리오</strong>: 구체적인 응용 시나리오에 맞게 설계된 평가 작업입니다.

<strong>（4）일반적인 평가 지표</strong>

다양한 기준에서는 서로 다른 평가 지표를 사용하며, 일반적인 지표들은 다음과 같습니다:

- <strong>정확도 지표</strong>: Accuracy(정확률), Exact Match(정확 매칭), F1 Score(F1 점수), 답변의 정확성을 측정합니다.
- <strong>효율성 지표</strong>: Response Time(응답 시간), Token Usage(토큰 사용량), 실행 효율성을 측정합니다.
- <strong>견고성 지표</strong>: Error Rate(오류율), Failure Recovery(장애 복구), 내결함성을 측정합니다.
- <strong>협업 지표</strong>: Communication Efficiency(통신 효율성), Task Completion(작업 완료도), 협업 효과를 측정합니다.

### 12.1.3 HelloAgents 평가 체계 설계

학습 곡선과 실용성을 고려하여, 이 장에서는 다음 평가 시나리오에 중점을 둡니다:

1. <strong>BFCL</strong>: 도구 호출 능력 평가
   - 선택 이유: 데이터셋 규모가 적절하고, 평가 지표가 명확하며, 커뮤니티가 활발합니다
   - 적용 시나리오: 에이전트의 함수 호출 정확성 평가

2. <strong>GAIA</strong>: 범용 AI 어시스턴트 능력 평가
   - 선택 이유: 작업이 실제적이고, 난이도가 분류되어 있으며, 종합성이 강합니다
   - 적용 시나리오: 에이전트의 종합적인 문제 해결 능력 평가

3. <strong>데이터 생성 품질 평가</strong>: LLM 생성 데이터 품질 평가
   - 선택 이유: 이 사례를 통해 에이전트를 사용하여 데이터를 생성하고, 데이터의 품질을 평가하는 전체 과정을 완전히 경험할 수 있습니다.
   - 적용 시나리오: 생성된 훈련 데이터, 테스트 데이터의 품질 평가
   - 평가 방법: LLM Judge, Win Rate, 수동 검증

## 12.2 BFCL: 도구 호출 능력 평가

### 12.2.1 BFCL 기준 소개

BFCL (Berkeley Function Calling Leaderboard)은 캘리포니아 대학교 버클리캠퍼스에서 출시한 함수 호출 능력 평가 기준입니다<sup>[1]</sup>. 에이전트 시스템에서 도구 호출(Tool Calling)은 핵심 능력 중 하나입니다. 에이전트는 다음 작업을 완료해야 합니다:

1. <strong>작업 요구사항 이해</strong>: 사용자의 자연어 설명에서 핵심 정보 추출
2. <strong>적절한 도구 선택</strong>: 사용 가능한 도구 집합에서 가장 적합한 도구 선택
3. <strong>함수 호출 구성</strong>: 함수 이름과 매개변수를 올바르게 입력
4. <strong>복잡한 시나리오 처리</strong>: 다중 함수 호출, 병렬 호출 등 고급 시나리오 지원

BFCL 기준은 네 가지 평가 카테고리를 포함하며, 난이도가 점진적으로 증가합니다. 가장 기본적인 단일 함수 호출(Simple)부터 시작하여, 여러 함수를 호출해야 하는 시나리오(Multiple), 여러 함수를 병렬로 호출해야 하는 복잡한 시나리오(Parallel), 마지막으로 함수 호출이 필요한지 여부를 판단해야 하는 시나리오(Irrelevance)까지 단계적으로 증가합니다. 이 네 가지 카테고리는 에이전트가 실제 응용에서 마주칠 수 있는 다양한 도구 호출 시나리오를 포함하며, 표 12.1과 같습니다:

<strong>（1）BFCL 데이터셋 구조</strong>

BFCL 데이터셋은 JSON 형식을 사용하며, 각 테스트 샘플은 다음 필드를 포함합니다:

```json
{
  "id": "simple_001",
  "question": "What's the weather like in Beijing today?",
  "function": [
    {
      "name": "get_weather",
      "description": "Get the current weather for a location",
      "parameters": {
        "type": "object",
        "properties": {
          "location": {
            "type": "string",

          "description": "The city name"
          }
        },
        "required": ["location"]
      }
    }
  ],
  "ground_truth": [
    {
      "name": "get_weather",
      "arguments": {
        "location": "Beijing"
      }
    }
  ]
}
```

<strong>주요 필드 설명:</strong>

- `question`: 사용자의 자연어 요청
- `function`: 사용 가능한 함수 목록 (함수 시그니처 및 설명 포함)
- `ground_truth`: 정답 (기대하는 함수 호출)

<strong>（2）AST 매칭 설명</strong>

BFCL은 <strong>AST 매칭(Abstract Syntax Tree Matching)</strong>을 핵심 평가 알고리즘으로 사용합니다. 따라서 아래에서 평가 전략에 대해 살펴볼 수 있습니다.

BFCL은 단순한 문자열 매칭이 아닌 추상 구문 트리(AST)를 이용한 스마트 매칭을 사용합니다. AST 매칭의 핵심 아이디어는: <strong>함수 호출을 구문 트리로 파싱한 다음, 트리의 구조와 노드 값을 비교하는 것</strong>입니다.

예측된 함수 호출 $P$와 정답 $G$가 주어지면, AST 매칭 함수는 다음과 같이 정의됩니다:

$$
\text{AST\_Match}(P, G) = \begin{cases}
1 & \text{if } \text{AST}(P) \equiv \text{AST}(G) \
0 & \text{otherwise}
\end{cases}
$$

여기서 $\text{AST}(x)$는 함수 호출을 추상 구문 트리로 파싱하는 것을 나타내며, $\equiv$는 구문 트리가 동등함을 나타냅니다.

두 구문 트리가 동등하려면 세 가지 핵심 조건을 충족해야 합니다: 함수 이름이 완전히 일치해야 하고(정확 매칭), 매개변수 키-값 쌍 집합이 동일해야 하며(순서 무시), 각 매개변수의 값이 의미적으로 동등해야 합니다(예: `2+3`은 `5`와 동등). 구체적인 매칭 과정에서, 함수 이름 매칭은 문자열 정확 매칭이 필요합니다. 예를 들어 `get_weather`와 `get_temperature`는 다른 함수로 간주됩니다. 매개변수 매칭은 AST를 사용하여 스마트하게 비교하며, 매개변수 순서가 달라도 허용(`f(a=1, b=2)`는 `f(b=2, a=1)`과 동등), 동등한 표현식 허용(`f(x=2+3)`은 `f(x=5)`와 동등), 다른 문자열 표현 허용(`f(s="hello")`는 `f(s='hello')`와 동등)합니다. 다중 함수 호출 시나리오에서 매칭 알고리즘은 동일한 수의 함수 호출을 요구하며, 각 함수 호출이 모두 일치해야 하지만 호출 순서는 달라도 됩니다(집합 매칭 사용).

<strong>AST 매칭 예시:</strong>

```python
# 예시1: 매개변수 순서가 다른 경우 (매칭 성공)
예측: get_weather(city="Beijing", unit="celsius")
정답: get_weather(unit="celsius", city="Beijing")
결과: ✅ 매칭 성공

# 예시2: 동등한 표현식 (매칭 성공)
예측: calculate(x=2+3)
정답: calculate(x=5)
결과: ✅ 매칭 성공

# 예시3: 함수 이름 오류 (매칭 실패)
예측: get_temperature(city="Beijing")
정답: get_weather(city="Beijing")
결과: ❌ 매칭 실패

# 예시4: 매개변수 값 오류 (매칭 실패)
예측: get_weather(city="Shanghai")
정답: get_weather(city="Beijing")
결과: ❌ 매칭 실패
```

<strong>（3）BFCL 평가 지표</strong>

BFCL은 다음 지표로 에이전트 성능을 평가합니다:

<strong>1. 정확률 (Accuracy)</strong>

정확률은 가장 핵심적인 지표로, AST 매칭에 성공한 샘플의 비율로 정의됩니다:

$$
\text{Accuracy} = \frac{1}{N} \sum_{i=1}^{N} \text{AST\_Match}(P_i, G_i)
$$

여기서:
- $N$은 총 샘플 수
- $P_i$는 $i$번째 샘플의 예측 결과
- $G_i$는 $i$번째 샘플의 정답
- $\text{AST\_Match}(P_i, G_i) \in \{0, 1\}$은 AST 매칭 함수

<strong>2. AST 매칭률 (AST Match Rate)</strong>

정확률과 동일하며, AST 매칭 알고리즘 사용을 강조합니다:

$$
\text{AST Match Rate} = \text{Accuracy}
$$

<strong>3. 카테고리별 정확률 (Category-wise Accuracy)</strong>

각 카테고리 $c \in \{\text{simple}, \text{multiple}, \text{parallel}, \ldots\}$에 대해 해당 카테고리의 정확률을 계산합니다:

$$
\text{Accuracy}_c = \frac{1}{|D_c|} \sum_{i \in D_c} \text{AST\_Match}(P_i, G_i)
$$

여기서 $D_c$는 카테고리 $c$의 샘플 집합이며, $|D_c|$는 해당 카테고리의 샘플 수입니다.

<strong>4. 가중 정확률 (Weighted Accuracy)</strong>

다양한 카테고리의 난이도 가중치를 고려합니다:

$$
\text{Weighted Accuracy} = \sum_{c} w_c \cdot \text{Accuracy}_c
$$

여기서 $w_c$는 카테고리 $c$의 가중치이며, $\sum_c w_c = 1$을 만족합니다.

<strong>5. 오류율 (Error Rate)</strong>

함수를 올바르게 호출하지 못한 샘플의 비율:

$$
\text{Error Rate} = 1 - \text{Accuracy} = \frac{1}{N} \sum_{i=1}^{N} (1 - \text{AST\_Match}(P_i, G_i))
$$

<strong>지표 해석:</strong>

- <strong>Accuracy = 1.0</strong>: 모든 샘플이 완전히 정확
- <strong>Accuracy = 0.8</strong>: 80%의 샘플이 정확, 20%의 샘플이 오류
- <strong>Accuracy = 0.0</strong>: 모든 샘플이 오류

<strong>카테고리별 정확률 예시:</strong>

```python
# 평가 결과 가정
simple_accuracy = 0.95      # Simple 카테고리: 95% 정확
multiple_accuracy = 0.82    # Multiple 카테고리: 82% 정확
parallel_accuracy = 0.68    # Parallel 카테고리: 68% 정확

# 가중 정확률 (가중치 동일 가정)
weighted_accuracy = (0.95 + 0.82 + 0.68) / 3 = 0.817
```

```python
from hello_agents import SimpleAgent, HelloAgentsLLM
from hello_agents.evaluation import BFCLDataset, BFCLEvaluator

# 1. 에이전트 생성
llm = HelloAgentsLLM()
agent = SimpleAgent(name="TestAgent", llm=llm)

# 2. 데이터셋 로드
dataset = BFCLDataset(
    bfcl_data_dir="./temp_gorilla/berkeley-function-call-leaderboard/bfcl_eval/data",
    category="simple_python"
)
data = dataset.load()

# 3. 평가기 생성
evaluator = BFCLEvaluator(
    dataset=dataset,
    category="simple_python",
    evaluation_mode="ast"  # AST 매칭 모드 사용
)

# 4. 평가 실행
results = evaluator.evaluate(agent, max_samples=10)

# 5. 결과 확인

print(f"정확률: {results['overall_accuracy']:.2%}")
print(f"정답 수: {results['correct_samples']}/{results['total_samples']}")

# 6. BFCL 형식으로 결과 내보내기 (선택사항)
evaluator.export_to_bfcl_format(
    results,
    output_path="./evaluation_results/my_results.json"
)
```

### 12.2.4 BFCL 공식 평가 도구 통합

앞서 HelloAgents 내장 평가 기능을 사용하는 방법을 배웠습니다. 실제로 `BFCLEvaluationTool`은 이미 <strong>BFCL 공식 평가 도구를 자동으로 통합</strong>하여, 권위 있고 비교 가능한 평가 결과를 얻을 수 있습니다.

### 12.2.5 핵심 컴포넌트 구현 세부사항

앞선 절에서 BFCL 평가 도구 사용 방법을 배웠습니다. 이제 HelloAgents 평가 시스템의 핵심 컴포넌트가 어떻게 구현되었는지 깊이 이해해 보겠습니다. 이러한 구현 세부사항을 이해하면 평가 시스템을 더 잘 사용할 수 있을 뿐만 아니라, 자신의 요구에 맞게 커스터마이즈하고 확장할 수 있습니다.

<strong>（1）BFCLDataset: 데이터셋 로더</strong>

BFCLDataset은 BFCL 데이터셋의 로드 및 관리를 담당합니다:

````python
class BFCLDataset:
    """BFCL 데이터셋 로더"""

    def __init__(self, category: str = "simple", local_data_path: Optional[str] = None):
        self.category = category
        self.local_data_path = local_data_path
        self.data = []

    def load(self) -> List[Dict[str, Any]]:
        """데이터셋 로드"""
        # 우선 로컬에서 로드
        if self.local_data_path:
            return self._load_from_local()
        # 그렇지 않으면 Hugging Face에서 로드
        return self._load_from_huggingface()
````

BFCL 데이터셋은 공식 저장소에 있으므로, 권장하는 방법은 로컬에 직접 클론하여 평가하는 것입니다. 로컬에서 찾을 수 없을 때만 Hugging Face에서 로드합니다.

<strong>（2）BFCLEvaluator: 평가 실행기</strong>

BFCLEvaluator는 평가 프로세스 실행을 담당합니다. 핵심은 `evaluate()` 메서드로, 전체 평가 과정을 조율합니다:

````python
class BFCLEvaluator:
    """BFCL 평가기"""

    def evaluate(self, agent: Any, max_samples: Optional[int] = None) -> Dict[str, Any]:
        """평가 실행"""
        results = []

        for item in self.dataset[:max_samples]:
            # 1. 프롬프트 구성
            prompt = self._build_prompt(item)

            # 2. 에이전트 호출
            response = agent.run(prompt)

            # 3. 함수 호출 추출
            predicted_calls = self._extract_function_calls(response)

            # 4. 정답과 비교
            is_correct = self._compare_calls(predicted_calls, item["ground_truth"])

            results.append({
                "id": item["id"],
                "prediction": predicted_calls,
                "ground_truth": item["ground_truth"],
                "is_correct": is_correct
            })

        return {"results": results, "total_samples": len(results)}
````

이 평가기의 설계에는 세 가지 핵심 포인트가 있습니다: 첫째, 프롬프트 구성 - 데이터셋의 질문과 함수 정의를 에이전트가 이해할 수 있는 프롬프트로 변환해야 합니다. 둘째, 함수 호출 추출 - 에이전트의 응답에서 함수 호출을 추출하며, 여러 형식(JSON, 코드 블록 등)을 지원해야 합니다. 셋째, AST 매칭 - 단순한 문자열 매칭보다 더 정확한 추상 구문 트리를 사용하여 함수 호출을 비교합니다.

함수 호출 추출 구현을 살펴보겠습니다:

```python
def _extract_function_calls(self, response: str) -> List[Dict[str, Any]]:
    """응답에서 함수 호출 추출

    여러 형식 지원:
    1. JSON 형식: {"name": "func", "arguments": {...}}
    2. 코드 블록 형식: ```python\nfunc(arg1=val1)\n```
    3. 일반 텍스트 형식: func(arg1=val1)
    """
    calls = []

    # JSON 파싱 시도
    try:
        json_match = re.search(r'\{.*\}', response, re.DOTALL)
        if json_match:
            data = json.loads(json_match.group())
            if isinstance(data, dict) and "name" in data:
                calls.append(data)
            elif isinstance(data, list):
                calls.extend(data)
    except json.JSONDecodeError:

        pass

    # 코드 블록 추출 시도
    code_blocks = re.findall(r'```(?:python)?\n(.*?)\n```', response, re.DOTALL)
    for code in code_blocks:
        # Python 함수 호출 파싱
        parsed_calls = self._parse_python_calls(code)
        calls.extend(parsed_calls)

    return calls
```

<strong>（3）BFCLMetrics: 지표 계산기</strong>

BFCLMetrics는 다양한 평가 지표 계산을 담당합니다:

````python
class BFCLMetrics:
    """BFCL 지표 계산기"""

    def compute_metrics(self, results: List[Dict[str, Any]]) -> Dict[str, Any]:
        """모든 지표 계산"""
        return {
            "accuracy": self._compute_accuracy(results),
            "ast_match_rate": self._compute_ast_match_rate(results),
            "parameter_accuracy": self._compute_parameter_accuracy(results),
            "f1_score": self._compute_f1_score(results),
            "category_statistics": self._compute_category_stats(results)
        }
````

<strong>AST 매칭 구현</strong>:

AST 매칭은 BFCL 평가의 핵심 기술입니다. 단순한 문자열 매칭보다 더 스마트하여 의미적으로 동등한 함수 호출을 식별할 수 있습니다:

```python
def _ast_match(self, pred_call: Dict, true_call: Dict) -> bool:
    """AST를 사용하여 함수 호출 매칭

    AST 매칭의 장점:
    1. 매개변수 순서 무시: func(a=1, b=2)는 func(b=2, a=1)과 동등
    2. 동등한 표현식 인식: 2+3은 5와 동등
    3. 공백 및 형식 차이 무시
    """
    # 1. 함수 이름이 완전히 일치해야 함
    if pred_call.get("name") != true_call.get("name"):
        return False

    # 2. 매개변수를 AST 노드로 변환
    pred_args = self._args_to_ast(pred_call.get("arguments", {}))
    true_args = self._args_to_ast(true_call.get("arguments", {}))

    # 3. AST 노드 비교
    return ast.dump(pred_args) == ast.dump(true_args)

def _args_to_ast(self, args: Dict[str, Any]) -> ast.AST:
    """매개변수 딕셔너리를 AST 노드로 변환"""
    # 가상의 함수 호출 구성
    code = f"func({', '.join(f'{k}={repr(v)}' for k, v in args.items())})"
    tree = ast.parse(code)
    return tree.body[0].value  # Call 노드 반환
```

<strong>（4）도구화 캡슐화: BFCLEvaluationTool</strong>

마지막으로, 이러한 컴포넌트들을 Tool로 캡슐화하여 에이전트가 직접 호출할 수 있게 합니다:

````python
class BFCLEvaluationTool(Tool):
    """BFCL 평가 도구"""

    def __init__(self, local_data_path: Optional[str] = None):
        super().__init__(
            name="bfcl_evaluation",
            description="에이전트의 도구 호출 능력 평가"
        )
        self.dataset = None
        self.evaluator = None
        self.metrics_calculator = BFCLMetrics()

    def run(self, parameters: Dict[str, Any]) -> str:
        """평가 실행"""
        # 1. 데이터셋 로드
        self.dataset = BFCLDataset(...)

        # 2. 평가기 생성
        self.evaluator = BFCLEvaluator(...)

        # 3. 평가 실행
        results = self.evaluator.evaluate(...)

        # 4. 지표 계산
        metrics = self.metrics_calculator.compute_metrics(...)

        # 5. JSON 결과 반환
        return json.dumps(results, ensure_ascii=False)
````

이 도구의 설계는 세 가지 핵심 원칙을 따릅니다: 먼저 Tool 기본 클래스를 상속하여 HelloAgents의 도구 사양을 준수하고 프레임워크와의 원활한 통합을 보장합니다. 다음으로 엄격한 매개변수 검증을 수행하여 필수 매개변수를 확인하고 친절한 오류 메시지를 제공하여 사용자 경험을 향상시킵니다. 마지막으로 결과를 형식화하여 파싱과 표시가 용이한 JSON 문자열을 반환합니다. 이러한 모듈식 설계를 통해 사용하기 쉬우면서도 유연한 평가 시스템을 구현했습니다.

### 12.2.6 확장 및 최적화 제안

앞서 HelloAgents를 사용한 BFCL 평가 방법을 익혔습니다. 현재 구현은 SimpleAgent를 기반으로 한 간단한 복현이며, BFCL 평가의 기본 기능을 완성한 것입니다. 실제 적용에서 BFCL 기준은 여러 난이도 수준과 시나리오를 포함하며, 리더보드에서 더 높은 점수를 얻으려면 추가적인 최적화와 확장이 필요합니다.

<strong>（1）현재 구현의 한계</strong>

현재 SimpleAgent 구현은 주로 평가 프로세스 구축에 집중되어 있으며, 도구 호출 능력에는 개선 여지가 있습니다. SimpleAgent는 커스텀 도구 호출 형식 `[TOOL_CALL:tool_name:parameters]`를 사용하는데, 이 형식은 LLM이 능동적으로 학습하고 사용해야 하므로, 복잡한 시나리오에서 네이티브 함수 호출(Function Calling)을 사용하는 에이전트보다 성능이 떨어질 수 있습니다. 또한 현재는 simple_python 등 기본 카테고리만 테스트했으며, multiple, parallel, irrelevance 등 더 복잡한 시나리오에는 맞춤형 최적화가 필요합니다.

<strong>（2）BFCL 점수 향상 방향</strong>

BFCL 평가 점수를 더욱 향상시키려면 다음 방향에서 접근할 수 있습니다. 먼저 에이전트의 도구 호출 능력을 최적화합니다. GPT-4, Claude 등 네이티브 함수 호출을 지원하는 LLM을 사용하거나, 프롬프트를 개선하여 LLM이 도구 호출 형식을 더 잘 이해하도록 합니다. 다음으로 도구 라이브러리를 확장합니다. BFCL 테스트에는 다양한 유형의 함수가 관련되므로, 테스트 데이터셋의 특성에 따라 일반적인 도구 유형을 미리 구현하여 에이전트의 도구 커버리지를 향상시킵니다. 셋째, 난이도 수준에 따라 다른 전략을 설계합니다. 예를 들어 multiple 시나리오에서는 에이전트가 다단계 도구 호출 시퀀스를 계획할 수 있어야 하고, parallel 시나리오에서는 병렬 실행 가능한 도구 호출을 식별해야 하며, irrelevance 시나리오에서는 도구 호출이 실제로 필요한지 판단해야 합니다.

<strong>（3）실천 제안</strong>

BFCL에서 더 좋은 결과를 얻으려는 개발자에게 다음 실천 전략을 권장합니다. 먼저 simple 카테고리부터 시작하여 기본적인 단일 함수 호출이 안정적으로 작동하는지 확인합니다. 이것이 이후 최적화의 기초입니다. 그 다음, multiple, parallel 등 더 복잡한 카테고리를 단계적으로 테스트하고, 실패 사례를 분석하여 에이전트의 취약 부분을 찾습니다. 최적화 과정에서 BFCL 리더보드의 고득점 모델을 참고하여 설계 아이디어와 최적화 기법을 배울 수 있습니다. 또한 공식 평가 도구로 검증하여 최적화 결과가 리더보드 기준과 일치하는지 확인하는 것을 권장합니다.

평가 시 추가 처리할 수 있는 제안을 요약합니다:

<strong>1. 점진적 평가</strong>

소규모 샘플부터 시작하여 샘플 수를 점진적으로 증가시킵니다:

```python
# 1단계: 빠른 테스트 (5개 샘플)
results_quick = bfcl_tool.run(agent, category="simple_python", max_samples=5)

# 2단계: 중간 규모 테스트 (50개 샘플)
if results_quick['overall_accuracy'] > 0.8:
    results_medium = bfcl_tool.run(agent, category="simple_python", max_samples=50)

# 3단계: 전체 평가 (전체 샘플)
if results_medium['overall_accuracy'] > 0.8:
    results_full = bfcl_tool.run(agent, category="simple_python", max_samples=0)
```

<strong>2. 다중 카테고리 평가</strong>

다양한 난이도의 작업을 평가합니다:

```python
categories = ["simple_python", "multiple", "parallel", "irrelevance"]

for category in categories:
    print(f"\n평가 카테고리: {category}")
    results = bfcl_tool.run(agent, category=category, max_samples=10)
    print(f"정확률: {results['overall_accuracy']:.2%}")
```

<strong>3. 비교 평가</strong>

다양한 설정의 에이전트를 비교합니다:

```python
# 설정1: 기본 프롬프트
agent1 = SimpleAgent(name="Agent-Default", llm=llm)
results1 = bfcl_tool.run(agent1, category="simple_python", max_samples=10)

# 설정2: 최적화된 프롬프트
agent2 = SimpleAgent(name="Agent-Optimized", llm=llm)
# ... 최적화된 시스템 프롬프트 설정 ...
results2 = bfcl_tool.run(agent2, category="simple_python", max_samples=10)

# 결과 비교
print(f"기본 설정 정확률: {results1['overall_accuracy']:.2%}")

### 12.3.1 GAIA 기준 소개

GAIA (General AI Assistants)는 Meta AI와 Hugging Face가 공동으로 출시한 평가 기준으로, AI 어시스턴트의 <strong>범용 능력</strong>을 평가하는 데 중점을 둡니다<sup>[2]</sup>. 도구 호출에 특화된 BFCL과 달리, GAIA는 실제 세계 작업에서 에이전트의 종합적인 성능을 평가합니다.

GAIA의 설계 철학은 <strong>실제 세계의 문제는 종종 여러 능력의 종합적인 활용이 필요하다</strong>는 것입니다. 우수한 AI 어시스턴트는 도구를 호출할 뿐만 아니라 다음도 갖춰야 합니다:

- <strong>다단계 추론</strong>: 복잡한 문제를 여러 하위 문제로 분해
- <strong>지식 활용</strong>: 내장 지식과 외부 지식 베이스 활용
- <strong>멀티모달 이해</strong>: 텍스트, 이미지, 파일 등 다양한 입력 처리
- <strong>웹 브라우징</strong>: 인터넷에서 최신 정보 획득
- <strong>파일 작업</strong>: 다양한 형식의 파일 읽기 및 처리

<strong>（2）준정확 매칭 소개</strong>

GAIA는 <strong>준정확 매칭(Quasi Exact Match)</strong> 평가 알고리즘을 사용합니다. 이는 GAIA 공식적으로 정의한 평가 기준입니다. 이 알고리즘의 핵심 아이디어는 <strong>먼저 답안을 정규화한 다음 정확 매칭을 수행하는 것</strong>입니다.

예측 답안 $A_{\text{pred}}$와 정답 $A_{\text{true}}$가 주어지면, 준정확 매칭 함수는 다음과 같이 정의됩니다:

$$
\text{Quasi\_Exact\_Match}(A_{\text{pred}}, A_{\text{true}}) = \begin{cases}
1 & \text{if } \mathcal{N}(A_{\text{pred}}) = \mathcal{N}(A_{\text{true}}) \
0 & \text{otherwise}
\end{cases}
$$

여기서 $\mathcal{N}(\cdot)$는 정규화 함수로, 답안 유형에 따라 다른 규칙을 적용합니다.

정규화 함수는 답안 유형에 따라 다른 규칙을 적용합니다. 숫자 유형의 경우 쉼표 구분자 제거(`1,000` → `1000`) 및 단위 기호 제거(`$100` → `100`, `50%` → `50`)가 필요합니다. 예를 들어 `"$1,234.56"`은 `"1234.56"`으로 정규화됩니다. 문자열 유형의 경우 소문자 변환(`"Apple"` → `"apple"`), 관사 제거(`"the apple"` → `"apple"`), 불필요한 공백 제거(`"hello  world"` → `"hello world"`), 끝 구두점 제거(`"hello."` → `"hello"`)가 필요합니다. 예를 들어 `"The United States"`는 `"united states"`로 정규화됩니다. 목록 유형의 경우 쉼표로 요소를 분리하고, 각 요소에 문자열 정규화를 적용하고, 알파벳 순서로 정렬한 후 재결합이 필요합니다. 예를 들어 `"Paris, London, Berlin"`은 `"berlin,london,paris"`로 정규화됩니다.

<strong>정규화 예시:</strong>

```python
# 숫자 답안
원본 답안: "$1,234.56"
정규화 후: "1234.56"

# 문자열 답안
원본 답안: "The United States of America"
정규화 후: "united states of america"

# 목록 답안
원본 답안: "Paris, London, Berlin"
정규화 후: "berlin, london, paris"
```

<strong>（3）GAIA 평가 지표</strong>

GAIA는 다음 지표로 에이전트 성능을 평가합니다:

<strong>1. 정확 매칭률 (Exact Match Rate)</strong>

정확 매칭률은 GAIA의 핵심 지표로, 준정확 매칭에 성공한 샘플의 비율로 정의됩니다:

$$
\text{Exact Match Rate} = \frac{1}{N} \sum_{i=1}^{N} \text{Quasi\_Exact\_Match}(A_{\text{pred},i}, A_{\text{true},i})
$$

여기서:
- $N$은 총 샘플 수
- $A_{\text{pred},i}$는 $i$번째 샘플의 예측 답안
- $A_{\text{true},i}$는 $i$번째 샘플의 정답
- $\text{Quasi\_Exact\_Match}(\cdot, \cdot) \in \{0, 1\}$은 준정확 매칭 함수

<strong>2. 수준별 정확률 (Level-wise Accuracy)</strong>

각 난이도 수준 $\ell \in \{1, 2, 3\}$에 대해 해당 수준의 정확률을 계산합니다:

$$
\text{Accuracy}_\ell = \frac{1}{|D_\ell|} \sum_{i \in D_\ell} \text{Quasi\_Exact\_Match}(A_{\text{pred},i}, A_{\text{true},i})
$$

여기서 $D_\ell$는 난이도 수준 $\ell$의 샘플 집합이며, $|D_\ell|$는 해당 수준의 샘플 수입니다.

<strong>3. 난이도 점진적 하락률 (Difficulty Progression Drop Rate)</strong>

난이도가 증가할 때 에이전트의 성능 감소를 측정합니다:

$$
\text{Drop Rate}_{\ell \to \ell+1} = \frac{\text{Accuracy}_\ell - \text{Accuracy}_{\ell+1}}{\text{Accuracy}_\ell}
$$

- $\text{Drop Rate}_{1 \to 2}$: Level 1에서 Level 2로의 하락률
- $\text{Drop Rate}_{2 \to 3}$: Level 2에서 Level 3으로의 하락률

<strong>4. 평균 추론 단계 수 (Average Reasoning Steps)</strong>

에이전트가 작업을 완료하는 데 필요한 평균 단계 수를 평가합니다:

$$
\text{Avg Steps} = \frac{1}{N_{\text{correct}}} \sum_{i \in \text{Correct}} \text{steps}_i
$$

여기서 $N_{\text{correct}}$는 올바르게 답한 샘플 수이며, $\text{steps}_i$는 $i$번째 샘플의 추론 단계 수입니다.

<strong>지표 해석:</strong>

- <strong>Exact Match Rate = 1.0</strong>: 모든 샘플이 완전히 정확
- <strong>Exact Match Rate = 0.5</strong>: 50%의 샘플이 정확, 50%의 샘플이 오류
- <strong>Drop Rate = 0.3</strong>: 난이도 증가로 인해 정확률이 30% 하락
- <strong>Drop Rate = 0.0</strong>: 난이도 증가가 정확률에 영향을 주지 않음 (이상적인 경우)

<strong>평가 예시:</strong>

10개 샘플을 평가했다고 가정하면, 결과는 표 12.4를 참조합니다:

<div align="center">
  <p>표 12.4 GAIA 데이터셋 난이도 수준 분포</p>
  <img src="https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/12-figures/12-table-4.png" alt="" width="85%"/>
</div>

이 사례의 지표를 계산하려면 아래 Python 스크립트를 참조하세요.

```python
# 1. 정확 매칭률
total_samples = 10
correct_samples = 7  # 샘플1,2,3,5,6,8,9
exact_match_rate = correct_samples / total_samples = 0.70  # 70%

# 2. 수준별 정확률
level_1_correct = 3  # 샘플1,2,3
level_1_total = 3
level_1_accuracy = 3 / 3 = 1.00  # 100%

level_2_correct = 2  # 샘플5,6
level_2_total = 3
level_2_accuracy = 2 / 3 = 0.67  # 67%

level_3_correct = 2  # 샘플8,9
level_3_total = 4
level_3_accuracy = 2 / 4 = 0.50  # 50%

# 3. 난이도 점진적 하락률
drop_rate_1_to_2 = (1.00 - 0.67) / 1.00 = 0.33  # 33%
drop_rate_2_to_3 = (0.67 - 0.50) / 0.67 = 0.25  # 25%

print(f"정확 매칭률: {exact_match_rate:.2%}")  # 70.00%
print(f"Level 1 정확률: {level_1_accuracy:.2%}")  # 100.00%
print(f"Level 2 정확률: {level_2_accuracy:.2%}")  # 66.67%
print(f"Level 3 정확률: {level_3_accuracy:.2%}")  # 50.00%
print(f"Level 1→2 하락률: {drop_rate_1_to_2:.2%}")  # 33.00%
print(f"Level 2→3 하락률: {drop_rate_2_to_3:.2%}")  # 25.00%
```

<strong>결과 분석:</strong>

- <strong>전체 성능</strong>: 70%의 정확 매칭률, 양호한 성능
- <strong>난이도 민감성</strong>: Level 1에서 Level 2로 33% 하락, 에이전트가 중간 난이도 작업에서 뚜렷한 성능 감소를 보임
- <strong>능력 한계</strong>: Level 3 정확률이 50%로, 에이전트가 복잡한 작업에서 개선의 여지가 있음

하락률이 클수록 에이전트가 복잡한 작업 처리 시 능력 감소가 더 뚜렷함을 의미합니다.

<strong>（4）GAIA 공식 시스템 프롬프트</strong>

GAIA는 모델 출력이 평가 형식에 맞도록 특정 시스템 프롬프트를 사용하도록 요구합니다:

```python
GAIA_SYSTEM_PROMPT = """You are a general AI assistant. I will ask you a question. Report your thoughts, and finish your answer with the following template: FINAL ANSWER: [YOUR FINAL ANSWER].

YOUR FINAL ANSWER should be a number OR as few words as possible OR a comma separated list of numbers and/or strings.

If you are asked for a number, don't use comma to write your number neither use units such as $ or percent sign unless specified otherwise.

If you are asked for a string, don't use articles, neither abbreviations (e.g. for cities), and write the digits in plain text unless specified otherwise.

If you are asked for a comma separated list, apply the above rules depending of whether the element to be put in the list is a number or a string."""
```

GAIA는 답안 형식에 엄격한 요구사항이 있습니다: 답안은 반드시 `FINAL ANSWER: [답안]` 형식으로 제공해야 하며, 숫자 유형의 답안은 쉼표 구분자와 단위 기호를 사용하지 않아야 하고, 문자열 유형의 답안은 관사와 약어를 사용하지 않아야 하며, 목록 유형의 답안은 쉼표로 구분하고 알파벳 순으로 정렬해야 합니다.

### 12.3.3 HelloAgents에서 GAIA 평가 구현

BFCL과 유사하게, 두 가지 평가 방법을 제공하며 <strong>방법 1</strong>을 권장합니다.

<strong>방법 1: GAIAEvaluationTool을 사용한 원클릭 평가</strong>

가장 간단한 방법으로, 데이터셋 다운로드, 평가 실행, 결과 내보내기, 보고서 생성을 자동으로 완료합니다:

```python
from hello_agents import SimpleAgent, HelloAgentsLLM
from hello_agents.tools import GAIAEvaluationTool

# GAIA 공식 시스템 프롬프트 (논문에서 가져옴)
GAIA_SYSTEM_PROMPT = """You are a general AI assistant. I will ask you a question. Report your thoughts, and finish your answer with the following template: FINAL ANSWER: [YOUR FINAL ANSWER].

YOUR FINAL ANSWER should be a number OR as few words as possible OR a comma separated list of numbers and/or strings.

If you are asked for a number, don't use comma to write your number neither use units such as $ or percent sign unless specified otherwise.

If you are asked for a string, don't use articles, neither abbreviations (e.g. for cities), and write the digits in plain text unless specified otherwise.

If you are asked for a comma separated list, apply the above rules depending of whether the element to be put in the list is a number or a string."""

# 1. 에이전트 생성 (GAIA 공식 시스템 프롬프트 사용)
llm = HelloAgentsLLM()
agent = SimpleAgent(
    name="TestAgent",
    llm=llm,
    system_prompt=GAIA_SYSTEM_PROMPT  # 핵심: GAIA 공식 프롬프트 사용
)

# 2. GAIA 평가 도구 생성
gaia_tool = GAIAEvaluationTool()

### 12.3.5 핵심 컴포넌트 구현 세부사항

GAIA 평가 시스템의 구현은 BFCL과 유사하지만, 범용 능력 평가를 위한 특수한 설계가 있습니다.

<strong>（1）GAIADataset: 멀티모달을 지원하는 데이터 로더</strong>

GAIA 데이터셋의 특징은 멀티모달 데이터(텍스트, 파일, 이미지 등)를 포함한다는 것입니다:

````python
class GAIADataset:
    """GAIA 데이터셋 로더

    HuggingFace에서 GAIA 데이터셋 로드를 지원 (제한된 데이터셋)
    """

    def __init__(
        self,
        level: Optional[int] = None,
        split: str = "validation",
        local_data_dir: Optional[str] = None
    ):
        self.level = level
        self.split = split
        self.local_data_dir = local_data_dir or "./data/gaia"
        self.data = []

    def load(self) -> List[Dict[str, Any]]:
        """데이터셋 로드"""
        # HuggingFace에서 다운로드
        items = self._load_from_huggingface()

        # 수준별 필터링
        if self.level:
            items = [item for item in items if item.get("level") == self.level]

        self.data = items
        return items

    def _load_from_huggingface(self) -> List[Dict[str, Any]]:
        """HuggingFace에서 GAIA 데이터셋 다운로드"""
        from huggingface_hub import snapshot_download
        import json

        # 데이터셋 다운로드
        repo_id = "gaia-benchmark/GAIA"
        local_dir = snapshot_download(
            repo_id=repo_id,
            repo_type="dataset",
            local_dir=self.local_data_dir,
            local_dir_use_symlinks=False
        )

        # JSONL 파일 로드
        data_file = Path(local_dir) / "2023" / self.split / "metadata.jsonl"
        items = []
        with open(data_file, 'r', encoding='utf-8') as f:
            for line in f:
                item = json.loads(line)
                items.append(self._standardize_item(item))

        return items
````

<strong>（2）GAIAEvaluator: GAIA 공식 평가 알고리즘 구현</strong>

GAIA의 평가는 <strong>준정확 매칭(Quasi Exact Match)</strong> 알고리즘을 사용하며, 특수한 답안 정규화 및 매칭 로직이 필요합니다:

````python
class GAIAEvaluator:
    """GAIA 평가기

    GAIA 공식 준정확 매칭(Quasi Exact Match) 평가 알고리즘 구현
    """

    def evaluate(self, agent: Any, max_samples: Optional[int] = None) -> Dict[str, Any]:
        """평가 실행"""
        dataset_items = self.dataset.load()

        if max_samples:
            dataset_items = dataset_items[:max_samples]

        results = []
        for i, item in enumerate(dataset_items, 1):
            # 1. 프롬프트 구성
            prompt = self._build_prompt(item["question"], item)

            # 2. 에이전트 호출

            response = agent.run(prompt)

            # 3. 답안 추출 (GAIA 형식: FINAL ANSWER: [답안])
            predicted_answer = self._extract_answer(response)

            # 4. 답안 정규화 (GAIA 공식 규칙)
            normalized_pred = self._normalize_answer(predicted_answer)
            normalized_truth = self._normalize_answer(item["final_answer"])

            # 5. 준정확 매칭
            exact_match = (normalized_pred == normalized_truth)

            results.append({
                "task_id": item["task_id"],
                "predicted": predicted_answer,
                "expected": item["final_answer"],
                "exact_match": exact_match,
                "level": item.get("level", 0)
            })

        return self._format_results(results)
````

GAIA는 다양한 유형의 답안을 처리하기 위한 특수 정규화 규칙을 사용합니다:

```python
def _normalize_answer(self, answer: str) -> str:
    """답안 문자열 표준화 (GAIA 공식 표준화 규칙)

    규칙:
    1. 숫자: 쉼표 구분자 및 단위 기호 제거
    2. 문자열: 관사 제거, 소문자 변환, 불필요한 공백 제거
    3. 목록: 쉼표로 구분, 알파벳 순서로 정렬
    """
    if not answer:
        return ""

    answer = answer.strip()

    # 쉼표로 구분된 목록인지 확인
    if ',' in answer:
        parts = [self._normalize_single_answer(p.strip()) for p in answer.split(',')]
        parts.sort()  # GAIA는 알파벳 순서로 정렬 요구
        return ','.join(parts)
    else:
        return self._normalize_single_answer(answer)

def _normalize_single_answer(self, answer: str) -> str:
    """단일 답안 표준화 (쉼표가 없는 답안)"""
    answer = answer.strip().lower()

    # 일반적인 관사 제거
    articles = ['the', 'a', 'an']
    words = answer.split()
    if words and words[0] in articles:
        words = words[1:]
        answer = ' '.join(words)

    # 통화 기호 및 퍼센트 기호 제거
    answer = answer.replace('$', '').replace('%', '').replace('€', '').replace('£', '')

    # 숫자의 쉼표 구분자 제거
    answer = re.sub(r'(\d),(\d)', r'\1\2', answer)

    # 불필요한 공백 제거
    answer = ' '.join(answer.split())

    # 끝의 구두점 제거
    answer = answer.rstrip('.,;:!?')

    return answer
```

GAIA는 모델 출력 형식이 `FINAL ANSWER: [답안]`이어야 합니다:

```python
def _extract_answer(self, response: str) -> str:
    """응답에서 답안 추출 (GAIA 형식)

    GAIA는 답안 형식이 FINAL ANSWER: [답안]이어야 합니다
    """
    # 먼저 GAIA 공식 형식의 답안 추출 시도
    final_answer_pattern = r'FINAL ANSWER:\s*(.+?)(?:\n|$)'
    match = re.search(final_answer_pattern, response, re.IGNORECASE | re.MULTILINE)
    if match:
        answer = match.group(1).strip()
        # 가능한 대괄호 제거
        answer = answer.strip('[]')
        return answer

    # 대안: 다른 답안 마커 찾기
    answer_patterns = [
        r'답안[：:]\s*(.+)',
        r'최종답안[：:]\s*(.+)',
        r'Final answer[：:]\s*(.+)',
        r'Answer[：:]\s*(.+)',
    ]

    for pattern in answer_patterns:
        match = re.search(pattern, response, re.IGNORECASE)
        if match:
            return match.group(1).strip()

    # 마커를 찾지 못한 경우, 마지막 비어있지 않은 줄 반환
    lines = response.strip().split('\n')
    for line in reversed(lines):
        line = line.strip()
        if line and not line.startswith('#'):
            return line

    return response.strip()
```

평가가 완료되면 GAIA 공식에서 요구하는 JSONL 형식으로 내보낼 수 있습니다:

```python
def export_to_gaia_format(
    self,
    results: Dict[str, Any],
    output_path: Union[str, Path],
    include_reasoning: bool = True
) -> None:
    """GAIA 공식 형식(JSONL)으로 내보내기

    GAIA 요구 형식:
    {"task_id": "xxx", "model_answer": "답안", "reasoning_trace": "추론 과정"}
    """
    output_path = Path(output_path)
    output_path.parent.mkdir(parents=True, exist_ok=True)

    with open(output_path, 'w', encoding='utf-8') as f:
        for result in results.get("detailed_results", []):
            entry = {
                "task_id": result["task_id"],
                "model_answer": result["predicted"]
            }

            if include_reasoning:
                entry["reasoning_trace"] = result.get("response", result["predicted"])

            f.write(json.dumps(entry, ensure_ascii=False) + '\n')
```

<strong>（3）GAIAEvaluationTool: 원클릭 평가 도구</strong>

GAIAEvaluationTool은 완전한 평가 프로세스를 캡슐화하여 원클릭 평가 기능을 제공합니다:

````python
class GAIAEvaluationTool(Tool):
    """GAIA 평가 도구

    원클릭 평가 기능 제공:
    1. HelloAgents 평가 실행
    2. GAIA 형식 결과 내보내기
    3. 평가 보고서 생성
    4. 제출 안내서 생성
    """

    def run(
        self,
        agent: Any,
        level: Optional[int] = None,
        max_samples: Optional[int] = None,
        local_data_dir: Optional[str] = None,
        export_results: bool = True,
        generate_report: bool = True
    ) -> Dict[str, Any]:
        """GAIA 원클릭 평가 실행"""
        # 단계1: HelloAgents 평가 실행
        results = self._run_evaluation(agent, level, max_samples, local_data_dir)

        # 단계2: GAIA 형식 결과 내보내기
        if export_results:
            self._export_results(results)

        # 단계3: 평가 보고서 생성
        if generate_report:
            self.generate_report(results)

        return results
````

영문으로 문제를 생성하기로 선택한 데는 네 가지 중요한 이유가 있습니다: 첫째, AIME 실제 문제와 일치(AIME는 영어 경시대회이므로 영문 문제 생성이 더 합리적), 둘째, 평가의 공정성 보장(LLM Judge 평가 시 영어 vs 영어가 더 공평), 셋째, 국제화 편의(영문 문제는 더 광범위하게 사용 가능), 넷째, 번역 문제 회피(중영문 번역의 정확성을 걱정할 필요 없음)입니다.

```python
def generate_and_save(self, num_problems: int = 30, output_dir: str = "data_generation/generated_data"):
    # Clean old checkpoints
    for file in os.listdir(output_dir):
        if file.startswith("checkpoint_") and file.endswith(".json"):
            os.remove(os.path.join(output_dir, file))

    # Generate with tqdm progress bar
    with tqdm(total=num_problems, desc="Generating AIME problems", unit="problem") as pbar:
        last_call_time = 0

        for i in range(num_problems):
            if last_call_time > 0:
                elapsed = time.time() - last_call_time
                if elapsed < self.delay_seconds:
                    time.sleep(self.delay_seconds - elapsed)

            start_time = time.time()
            problem = self.generate_single()
            last_call_time = time.time()
            generation_time = last_call_time - start_time

            pbar.set_postfix({
                "topic": problem.get("topic", "N/A"),
                "answer": problem.get("answer", "N/A"),
                "time": f"{generation_time:.1f}s"
            })
            pbar.update(1)

    return generated_data_path
```

LaTeX 수학 공식 지원: 생성된 AIME 문제에는 LaTeX 수학 공식이 포함되어 있어 JSON 파싱에 특수 처리가 필요합니다:

def _parse_response(self, response: str) -> dict:
    import re
    if "```json" in response:
        json_str = response.split("```json")[1].split("```")[0].strip()
    else:
        json_str = response.strip()
    try:
        problem_data = json.loads(json_str)
    except json.JSONDecodeError:
        # LaTeX 이스케이프 문제 수정
        fixed_json_str = re.sub(r'(?<!\\)\\(?!["\\/bfnrtu])', r'\\\\', json_str)
        problem_data = json.loads(fixed_json_str)
    return problem_data
```

```python
class AIDataset:
    def __init__(
        self,
        dataset_type: str = "generated",
        data_path: str = None,
        year: int = None
    ):
        self.dataset_type = dataset_type
        self.data_path = data_path
        self.year = year  # real 유형에만 사용, 기본값 2025

    def load(self):
        if self.dataset_type == "generated":
            return self._load_generated_data()
        elif self.dataset_type == "real":
            return self._load_real_data()

    def _load_real_data(self):
        from huggingface_hub import snapshot_download
        repo_id = "math-ai/aime25"
        local_dir = snapshot_download(repo_id=repo_id, repo_type="dataset")
        data_file = list(Path(local_dir).glob("*.jsonl"))[0]
        data = []
        with open(data_file, 'r', encoding='utf-8') as f:
            for line in f:
                if line.strip():
                    data.append(json.loads(line))
        problems = []
        for idx, item in enumerate(data):
            problem = {
                "problem_id": item.get("id", f"aime_2025_{idx}"),
                "problem": item.get("problem", ""),
                "answer": item.get("answer", ""),
                "solution": item.get("solution", ""),
            }
            problems.append(problem)
        return problems
```

AIME 2025 데이터셋만 사용하는 데는 네 가지 이유가 있습니다: 첫째, 데이터의 시의성(2025년이 가장 최신 AIME 경시대회 데이터), 둘째, 유지 관리 단순화(하나의 데이터셋만 유지하여 코드가 더 간결), 셋째, 형식 통일(JSONL 형식, 필드 이름이 소문자로 통일), 넷째, 충분한 대표성(30개 문제로 생성 품질 평가에 충분).

def run_complete_evaluation(
    num_problems: int = 30,
    delay_seconds: float = 3.0

    # 단계1: AIME 문제 생성
    generator = AIMEGenerator(delay_seconds=delay_seconds)

    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")

    os.makedirs(evaluation_dir, exist_ok=True)

    llm_judge_tool = LLMJudgeTool(llm=llm)
    llm_judge_result_json = llm_judge_tool.run({
        "generated_data_path": generated_data_path,
        "reference_year": 2025,
        "max_samples": num_problems,
        "output_dir": os.path.join(evaluation_dir, "llm_judge"),
        "judge_model": "gpt-4o"
    })

    # 단계2.2: Win Rate 평가
    win_rate_tool = WinRateTool(llm=llm)
    win_rate_result_json = win_rate_tool.run({
        "generated_data_path": generated_data_path,
        "reference_year": 2025,
        "num_comparisons": min(num_problems, 20),
        "output_dir": os.path.join(evaluation_dir, "win_rate"),
        "judge_model": "gpt-4o"
    })

    # 단계3: 종합 보고서 생성
    comprehensive_report_path = os.path.join(evaluation_dir, "comprehensive_report.md")
    report = generate_comprehensive_report(
        generated_data_path,
        json.loads(llm_judge_result_json),
        json.loads(win_rate_result_json)
    )
    with open(comprehensive_report_path, 'w', encoding='utf-8') as f:
        f.write(report)

데이터 생성 측면에서, 적절한 지연 시간(2-3초)을 사용하여 API 속도 제한을 피해야 하고, 중단으로 인한 손실을 방지하기 위해 체크포인트 저장을 활성화해야 하며, 먼저 소규모 테스트(10개)를 통해 문제가 없음을 확인한 후 대규모 생성을 진행하고, 생성 품질을 정기적으로 확인하여 프롬프트를 적시에 조정해야 합니다. 평가 전략에서는 LLM Judge와 Win Rate 두 가지 방법을 결합하는 것을 권장하며, LLM Judge는 절대적 품질 평가에, Win Rate는 상대적 품질 비교에, 수동 검증은 최종 품질 확인에 사용합니다. 품질 기준에서는 LLM Judge 평균 점수가 4.0/5.0 이상, Win Rate가 45% 이상(50%에 가까울수록 좋음), 합격률이 80% 이상, 수동 검증 통과율이 90% 이상이 되는 것을 권장합니다. 반복 최적화 과정에서는 평가 결과에 따라 생성 프롬프트를 조정하고, 낮은 점수 문제의 공통 문제를 분석하며, 높은 점수 문제의 장점을 참조하고, 생성 전략을 지속적으로 개선해야 합니다.

이 절의 학습을 통해 HelloAgents 프레임워크를 사용하여 데이터 생성 품질을 평가하는 방법을 익혔습니다. LLM Judge 평가, Win Rate 평가, 수동 검증의 세 가지 방법을 포함한 이 완전한 평가 체계는 생성 데이터의 고품질을 보장하고 AI 시스템의 훈련 및 테스트에 신뢰할 수 있는 데이터 지원을 제공할 수 있습니다.

LLM Judge와 Win Rate 평가에 대해 HelloAgents는 도구 통합도 진행했으며 완전한 예시 코드를 제공합니다. 이 두 가지 평가 방법의 구체적인 구현 세부사항에 관심이 있다면 예시 코드를 참조할 수 있습니다.




## 12.5 이 장의 요약

이 장에서 우리는 HelloAgents 프레임워크를 위한 완전한 성능 평가 시스템을 구축했습니다. 배운 핵심 내용을 돌아보겠습니다:

**（1）평가 체계 개요**

우리는 에이전트의 다양한 능력 차원을 포괄하는 3계층 평가 체계를 구축했습니다. 첫째, 도구 호출 능력 평가(BFCL)는 에이전트의 함수 호출 정확성을 평가하는 데 집중하며, simple, multiple, parallel, irrelevance의 네 가지 카테고리를 포함하고 AST 매칭 기술을 사용하여 정밀하게 평가합니다. 둘째, 범용 능력 평가(GAIA)는 에이전트의 종합적인 문제 해결 능력을 평가하며, 세 가지 난이도 수준의 466개 실제 세계 문제를 포함하고 다단계 추론, 도구 사용, 파일 처리 등의 능력에 집중합니다. 셋째, 데이터 생성 품질 평가(AIME)는 LLM이 생성한 데이터의 품질을 평가하며, LLM Judge와 Win Rate 두 가지 방법을 사용하고, 수동 검증과 종합 보고서 생성을 지원하여 생성 데이터가 참조 데이터의 품질 기준에 도달하도록 보장합니다.

**（2）핵심 기술 포인트**

기술 구현에서 우리는 여섯 가지 핵심 기술 포인트를 채택했습니다. 첫째, 모듈식 설계로 평가 시스템은 데이터 계층(Dataset이 데이터 로딩과 관리 담당), 평가 계층(Evaluator가 평가 프로세스 실행 담당), 지표 계층(Metrics가 다양한 평가 지표 계산 담당)의 3계층 아키텍처를 채택합니다. 둘째, 도구화 캡슐화로 모든 평가 기능을 Tool로 캡슐화하여 에이전트가 직접 호출하거나, 워크플로에 통합하거나, 통일된 인터페이스를 통해 사용할 수 있습니다. 셋째, AST 매칭 기술로 추상 구문 트리를 사용하여 함수 호출을 매칭하며, 단순 문자열 매칭보다 더 스마트하고 매개변수 순서를 무시하고, 동등한 표현식을 인식하며, 형식 차이를 무시할 수 있습니다. 넷째, 멀티모달 지원으로 GAIA 평가는 텍스트 질문, 첨부 파일, 이미지 입력 등 멀티모달 데이터를 지원합니다. 다섯째, LLM Judge 평가로 LLM을 심사위원으로 사용하여 생성 데이터 품질을 평가하고, 다차원 채점(정확성, 명확성, 난이도 일치, 완전성)과 자동화 평가 흐름, 자세한 평가 보고서를 제공하며, 커스텀 평가 차원 및 기준을 지원합니다. 여섯째, Win Rate 비교 평가로 쌍별 비교를 통해 생성 품질(생성 데이터 vs 참조 데이터)을 평가하고, LLM이 어느 것이 더 좋은지 판단하여 승률 통계를 계산하며, 50%에 가까울수록 품질이 비슷함을 의미합니다.

**（3）확장 방향**

이 장의 평가 시스템을 기반으로 네 가지 방향으로 확장할 수 있습니다. 첫째, 새로운 평가 기준 추가: BFCL과 GAIA의 구현 패턴을 참조하여 Dataset, Evaluator, Metrics 세 가지 컴포넌트를 구현하고 Tool로 캡슐화합니다. 둘째, 커스텀 평가 지표: Metrics 클래스에 새로운 지표 계산 방법을 추가하고 구체적인 응용 시나리오에 맞는 지표를 설계합니다. 셋째, CI/CD 흐름 통합: 코드 커밋 시 자동으로 평가를 실행하고, 성능 임계값을 설정하여 성능 저하를 방지하며, 평가 보고서를 생성하고 보관합니다. 넷째, 데이터 생성 평가 확장: 더 많은 데이터 유형(코드, 대화, 문서 등)을 지원하고, 더 많은 평가 차원(혁신성, 다양성 등)을 추가하며, 더 많은 참조 데이터셋을 통합하고, 다중 모델 비교 평가를 지원합니다.

2. BFCL(Berkeley Function Calling Leaderboard)은 도구 호출 능력 평가의 중요한 기준입니다. 12.2절의 내용을 기반으로 깊이 생각해 보세요:

   > **힌트**: 이것은 실습 문제입니다. 직접 조작해 보는 것을 권장합니다.

   - 12.2.3절의 AST 매칭 알고리즘에서 추상 구문 트리를 비교하여 함수 호출이 올바른지 판단합니다. 분석해 보세요: AST 매칭이 단순한 문자열 매칭보다 더 적합한 이유는 무엇인가요? 어떤 상황에서 AST 매칭이 오판(거짓 양성 또는 거짓 음성)을 생성할 수 있나요? AST 매칭 알고리즘의 정확성을 높이기 위해 어떻게 개선할 수 있나요?
   - BFCL 데이터셋은 simple, multiple, parallel, irrelevance의 네 가지 카테고리를 포함합니다. 각 카테고리에 대해 에이전트가 해당 카테고리에서 경계 상황이나 오류가 발생하기 쉬운 시나리오를 테스트할 수 있는 새로운 테스트 샘플을 2-3개씩 설계하세요.
   - 12.2.4절의 코드를 기반으로 BFCL 평가기를 확장하고 다음 기능을 추가하세요: (1) 도구 호출의 실행 순서 평가 지원(의존 관계가 있는 여러 도구 호출에 대해); (2) 도구 호출 효율성 평가(최소한의 호출 횟수 사용 여부 등); (3) 상세한 오류 분석 보고서 생성(어떤 유형의 오류가 가장 많이 발생하는지 등).

3. GAIA(General AI Assistants)는 에이전트의 종합 능력을 평가합니다. 12.3절의 내용을 기반으로 다음 확장 실습을 완료하세요:

   > **힌트**: 이것은 실습 문제입니다. 직접 조작해 보는 것을 권장합니다.

   - 12.3.2절에서 GAIA의 세 가지 난이도 수준(Level 1/2/3)을 소개했습니다. 분석해 보세요: 이 세 가지 수준은 작업 복잡도, 필요한 능력, 평가 기준 등에서 어떤 차이가 있나요? Level 4(초고난이도)를 설계한다면 어떤 유형의 작업을 포함해야 할까요?
   - GAIA는 "준정확 매칭" 알고리즘을 사용하여 답안의 정확성을 평가합니다. 분석해 보세요: 이 방법은 답안의 다양성을 어떻게 처리하나요("42", "사십이", "42.0"은 모두 정답으로 인정되어야 함)? 어떤 상황에서 준정확 매칭이 충분하지 않을까요? 의미적으로 동등한 답안을 처리할 수 있는 더 스마트한 답안 매칭 알고리즘을 설계하세요.

4. LLM Judge는 대형 언어 모델을 사용하여 평가하는 새로운 방법입니다. 12.4절의 내용을 기반으로 깊이 분석하세요:

   - 12.4.2절에서 GPT-4를 심사위원으로 사용하여 에이전트의 답변 품질을 평가합니다. 분석해 보세요: LLM Judge가 전통적인 규칙 매칭이나 지표 계산에 비해 어떤 장점이 있나요? 어떤 잠재적인 편향이나 한계가 있나요(특정 답변 스타일에 대한 선호도, 길이에 대한 민감도 등)?
   - LLM Judge의 채점 기준 설계는 매우 중요합니다. 다음 세 가지 다양한 평가 시나리오에 대한 자세한 채점 기준(평가 차원, 가중치, 예시 포함)을 설계하세요: (1) 코드 생성 품질 평가; (2) 창의적 글쓰기 품질 평가; (3) 기술 문서 품질 평가.
   - 12.4.3절에서 여러 LLM Judge를 사용하여 "심사단" 방식 평가를 수행할 수 있다고 언급했습니다. "다중 심사위원 평가 시스템"을 설계하세요: 3-5개의 다양한 LLM(GPT-4, Claude, Qwen 등)을 심사위원으로 사용할 경우, 어떻게 점수를 집계하나요? 심사위원 간 의견 불일치를 어떻게 처리하나요? 이상 점수를 어떻게 감지하고 필터링하나요?

5. 에이전트 평가의 실제 적용에는 여러 측면을 고려해야 합니다. 생각해 보세요:

   - 실제 프로젝트에서 평가는 종종 "평가 비용"과 "평가 품질" 사이에서 균형을 찾아야 합니다. "계층적 평가 전략"을 설계하세요: (1) 빠른 평가(저비용, 일상적인 개발 반복에 사용); (2) 표준 평가(중간 비용, 버전 출시 전에 사용); (3) 전면 평가(고비용, 주요 업데이트 또는 외부 출시에 사용). 각 계층은 어떤 평가 항목을 포함해야 하나요? 평가 흐름을 어떻게 설계하나요?
   - 에이전트의 성능은 시간이 지남에 따라 변할 수 있습니다(의존하는 외부 API 변경, 사용자 요구 변경 등). "지속적 평가 시스템"을 설계하세요: 정기적으로 자동으로 평가를 실행하고, 에이전트 성능 변화 추세를 모니터링하며, 성능이 하락하면 적시에 알림을 보낼 수 있어야 합니다. 이 시스템은 어떤 컴포넌트를 포함해야 하나요? 알림 규칙을 어떻게 설계하나요?
   - 평가 결과는 다양한 대상(개발자, 프로덕트 매니저, 사용자 등)에게 명확하게 제시되어야 합니다. "평가 보고서 생성 시스템"을 설계하세요: 대상 유형에 따라 자동으로 다양한 상세 수준의 보고서를 생성할 수 있어야 합니다. 개발자 보고서에는 어떤 기술적 세부사항이 포함되어야 하나요? 프로덕트 매니저 보고서는 어떤 비즈니스 지표를 강조해야 하나요? 사용자 보고서는 어떻게 간소화하고 시각화해야 하나요?
